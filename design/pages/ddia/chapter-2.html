Chapter 2 — Data Models & Query Languages

Big Picture: Data models are the single most consequential decision in system design — they determine how you think about the problem. Applications layer models: objects in code → rows in DB → bytes on disk → electrical signals. Each layer hides complexity below it. This chapter compares the three dominant models: Relational, Document (NoSQL), and Graph.

1 Relational Model

  • Proposed by Edgar Codd, 1970. Data organized as relations (tables) with tuples (rows). Query language (SQL) is declarative — say what, not how.
  • Before relational DBs, developers navigated IMS/CODASYL pointer-based hierarchical structures manually — the storage representation was the programming interface.
  • Key win: SQL separates the query from the execution plan. The optimizer chooses the best index and join strategy — developers declare intent.
  • Best for: banking, e-commerce, ERP, anything with complex many-to-many relationships, ACID transactions, ad-hoc queries.
Normalization & Impedance Mismatch
  • Normalization: store each fact exactly once; reference via foreign keys. "Google" → "Alphabet" rename = 1 UPDATE. In a denormalized system: find and update every occurrence.
  • Impedance mismatch: application code uses objects with nested lists (User → List<Job>); relational DB stores flat tables. ORMs bridge this gap but introduce N+1 query problems and awkward JOIN-heavy queries.
LinkedIn Resume — Relational (Normalized) Layout
  Table: users          id=1, name='Alice', headline='Eng Lead'
                          │
         ┌────────────────┼────────────────────────────────┐
         ▼                ▼                                ▼
  Table: positions     Table: education             Table: skills
  {user_id=1,          {user_id=1,                  {user_id=1, skill='Java'}
   title='Eng Lead',    school='MIT',                {user_id=1, skill='Kafka'}
   company='Google'}    degree='B.S. CS'}
  {user_id=1,
   title='Sr Eng',
   company='Stripe'}

  Full profile = JOIN across 4 tables. Changing 'Google' to 'Alphabet'?
  One UPDATE on positions table — all of Alice's jobs update automatically.

2 Document Model (NoSQL)

  • Stores self-contained JSON/BSON documents. One document = all related data. No JOINs needed for typical reads.
  • Schema-on-read: structure is implicit; interpreted at read time. Add new fields without migrations — just start writing them.
  • Locality advantage: all data for one entity stored contiguously → single disk seek to read the entire profile.
  • Best for: document-like data with one-to-many nesting, flexible/evolving schema, high write throughput, tree-structured data.
JSON — Same Resume as a Document (MongoDB/DynamoDB)
{
  "user_id": 251,
  "name": "Alice",
  "headline": "Engineering Lead at Google",
  "positions": [
    { "title": "Engineering Lead", "company": "Google",  "start": "2020-03" },
    { "title": "Senior Engineer",  "company": "Stripe",  "start": "2018-01" }
  ],
  "education": [
    { "school": "MIT", "degree": "B.S. Computer Science", "year": 2016 }
  ],
  "skills": ["Java", "Kafka", "Kubernetes", "Go"]
}

// One GET retrieves the entire profile. No JOINs.
// Adding "certifications" field? Just start writing it — zero schema migration.
// Reading on old clients? They see the field is missing → handle as empty array.
Schema-on-Write vs Schema-on-Read
Schema Evolution — Relational vs Document
  Relational (schema-on-write):
    ALTER TABLE users ADD COLUMN phone VARCHAR(20) DEFAULT NULL;
    -- DB enforces schema at write. Migration runs on live table.
    -- Large tables: hours of locking + careful zero-downtime migration.

  Document (schema-on-read):
    // Just start writing new documents with the "phone" field
    { "user_id": 1, "name": "Alice", "phone": "+1-555-0100" }
    // Old documents: no "phone" field — app code handles:
    phone = doc.get("phone", "N/A")

  Analogy: schema-on-write = statically typed language (compiler enforces at compile time)
           schema-on-read  = dynamically typed language (error found at runtime)

3 Relational vs Document — Decision Framework

CriterionRelational (SQL)Document (NoSQL)
Data fitTabular, many-to-many, normalizedTree-like, one-to-many, self-contained
JOINsNative, optimizer-managedMust denormalize or join in app code
SchemaSchema-on-write (migrations required)Schema-on-read (flexible, evolving)
TransactionsMulti-row ACID across tablesUsually single-document ACID only
Horizontal scaleHarder (cross-shard JOINs)Designed for sharding
ConsistencyStrong (FK constraints, CHECK)Eventual (no FK enforcement)
ExamplesPostgreSQL, MySQL, Oracle, CockroachDBMongoDB, DynamoDB, Firestore, Couchbase
If your data has many-to-many relationships (users → follows → posts → comments → likes), the document model forces you to either duplicate data (denormalize) or implement JOINs in application code — which is always worse than having the DB do it.

4 Declarative vs Imperative Query Languages

JavaScript vs SQL — Filtering Sharks
// Imperative (tell HOW — you control the execution):
var sharks = [];
for (var i = 0; i < animals.length; i++) {
  if (animals[i].family === "Sharks") {
    sharks.push(animals[i]);
  }
}
// You define loop, check, accumulation. Optimizer cannot change this.

// Declarative (tell WHAT — optimizer decides HOW):
SELECT * FROM animals WHERE family = 'Sharks';
-- DB can: use an index, parallelize the scan, reorder predicates,
-- use bitmap scan, push down to storage layer. All transparent to you.
-- Same query runs better after adding an index — no code change needed.
  • Declarative benefits: optimizer freedom; automatic parallelization; abstraction from storage engine changes; concise code.
  • CSS is declarative: li.selected > p { color: blue; } — you don't write a DOM traversal loop. Browser optimizes rendering.
  • Cypher, SPARQL, SQL, XPath — all declarative. The engine decides execution strategy.

5 Graph-Like Data Models

  • When relationships are the central concern, use a graph database. Vertices (nodes) + edges (relationships). Both can have properties.
  • Any vertex can relate to any other — no schema restriction. Highly flexible for evolving relationship types.
  • Used for: social networks, fraud detection, recommendation engines, knowledge graphs, route optimization.
Cypher (Neo4j) — Find people who emigrated from US to Europe
MATCH
  (person) -[:BORN_IN]->  ()-[:WITHIN*0..]-> (us:Location {name:'United States'}),
  (person) -[:LIVES_IN]-> ()-[:WITHIN*0..]-> (eu:Location {name:'Europe'})
RETURN person.name

-- [:WITHIN*0..] = traverse WITHIN edges recursively (0 or more hops)
-- This is a variable-length path traversal.
-- Equivalent SQL requires WITH RECURSIVE CTEs — verbose and often slower.
-- Graph DB stores physical adjacency → traversal is O(relationships visited),
-- not O(table size). SQL JOIN is O(matched rows across both tables).

🧠 FAANG Interview Questions

Q1Instagram has photos, users, follows, comments, likes. Which databases would you use and why? (Polyglot persistence)
Answer

Use polyglot persistence — purpose-select each store for its access pattern. No single database optimally handles objects, blobs, graphs, counters, full-text, and timelines simultaneously. The cost is operational complexity; the payoff is each system runs at peak efficiency.

Store Selection
  • User profiles + post metadata → PostgreSQL: Requires ACID transactions (follow/unfollow must be atomic, billing must not double-charge), complex JOIN queries (user analytics, moderation dashboards), and foreign-key integrity. Shard by user_id at scale (>100M rows).
  • Photos / videos → S3 + CloudFront CDN: Binary objects belong in object storage, not row-based DBs. PostgreSQL stores only the S3 key. CloudFront serves the bytes from the edge node nearest the user, cutting latency from 200ms to <20ms globally.
  • Home feed / timeline → Redis sorted sets: ZADD feed:{user_id} {unix_ms} {post_id}. Fan-out on write (pre-computed per user). Read = ZRANGE O(log N). Hot data stays in RAM; cold pages to disk. Typical feed read is <1ms.
  • Likes / view counts → Redis INCR or Cassandra counters: Millions of increments per second; eventual consistency is fine (off-by-one on a like count matters nothing). Cassandra counter columns handle write-heavy workloads across nodes without hot-row contention.
  • Hashtag / caption search → Elasticsearch: Inverted index for full-text, geo-search for nearby posts, faceted search by date/popularity. CDC (Debezium) streams changes from PostgreSQL into Elasticsearch to keep them in sync.
  • "People you may know" / explore → offline batch + lookup table: Real-time graph traversal doesn't scale to 1B users. Run a daily Spark job over the social graph, persist top-50 recommendations per user in DynamoDB, serve in O(1).
Interview differentiator: Make the trade-off explicit — "The cost of polyglot persistence is that I now have 5 systems to monitor, back up, and keep consistent. I justify that cost when the alternative is a single overloaded database that can't meet the read/write SLAs for any of the use cases."
Q2When would you choose MongoDB over PostgreSQL? What specific risks must you mitigate?
Answer

Choose MongoDB when your data is genuinely document-shaped (self-contained, variable-structure, read as a unit) and horizontal write scale is a day-one requirement. Avoid it when you need strong referential integrity, complex multi-entity joins, or multi-document ACID transactions — those scenarios strongly favour PostgreSQL.

When MongoDB Wins
  • Product catalog with heterogeneous attributes: A TV has {screen_size, resolution, HDR_support}; a shirt has {size, color, fabric}. Forcing this into relational columns requires the EAV (Entity-Attribute-Value) anti-pattern — worse than MongoDB.
  • CMS / blog content: An article with embedded sections, embedded authors, embedded tags — loaded and rendered as a unit. No cross-document joins needed.
  • Rapid iteration phases: When the schema changes weekly, schema-on-read avoids the ALTER TABLE overhead. Once the schema stabilises, consider migrating to PostgreSQL.
Risks and Mitigations
  • No FK constraints → orphaned documents: Enforce referential integrity at the application layer. Add integration tests that verify no orphans exist. Use schema validation (MongoDB 3.6+ JSON Schema) to enforce required fields.
  • Schema drift: email in old docs, emailAddress in new docs. Enforce consistent field names via application-level schema classes (e.g., Mongoose schemas, Java POJOs). Run a nightly audit query that checks for field name inconsistencies.
  • Weak multi-document transactions: MongoDB multi-document transactions (v4.0+) exist but have higher overhead than Postgres transactions. If your operations frequently span multiple documents atomically, restructure to embed or move to PostgreSQL.
  • The anti-pattern trap: Teams choose Mongo for "flexibility," then rebuild a relational model in app code. If after 6 months you're doing in-app JOINs, you've recreated SQL — but without indexes, query planner, or FK enforcement.
Interview differentiator: "I treat MongoDB as a deliberate choice for document workloads, not a default. My decision tree: Does this data have many-to-many relationships? → PostgreSQL. Is the schema self-contained and variable? → MongoDB. Do I need sub-millisecond key lookups at massive scale? → DynamoDB."
Q3Add a "phone_number" column to a 500M-row production table in PostgreSQL without taking downtime.
Answer

Use the expand-migrate-contract three-phase pattern: first add the column as nullable (instant DDL), then backfill in small batches (background, no locks), then enforce the constraint once all rows have data. Never run a locking DDL on a large table in production without an OSC tool.

Phase 1 — Add Column (instantaneous)
-- In PostgreSQL ≥ 11: adding a nullable column updates only the catalog
ALTER TABLE users ADD COLUMN phone_number VARCHAR(20) DEFAULT NULL;
-- Completes in milliseconds, zero table lock
Phase 2 — Backfill in Batches (background)
-- Process 5000 rows at a time with a sleep between batches
-- to avoid lock queuing and replication lag
DO $$
DECLARE batch_start BIGINT := 1;
BEGIN
  LOOP
    UPDATE users SET phone_number = lookup_phone(id)
    WHERE id BETWEEN batch_start AND batch_start + 4999
    AND phone_number IS NULL;
    batch_start := batch_start + 5000;
    EXIT WHEN batch_start > 500000000;
    PERFORM pg_sleep(0.05);  -- 50ms pause
  END LOOP;
END $$;
Phase 3 — Enforce NOT NULL (once backfill completes)
-- PostgreSQL 12+: validate constraint without full table scan
ALTER TABLE users ADD CONSTRAINT phone_not_null
  CHECK (phone_number IS NOT NULL) NOT VALID;
ALTER TABLE users VALIDATE CONSTRAINT phone_not_null;
-- Then promote to NOT NULL:
ALTER TABLE users ALTER COLUMN phone_number SET NOT NULL;
  • Also use CREATE INDEX CONCURRENTLY to add indexes without blocking reads/writes. Regular CREATE INDEX holds a lock for the entire build.
  • Tooling: gh-ost (GitHub's Online Schema Change tool), pt-online-schema-change (Percona), or pg_repack for PostgreSQL. These shadow the table, replicate changes, then atomically swap — zero downtime for any DDL.
  • Decouple deploy from migration: Feature flags let you deploy new code that writes phone_number before the column exists, or reads it gracefully when null. Migrate the schema independently from deploying code.
Interview differentiator: The interviewer is testing whether you know about table locks. Lead with: "The naive ALTER TABLE would acquire an ACCESS EXCLUSIVE lock on a 500M-row table and block all reads and writes for hours. Here's how I avoid that…" This immediately signals production experience.
Q4Design LinkedIn's "People You May Know." What data model, algorithm, and serving architecture?
Answer

Model the social graph as vertices (users) + edges (connections). Compute recommendations offline in batch using a graph algorithm (Jaccard similarity of neighbor sets), store results in a fast serving store, and serve in O(1). Real-time graph traversal at LinkedIn's scale (900M users, avg 300 connections) would hit millions of nodes per query — offline batch is the correct architecture.

Graph Data Model
  • Vertices: users with properties {user_id, name, skills[], company, school}
  • Edges: undirected CONNECTED(user_a, user_b, connected_at)
  • Problem to solve: For user A, find users B where |neighbors(A) ∩ neighbors(B)| is high but no direct edge A–B exists.
Offline Batch Architecture
  • Daily Spark GraphX job: Load the full adjacency list (compressed: ~50 GB for 900M users × 300 avg connections). For each user, compute Jaccard(A, B) = |A∩B| / |A∪B| for all 2-hop neighbors. Rank top 50 by score.
  • Serving: Write results to DynamoDB: PK=user_id, SK=rank → {recommended_id, score, reason}. API reads top-20 in O(1) at <5ms. Cache in Redis for hot users.
  • Weighting signals: Mutual connections (strongest), same company (+1.5×), same school (+1.2×), same skills (+1.1×), profile view history (+0.8×). Exclude already-connected, blocked, or deactivated users.
  • Near-real-time updates: New connection event triggers incremental computation (only affected user's neighborhood). Kafka event → Flink streaming job → update DynamoDB recommendation row. Full batch recalculates everything nightly.
Interview differentiator: Explicitly justify why you're NOT using a real-time graph database: "Neo4j is excellent for real-time 2-hop traversal up to millions of nodes. At LinkedIn's scale — 900M users, 30B+ edges — a single 3-hop traversal could touch billions of nodes. The offline batch + serving table pattern is the industry-standard solution for large-scale recommendations."
Q5Explain the N+1 query problem and all the ways to solve it in a production REST API.
Answer

The N+1 problem occurs when code fetches a list of N records then executes one additional DB query per record to fetch related data — turning 2 DB calls into N+1. At 1000 req/s with N=100, that's 100,000 extra DB queries per second. This single bug has taken down production databases at multiple companies.

The Problem — Concrete Example
// WRONG — N+1 queries
List<Post> posts = postRepo.findAll(limit=100);  // Query 1
for (Post p : posts) {
    User author = userRepo.findById(p.getAuthorId()); // Query 2..N+1
    p.setAuthorName(author.getName());
}
Solution 1 — JOIN (single query)
-- One query fetches everything
SELECT p.id, p.title, u.name AS author_name
FROM posts p
JOIN users u ON p.author_id = u.id
LIMIT 100;
-- JPA: @Query("SELECT p FROM Post p JOIN FETCH p.author LIMIT 100")
Solution 2 — Batch loading (DataLoader pattern)
// Collect all author IDs, fetch in ONE query
Set<Long> authorIds = posts.stream()
    .map(Post::getAuthorId).collect(toSet());
Map<Long, User> authorsById = userRepo.findAllById(authorIds)
    .stream().collect(toMap(User::getId, identity()));
// SELECT * FROM users WHERE id IN (1,2,3,...,100)  ← single query
posts.forEach(p -> p.setAuthor(authorsById.get(p.getAuthorId())));
  • DataLoader (GraphQL): Facebook's DataLoader batches and caches all field-level fetches within a single request. Instead of N resolver calls each firing a DB query, DataLoader coalesces them into one WHERE id IN (...) call.
  • Detection: Hibernate statistics logging, Rails Bullet gem, Django Debug Toolbar. Alert if any single request fires >10 queries — almost always a sign of N+1.
  • Caching layer: For hot reference data (author profiles change rarely), add a read-through cache: check Redis first; on miss, fetch from DB and cache for 5 min. Reduces N+1 queries to N cache hits — much cheaper than N DB queries.
Interview differentiator: Quantify the impact: "At 500 req/s, N+1 with N=50 generates 25,000 extra DB queries/sec. A typical PostgreSQL instance handles ~10,000 queries/sec comfortably. This single bug would saturate the database and take the service down." Numbers like these immediately demonstrate production-level thinking.
