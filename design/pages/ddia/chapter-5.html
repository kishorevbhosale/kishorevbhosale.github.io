Chapter 5 — Replication

Big Picture: Replication = keeping copies of data on multiple machines. Goals: reduce read latency (serve from nearby region), increase availability (survive node failures), increase read throughput. The hard part: keeping replicas in sync while continuously accepting writes.

1 Single-Leader Replication

  • One node designated as leader (master/primary). All writes go to leader only. Reads can go to any follower (or leader for strong consistency).
  • Leader sends replication log to followers. Followers apply changes in the same order → eventually identical data.
  • Used by: PostgreSQL (streaming replication), MySQL (binlog replication), MongoDB (oplog), Kafka (ISR-based), Redis (AOF streaming).
Single-Leader Architecture
  ┌─────────────┐
  │   Client    │ ──── WRITE ────►  ┌──────────────┐
  └─────────────┘                   │    LEADER    │ ──── replication log ──► Follower 1
                                    │  (Primary)   │ ──── replication log ──► Follower 2
  ┌─────────────┐                   │              │ ──── replication log ──► Follower 3
  │   Client    │ ──── READ  ◄────  └──────────────┘                              │
  └─────────────┘   (or any follower)                   Client READ ◄─────────────┘
Sync vs Async Replication
Synchronous vs Asynchronous Replication
  SYNCHRONOUS (leader waits for follower ACK):
  Client → Leader writes → sends to Follower 1 → waits → Follower 1 ACKs → ACK client
  ✅ No data loss on leader failover (Follower 1 has all data)
  ❌ One slow/unreachable follower BLOCKS all writes. Impractical for all-sync.

  SEMI-SYNCHRONOUS (MySQL default — one sync follower, rest async):
  Client → Leader writes → sends to Follower 1 (sync) → ACK to client
                        → also sends to Follower 2,3 (async, best-effort)
  ✅ Guarantees at least one follower is current. Survives leader crash.
  ❌ Small chance of data loss if both leader AND sync follower die simultaneously.

  FULLY ASYNCHRONOUS (default in PostgreSQL streaming replication):
  Client → Leader writes → ACK client immediately → replicates in background
  ✅ Lowest write latency, highest throughput, leader never blocked.
  ❌ Leader crash before sync → acknowledged writes LOST. RPO > 0.
Failover — The 3 Dangerous Problems
These are real production incidents. Every organization with automatic failover has been burned by at least one of these.
  • Write loss (GitHub 2012): MySQL async replication. Old leader had 1 minute of unsynced writes. New leader promoted, had old state. Old leader's PKs (auto-increment) were reused. MySQL and Redis (using those PKs as cache keys) served wrong data to users for hours. Fix: use sync replication for critical data OR accept RPO > 0 with monitoring.
  • Split brain: network partition → both old and new leader accept writes simultaneously → conflicting data. Fix: STONITH (Shoot The Other Node In The Head) — physically power off old leader before promoting new one. Fencing tokens (next chapter).
  • Timeout tuning dilemma: Too short → normal GC pause triggers failover → cascading load on new leader → second failover → cluster thrashing. Too long → users wait N seconds for dead leader. Solution: adaptive timeout (Phi Accrual Failure Detector).

2 Replication Lag — 3 Problems You Must Handle

Problem 1: Read-Your-Own-Writes
Read-Your-Own-Writes Violation
  t=0: User Alice posts comment "Hello!" → write goes to Leader
  t=1: Alice refreshes page → LB routes to Follower A (lagging 3 seconds)
       Follower A: "Comment not found" → Alice posts duplicate!
  t=4: Follower A catches up → comment visible twice

  Fix 1: Read Alice's OWN data from leader always.
         (Other users reading Alice's comments = OK to use followers)
  Fix 2: Track "last_write_timestamp" in Alice's session cookie/JWT.
         LB: only route reads to replicas with data_age ≤ session timestamp.
         If no such replica available → route to leader.
  Fix 3: Sticky sessions for 30-60s after a write (simple but wastes follower capacity).
Problem 2: Monotonic Reads
  • Alice refreshes twice: Request 1 → Follower A (up-to-date, shows 5 comments). Request 2 → Follower B (lagging, shows 3 comments). Data appears to go backwards in time.
  • Fix: sticky replica routing per user. Hash(user_id) → deterministic replica. Client always sees non-decreasing version of data. If assigned replica fails → reroute (user sees one potential jump backwards, acceptable).
Problem 3: Consistent Prefix Reads
  • Alice writes "Q: How are you?" to Shard 1. Bob writes "A: Fine!" to Shard 2. Observer reads Shard 2 (faster replica) then Shard 1 (slower replica) → sees answer before question. Conversation appears backwards.
  • Fix: Route causally related writes to the same partition. Alternatively: use causal dependency metadata (happens-before tracking) in read/write path.

3 Multi-Leader Replication

  • Each datacenter has its own leader. Writes accepted locally (low latency). Leaders asynchronously replicate to each other.
  • Benefit: inter-DC link failure → both DCs continue independently. Local writes = low latency (no cross-Atlantic round trip).
  • Problem: two leaders accept conflicting writes for the same record → write conflicts.
Multi-Leader — Write Conflict Resolution Strategies
  User updates title: US DC sets "Alice" at t=10:05:01; EU DC sets "Alicia" at t=10:05:00
  Both committed locally. During sync: conflict detected.

  Strategy 1: Last Write Wins (LWW) — highest timestamp wins.
    "Alice" (t=10:05:01) beats "Alicia" (t=10:05:00) → title = "Alice"
    ⚠️ Dangerous: clock skew (100ms NTP drift) can silently discard newer writes.

  Strategy 2: Merge — keep both as siblings.
    title = {"Alice", "Alicia"}
    On next read: app code or user resolves.
    Good for: shopping cart (union of items), collaborative docs (CRDT).

  Strategy 3: Custom conflict handler.
    Application registers an on-conflict function.
    Called synchronously on write (Riak) or asynchronously on read (CouchDB).

  Strategy 4: Avoid conflicts (best).
    Route all writes for one entity to one designated DC.
    Falls apart only if that DC fails or user moves region.

4 Leaderless Replication (Dynamo-Style)

  • No designated leader. Client (or coordinator) writes to multiple replicas directly.
  • Quorum condition: w + r > n ensures overlap → at least one node in read set has the latest write.
  • Read repair: during a read, if a stale replica is detected, the reader writes the newer value back to it.
  • Anti-entropy: background process compares replicas and syncs differences.
Quorum Reads and Writes (n=5, w=3, r=3)
  WRITE (w=3 required):
    Client → Replicas 1,2,3: write v=5  ✓ (3 ACKs received)
    Replicas 4,5: write failed/slow     (still at v=4 — temporarily stale)

  READ (r=3 required):
    Client → Replicas 1, 3, 4:
      Replica 1 → v=5 ✓    Replica 3 → v=5 ✓    Replica 4 → v=4 (stale!)
    Quorum says: take v=5 (highest version wins)
    Read repair: send v=5 to Replica 4 in background

  Why w+r > n guarantees freshness:
    w=3 + r=3 = 6 > n=5 → write set and read set MUST overlap by at least 1 node
    That overlap node has the latest written value → client always gets current data.

🧠 FAANG Interview Questions

Q1A customer placed an order but can't see it in "My Orders" immediately after. Root cause and fix?
Answer

This is a classic Read-After-Write (RAW) consistency violation. The write went to the primary; the read was routed to an asynchronous replica that hadn't yet received the replication update. Replication lag = milliseconds to seconds. The fix is to route "my own recent writes" reads to the primary, or use a session token to route to a replica that has caught up.

Root Cause Detail
  • MySQL async replication: primary commits write to binlog, replica pulls and replays asynchronously. Under normal load: lag = 10–200ms. Under heavy write load: lag can spike to seconds or minutes.
  • When "My Orders" endpoint routes to a replica via round-robin load balancing, there's a race: "Will the replica have applied the latest binlog entry before this read arrives?" Answer: sometimes no.
Fix Options
  • Option 1 — Route "own writes" to primary: Any endpoint that must reflect the user's own recent writes uses the primary connection pool. Annotate endpoints: @PrimaryDataSource. Simple and reliable; downside: increases primary load.
  • Option 2 — Session-based replication position tracking: After write, store the binlog position in the user's session (or a cookie). Next read request: middleware (Vitess, ProxySQL) checks which replica has replayed past that position. Route to qualifying replica; fall back to primary only if none qualify. This keeps read load on replicas while guaranteeing read-after-write.
  • Option 3 — Client-side monotonic reads: After a successful write, include a read_after_write_ts header in the response. Client sends it on the subsequent GET. Server rejects requests to stale replicas (lag > read_after_write_ts) and re-routes.
Monitoring
  • Track Seconds_Behind_Master (MySQL) or pg_stat_replication.write_lag (PostgreSQL) as a metric. Alert at >5s lag. Lag spikes are a leading indicator of primary overload before it causes customer-visible issues.
Interview differentiator: Name all four replication consistency models explicitly: read-after-write (users see their own writes), monotonic reads (users don't see time go backward — same user always reads from same replica), consistent prefix reads (causal ordering), and monotonic writes (writes from same client applied in order). "This specific bug is a read-after-write violation — one of four common replication anomalies."
Q2Design a globally distributed user profile system (10 datacenters worldwide). How do you handle multi-region replication and write conflicts?
Answer

Use multi-leader (active-active) replication — one leader per region accepts local writes (sub-10ms latency for local users). Async cross-region replication (avoid 150–300ms cross-Atlantic/Pacific round trip on every write). Handle conflicts per field type: LWW with Hybrid Logical Clocks for display preferences; home-region routing for critical fields (email, password).

Architecture
  • One leader per region: US-EAST, US-WEST, EU-WEST, AP-SOUTHEAST. GeoDNS routes each user to nearest region. Local writes are acknowledged in <5ms. Async replication to other regions: ~150ms cross-Atlantic, ~200ms cross-Pacific.
  • Conflict resolution by field type:
  • Display preferences (theme, language, notification settings): Last-Write-Wins using Hybrid Logical Clocks (HLC). HLC = max(wall_clock, last_seen_HLC) + 1 — monotonically increasing even across machines with clock skew. "Wrong theme for 150ms before sync" is acceptable. LWW is safe here.
  • Critical fields (email, phone, password_hash): Designate a "home region" per user (region where account was created). All writes to critical fields are routed to the home region regardless of current location. This eliminates the conflict problem entirely for sensitive data.
  • Set-valued fields (blocked users, saved items): Use CRDTs (Add-Wins Set or Remove-Wins Set). Mathematically defined merge semantics — no "which one wins" judgment needed.
Google Spanner Alternative
  • TrueTime GPS/atomic clock + Paxos per shard → linearizable global transactions. Every write waits for TrueTime uncertainty window (~7ms) before committing to ensure global ordering. Google accepts 5–7ms added write latency for strong consistency. Almost no one outside Google can afford to build this.
Interview differentiator: Explicitly name the multi-leader trade-off: "Every multi-leader system trades write latency for consistency. Strong consistency requires cross-region coordination on every write (Spanner-style). Eventual consistency with per-field conflict rules is what 99% of companies actually build. I'd design for LWW + CRDT + home-region routing and avoid building distributed consensus unless the business requirement truly demands linearizability."
Q3Cassandra cluster with n=3, w=ALL, r=ONE. Explain what this guarantees, why you'd use it, and what the failure risk is.
Answer

w=ALL, r=ONE means: writes are only acknowledged after all 3 replicas confirm (slowest write possible, zero tolerance for write-time node failure), but reads are satisfied by the first replica that responds (fastest read possible). This satisfies w+r=4 > n=3, which theoretically guarantees strong consistency — every read will find the latest write. But it sacrifices write availability: any single node failure blocks ALL writes.

When You'd Choose This
  • Write-rarely, read-millions pattern: product catalog, exchange rates, configuration data. These are written once a day but read millions of times per second. r=ONE gives sub-millisecond reads served by the nearest replica with zero coordination overhead.
  • Acceptable trade-off: writing 3 rows/day and the write takes 50ms instead of 15ms — negligible. But the reads being 0.5ms instead of 2ms (QUORUM) is significant at millions of reads/day.
The Failure Risk
  • w=ALL means a single node failure blocks all writes. In a 3-node cluster, nodes go down for: rolling upgrades, hardware failure, network partition, GC pauses. Probability of at least one node being unavailable in any given week in a 3-node cluster ≈ high. All writes fail during that window.
  • Stale read edge case: If node N2 was DOWN during a write (which failed and was retried), N2 comes back with stale data. r=ONE might hit N2 → stale read. Hinted handoffs help but don't guarantee N2 gets all writes it missed.
  • Production default: QUORUM/QUORUM (w=2, r=2 for n=3). Tolerates 1 node failure for both reads AND writes. Provides strong consistency (w+r=4 > n=3). p99 read adds ~1ms for coordinator acknowledgement from 2 replicas.
Interview differentiator: Lead with the quorum formula: "Strong consistency requires w + r > n. For n=3: QUORUM+QUORUM (2+2=4>3), ALL+ONE (3+1=4>3), ONE+ALL (1+3=4>3) all satisfy it. The choice is where you want to pay the consistency tax: at write time or at read time. For read-heavy workloads, pay at write time (w=ALL); for write-heavy workloads, pay at read time (r=ALL). Most systems use QUORUM on both sides for balanced availability."
Q4Explain version vectors and why Last-Write-Wins (LWW) is dangerous in distributed systems.
Answer

Version vectors track causal history to detect concurrent writes — two writes that happened independently without either knowing about the other. LWW uses wall-clock timestamps to resolve conflicts by discarding the "older" write, but clock skew between machines (50ms–1s typical, can spike to minutes during NTP corrections) means the "older" timestamp might actually be the causally later write — resulting in silent data loss with no error message.

Version Vector Mechanics
  • Format: [node1_counter, node2_counter, node3_counter]. Each write on node_i increments counter_i. Full vector attached to every value.
  • Happens-before: V1 happened-before V2 if every component of V1 ≤ corresponding component of V2 (and at least one is strictly less). Safe to discard V1.
  • Concurrent writes: [2,1,0] vs [1,2,0] — neither dominates. These are concurrent: happened independently. Must merge or present conflict to application.
  • Riak uses version vectors. On conflict: returns both versions as "siblings." Application resolves (keep newer, merge both, ask user).
Why LWW Is Dangerous
  • Clock skew is real and common: NTP adjustments: ±50ms normally, ±500ms during sync, up to minutes during VM live migration or leap second handling.
  • The failure scenario: Client A writes x=1 at T=100ms. Client B writes x=2 at T=90ms (B's clock is 10ms behind — later in real time but lower timestamp). LWW picks x=1 (higher timestamp). B's write x=2 is silently discarded — no exception, no error log, no alert. You discover it when a user asks "why was my cart reset?"
  • LinkedIn incident: NTP resync caused node clocks to jump backward by 500ms. ~30 minutes of writes with recent timestamps were silently overwritten by writes with earlier (but now-dominant) timestamps. Data loss discovered hours later via data quality monitoring.
  • Safe uses of LWW: Fields where "last update wins" is genuinely correct business semantics (user's display name, theme preference, last-seen timestamp). Never for financial data, append-based counters, or set-valued fields.
Interview differentiator: "LWW is dangerous because it's silent — there's no error, no warning, just missing data. The safe alternative is either (a) version vectors + application-level merge, (b) CRDTs with mathematically defined merge semantics, or (c) Paxos/Raft-based single-leader with no conflict possible. I use LWW only when I can prove the business semantics are genuinely last-write-wins and when clock accuracy is verified via tight NTP + monitoring."
