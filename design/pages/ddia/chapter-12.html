Chapter 12 — The Future of Data Systems

Big Picture: No single database does everything well. The modern answer: compose specialized systems with an event log as the source of truth, derived views computed from it, and end-to-end correctness enforced through idempotency and integrity constraints. This chapter synthesizes the entire book and adds data ethics.

2 Unbundling the Database

  • Traditional DB bundles: storage + indexing + query optimizer + transaction manager + replication + backup — all in one system.
  • Modern pattern: Kafka as the global WAL. Specialized derived views for each query pattern. Everything composable.
Unbundled Data System — Event Log + Specialized Derived Views
  WRITES (source of truth — Kafka):
  Application → Kafka topic "user-events" (immutable, durable log)

  DERIVED VIEWS (each optimized for one access pattern):
  ┌─────────────────────────────────────────────────────────────────────┐
  │ Flink job → PostgreSQL     : primary OLTP queries, joins, reports   │
  │ Flink job → Elasticsearch  : full-text search, hashtag indexing     │
  │ Flink job → Redis          : user session cache, hot data, feeds    │
  │ Flink job → BigQuery       : analytical dashboards, ad-hoc queries  │
  │ Flink job → S3 Parquet     : ML training data, data lake            │
  │ Flink job → DynamoDB       : low-latency key-value API reads        │
  └─────────────────────────────────────────────────────────────────────┘
  Each "database" is a VIEW of the Kafka log, optimized for its pattern.
  Kafka is the SINGLE SOURCE OF TRUTH. Everything else is derivable.
  Bug in Elasticsearch index? Replay Kafka from the error → rebuild correctly.
This is Event Sourcing + CQRS at internet scale. The "database" is not a single system — it's an event log with specialized projections. Martin Fowler, Greg Young, and Pat Helland all converge on this pattern for scalable, maintainable data architectures.

3 End-to-End Correctness

Idempotency Keys
  • Problem: networks are unreliable. A request can time out with unknown state: did the server process it or not? Retrying causes double-execution for non-idempotent operations.
  • Solution: generate a UUID before the operation. Include it in every attempt. Server caches result by UUID.
Java — Idempotency Key Pattern (Stripe-style payment API)
// CLIENT: generate UUID once per payment intent (not per HTTP call!)
String idempotencyKey = UUID.randomUUID().toString();

// Retry on timeout/failure with SAME key:
for (int attempt = 0; attempt < 3; attempt++) {
    try {
        return paymentClient.charge(amount, currency,
            Map.of("Idempotency-Key", idempotencyKey));
    } catch (TimeoutException e) {
        Thread.sleep(1000 * (1 << attempt)); // exponential backoff
    }
}

// SERVER: check idempotency cache before processing
@PostMapping("/charge")
public ResponseEntity charge(@RequestBody ChargeRequest req,
                             @RequestHeader("Idempotency-Key") String key) {
    // Check cache first (Redis with TTL or idempotency_requests table)
    Optional<ChargeResult> cached = idempotencyCache.get(key);
    if (cached.isPresent()) {
        return ResponseEntity.ok(cached.get()); // return SAME response
    }

    // Process payment (actual charge)
    ChargeResult result = stripe.charge(req);

    // Store result with idempotency key (atomic with the charge if possible)
    idempotencyCache.put(key, result, Duration.ofDays(7));
    return ResponseEntity.ok(result);
}

4 Data Ethics & Privacy

  • Unintended data collection: Location data for navigation → also reveals daily routine, home address, religious attendance, medical clinic visits. Collected for one purpose; potentially harmful when used for another.
  • Algorithmic bias: ML models trained on historical data encode historical inequalities. Recidivism prediction (COMPAS) → racially biased. Hiring algorithms → gender biased. Lending models → ZIP code as proxy for race. Hard to detect; requires active fairness auditing.
  • GDPR "Right to be Forgotten": Deleting a user from every derived view (Elasticsearch, BigQuery, DynamoDB, S3, Redis) is expensive and error-prone. Crypto-shredding is the scalable solution.
Crypto-Shredding — Scalable GDPR Compliance
Crypto-Shredding — Logical Deletion via Key Destruction
  DATA STORAGE:
  ─────────────────────────────────────────────────────────────────────
  On user registration:
    1. Generate user-specific AES-256-GCM key
    2. Store key in Key Management Service (AWS KMS / HashiCorp Vault)
    3. Encrypt all PII fields before writing to any storage:
       events.user_email = encrypt(key, "alice@example.com")
       events.user_name  = encrypt(key, "Alice")
  
  All downstream stores (Elasticsearch, S3, BigQuery, Kafka) store CIPHERTEXT only.

  "RIGHT TO BE FORGOTTEN" REQUEST:
  ─────────────────────────────────────────────────────────────────────
  1. AWS KMS: ScheduleKeyDeletion(user_key_id, 7-day pending period)
  2. After deletion: all ciphertext is permanently unreadable.
     No exception, no error — decrypt() returns garbage.
  
  Effect: user's data is "deleted" from ALL systems simultaneously
          without touching any derived views, indexes, or archives.
  Cost: O(1) — just one KMS API call. vs O(N) for physically deleting from all stores.

  Non-PII (aggregated metrics, anonymized analytics): retained — no key needed.
  Used by: WhatsApp backups, Apple iCloud, Signal.

🧠 FAANG Interview Questions

Q1Design the data architecture for an Uber-scale ride-sharing app using unbundled databases with Kafka as the central event log.
Answer

Use Kafka as the immutable event log (source of truth) and derive purpose-built, read-optimized views for each service: PostgreSQL for transactional records, Redis Geo for real-time driver locations, DynamoDB for precomputed driver earnings, BigQuery for analytics, Elasticsearch for search, and Redis feature store for ML inference. Each derived store is rebuilt by replaying from Kafka — zero data is lost, any store can be rebuilt from scratch.

Kafka Event Topics
  • ride-events: RideRequested, DriverMatched, PickupStarted, PickupCompleted, RideCompleted, PaymentProcessed, RideRated
  • driver-location-events: LocationUpdated every 4s per driver (high frequency, compacted — only latest position per driver_id)
  • payment-events: ChargeInitiated, ChargeSucceeded, ChargeFailed, RefundRequested
Derived Stores (Each Built by a Flink Consumer)
  • PostgreSQL (OLTP): Ride records, user accounts, billing history. ACID transactions for payment disputes. Sharded by user_id. Flink consumer updates ride status from ride-events.
  • Redis Geo (driver matching): Flink consumes driver-location-events → GEOADD drivers:active {lat} {lon} {driver_id}. Driver matching API: GEORADIUS origin 2km ASC COUNT 10 returns nearest available drivers in O(N + log M). TTL auto-removes drivers who go offline (no heartbeat in 30s).
  • DynamoDB (driver earnings API): Flink aggregates earnings weekly per driver from payment-events → PK=driver_id SK=week_id → earnings. Driver earnings API reads at O(1), <5ms. No SQL aggregation at query time.
  • BigQuery/Redshift (analytics): Revenue per city per hour, driver utilization, surge pricing heatmaps. Fed by Flink CDC consumer reading from PostgreSQL + ride-events. Latency: minutes.
  • Elasticsearch (search): Pickup location autocomplete, ride history search for support teams. Updated via CDC from PostgreSQL.
  • Redis feature store (ML inference): ETA prediction features (time_of_day, distance, traffic_level, historical_segment_speed) precomputed by Flink → served to model inference API at <1ms.
Fault Tolerance
  • Any derived store can be rebuilt by replaying from Kafka: fix the Flink consumer bug → replay from offset=0 → correct store rebuilt. Kafka retention: 30 days for transactional events; indefinite for compacted topics (latest position per driver). This is the "unbundled database" pattern — Kafka is the WAL; stores are read-optimized views.
Interview differentiator: "The key architectural principle here is that Kafka acts as the distributed WAL — the single source of truth. All derived stores are caches of that truth. If any store becomes corrupt, incorrect, or needs a schema migration, you simply replay from Kafka. This is operationally far superior to tight coupling where services directly update each other's databases. Uber, LinkedIn, and Netflix all evolved to this architecture."
Q2Payment service sends duplicate charges during retries. Design a complete idempotency solution across HTTP, Kafka, and external gateways.
Answer

Duplicates enter at three distinct layers — each requires its own defense. Layer 1: HTTP client retries → idempotency key in header, DB-cached result. Layer 2: Kafka at-least-once redelivery → offset-based deduplication in the same DB transaction as the business effect. Layer 3: External gateway calls → propagate your idempotency key to the gateway. All three layers required; missing any one causes duplicate charges.

Layer 1 — HTTP Idempotency (Client → Service)
// Client: generate UUID before the intent (not before each attempt)
String idempotencyKey = UUID.randomUUID().toString();
// Persist it locally before making the call
// Send with EVERY attempt (retry = same key)
POST /payments
Idempotency-Key: a7f3-9b12-4c5d-...
{"amount": 100, "currency": "USD"}
Layer 2 — Service Implementation
@Transactional
public PaymentResult processPayment(String key, PaymentRequest req) {
    // 1. Check idempotency table FIRST
    Optional<PaymentResult> cached = idempotencyRepo.findById(key);
    if (cached.isPresent()) return cached.get();  // return same result, no charge

    // 2. Call gateway with SAME key (propagate to prevent double-charge at gateway)
    GatewayResponse gw = gateway.charge(req, Map.of("idempotency_key", key));

    // 3. Save business record + idempotency cache in ONE atomic transaction
    PaymentResult result = toResult(gw);
    paymentRepo.save(new Payment(req, result));   // business record
    idempotencyRepo.save(key, result);             // deduplication cache
    return result;
    // If service crashes after gateway charge but before commit:
    // → on retry, gateway returns same result (idempotent) via the same key
    // → DB transaction commits both records together
}
Layer 3 — Kafka Consumer Idempotency
  • Kafka at-least-once: consumer restarts replay unacked messages. Fix: use (partition, offset) as idempotency key. Before processing: check processed_offsets table. After processing: insert offset record in the SAME DB transaction as the business effect. On replay: offset already committed → skip. This gives exactly-once semantics without Kafka transactions.
Monitoring
  • Track idempotency cache hit rate. Normal: <0.1%. Hit rate > 1% → upstream retry policy is too aggressive (too many retries, too short timeout). Alert and tune.
Interview differentiator: "The idempotency key must be generated BEFORE the first attempt, not inside the retry loop. If you generate a new UUID on each retry, you get no deduplication. The key represents the intent ('charge this user for this cart'), not a single attempt. Stripe's documentation explicitly says: generate the key, persist it, then use it for all retries of that operation."
Q3GDPR right-to-erasure: a user requests deletion from your Kafka-backed event sourcing system. How do you comply?
Answer

Kafka is an immutable append-only log — you cannot delete individual messages. Physical deletion from all downstream stores (Elasticsearch, BigQuery, DynamoDB, S3 data lake) is O(N) systems and takes days. The correct solution is crypto-shredding: encrypt all PII with a per-user key from day one; on deletion, destroy the key. All ciphertext in all systems becomes permanently unreadable, simultaneously, in a single API call.

Design: Crypto-Shredding
  • At registration: Generate AES-256-GCM key per user → store in AWS KMS with IAM-controlled access. All PII fields encrypted before writing to Kafka: event.email = KMS.encrypt(user_key_id, "alice@example.com"). All events and all derived stores contain only ciphertext for PII fields (name, email, phone, address).
  • On deletion request (GDPR Article 17):
    • Destroy the KMS key: aws kms schedule-key-deletion --key-id {user_key_id} --pending-window-in-days 7
    • After 7 days: key permanently destroyed. All ciphertext in ALL systems (Kafka, Elasticsearch, BigQuery, S3) becomes permanent gibberish. Zero separate deletion operations required.
    • Publish a tombstone event to Kafka: key=user_id, value=null → triggers log compaction to remove old events for that user_id from Kafka log over time.
  • Non-PII retention: Aggregated analytics (revenue per city, product popularity heatmaps) contain no user-identifiable data → retained legally. Audit which derived views contain PII vs pure aggregates at design time.
  • Audit trail (no PII): Record: deletion_request_id, timestamp, SHA-256(user_id), key_id, key_deletion_confirmed_at. Demonstrates compliance to regulators without storing PII in the audit log.
Compliance Testing (Automated CI Check)
@Test
void gdprDeletionMakesDataInaccessible() {
    User user = createUser("alice@test.com");
    writeEvents(user.id, "purchase", "login", "profile_update");
    
    gdprService.requestDeletion(user.id);
    
    // All reads must return unreadable ciphertext or empty
    assertThat(kafka.read(user.id)).allMatch(e -> e.decryptedEmail() == null);
    assertThat(elasticsearch.getProfile(user.id).email()).isNull();
    assertThat(bigquery.getUserActivity(user.id)).isEmpty();
}
Interview differentiator: "Crypto-shredding is accepted as equivalent to physical deletion by multiple EU Data Protection Authorities (German DPA, French CNIL, UK ICO) as long as the encryption is strong (AES-256) and the key is truly destroyed (not just inaccessible). It's architecturally elegant: instead of hunting down user data across dozens of systems, you solve it once at the encryption layer and get compliance for all current and future systems automatically."
Q4Your service has p99 latency = 4 seconds but p50 = 50ms. Diagnose and fix using the three observability pillars.
Answer

A bimodal latency distribution (p50=50ms, p99=4s) means most requests are fast but ~1% are extremely slow. This is almost always caused by: N+1 queries, a missing database index, lock contention, or a slow downstream call. Use the three pillars in order: metrics → logs → traces. Each layer narrows the hypothesis until you find the root cause.

Pillar 1 — Metrics: "Something is wrong, and where"
  • Grafana p99 heatmap per endpoint. Identify the specific endpoint(s) where p99 diverges from p50. Correlate with: deployment timestamp, traffic volume spike, time of day (batch jobs running), specific AZ or region. Narrow from "the service is slow" to "the /user-profile endpoint is slow since yesterday's 2pm deploy."
Pillar 2 — Logs: "What is happening in slow requests"
-- CloudWatch Insights query to find slow request patterns
fields @timestamp, endpoint, latency_ms, db_query_count, user_id
| filter latency_ms > 1000
| stats count() as slow_count by endpoint, db_query_count
| sort slow_count desc

-- Typical finding:
-- endpoint=/api/user-profile  db_query_count=150  slow_count=450
-- endpoint=/api/user-profile  db_query_count=2    slow_count=0
-- → N+1 problem: 150 queries = loading user + 149 related posts individually
Pillar 3 — Traces: "Where does the time go"
Jaeger trace (slow request, 4020ms total):
  ┌─ POST /api/user-profile ─────────────────────── 4020ms ─┐
  │  ├─ AuthService ──── 12ms                                │
  │  └─ UserProfileService ────────────────── 4005ms ───────┤
  │     ├─ SELECT user WHERE id=123 ──── 3ms                 │
  │     ├─ SELECT post WHERE user=123 ── 4ms (post #1)       │
  │     ├─ SELECT post WHERE user=123 ── 4ms (post #2)       │
  │     └─ ... × 147 more identical queries ──── 3990ms ────┘
  Root cause: N+1 — 149 individual post queries, each 4ms × 149 = 596ms
  + connection pool wait: 3400ms (pool exhausted by concurrent N+1 requests)
Fix and Validate
  • N+1 → JOIN FETCH: SELECT u.*, p.* FROM users u LEFT JOIN posts p ON p.user_id=u.id WHERE u.id=123. 1 query instead of 150. p99: 4000ms → 8ms.
  • Missing index → EXPLAIN ANALYZE: CREATE INDEX CONCURRENTLY idx_posts_user_id ON posts(user_id). Sequential scan → index scan.
  • After fix: Monitor the p99 histogram. Bimodal distribution should collapse to unimodal. Track the improvement: 4s → 100ms = 40× better. Document in postmortem.
Interview differentiator: "The three pillars have a natural drill-down hierarchy: metrics tell you THAT something is wrong and WHICH component; logs tell you WHAT pattern differentiates slow from fast requests; traces tell you WHERE in the call graph the time is spent. You always start with metrics (cheapest to query) and only pull traces for specific slow requests (expensive, but pinpoint accurate). This structured approach finds root causes in minutes, not hours."
