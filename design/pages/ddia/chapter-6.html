Chapter 6 — Partitioning (Sharding)

Big Picture: A single machine can't store petabytes. Partitioning splits a large dataset across multiple nodes so data and query load scale horizontally. The goal: even distribution (no hot spots). The challenges: choosing the partition key, handling secondary indexes, and rebalancing without downtime as the cluster grows.

1 Partitioning by Key Range

  • Like an encyclopedia: A-C on Node 1, D-G on Node 2, etc. Boundaries chosen to spread data evenly (may not be alphabetically equal).
  • Advantage: efficient range queries — "all events from Jan 1-7" = only read the January partition.
  • Hot spot danger: if keys are sequential (e.g., timestamp), all recent writes go to the same partition. January 7th partition becomes a hot spot while others are idle.
  • Fix for timestamp keys: prefix with sensor/device ID: key = (sensor_id, timestamp). Writes for sensor 5 go to sensor-5 partition; writes for sensor 6 to sensor-6 partition → even distribution across all sensors.
  • Used by: HBase, Google BigTable, Google Spanner, RethinkDB, MongoDB (with range sharding).

2 Partitioning by Hash

  • Apply a deterministic hash function to the key → output uniformly distributed integers → assign to partitions by range.
  • Good hash functions: MD5 (MongoDB), Murmur3 (Cassandra), FNV (Voldemort). They distribute skewed inputs uniformly.
  • Advantage: no hot spots for write-heavy sequential keys. User IDs, UUIDs, product IDs all distribute evenly.
  • Disadvantage: range queries impossible without scatter-gather. "Users with ID 100-200" = must query all partitions.
Cassandra — Compound Primary Key: Best of Both Worlds
CREATE TABLE user_events (
  user_id    UUID,
  event_time TIMESTAMP,
  event_type TEXT,
  PRIMARY KEY (user_id, event_time)
);
-- user_id = PARTITION KEY → hashed → determines which node
-- event_time = CLUSTERING KEY → sorted within partition

-- Query: "Alice's events in the last 7 days" → one partition, range scan
SELECT * FROM user_events
WHERE user_id = alice_uuid
  AND event_time > now() - 7d;
-- → single partition lookup! O(log N + result_size)

-- Query: "All users' events in last 7 days" → scatter/gather all partitions
SELECT * FROM user_events WHERE event_time > now() - 7d;
-- → must hit ALL partitions (N queries, merge results) — unavoidable
-- → Cassandra does not support this efficiently. Use Spark batch job instead.
Hot Spot Mitigation — Key Salting
Salting a Hot Key (Celebrity Tweet Problem)
  Without salting: tweet_987654321 always hashes to Partition 7.
  Celebrity posts tweet → 50M reads/sec hit Partition 7 → hot spot.

  With salting (add random suffix 000-099):
  ─────────────────────────────────────────────────────────────
  WRITE:  pick suffix = random(0, 100)
          write tweet to:  tweet_987654321_042  → Partition 42
          (also write to:  tweet_987654321_015  → Partition 15, etc.)
          (or write to single suffix on write, read all on read)

  READ:   fetch tweet_987654321_000 through tweet_987654321_099 (100 reads)
          merge results (deduplicate, take consistent version)
  ─────────────────────────────────────────────────────────────
  Trade-off: 100× more read I/O to eliminate write hot spot.
  Only worth it for truly hot keys (celebrities, viral content, global events).
  Application must know WHICH keys are hot (use a hot-key detection algorithm).

4 Consistent Hashing & Rebalancing

Why NOT hash(key) mod N
  • Nodes = 10. hash(key) mod 10 → keys distributed. Add one node → mod 11 → ~90% of all keys now map to different partitions → massive data migration → cluster unusable for hours during rebalance.
Consistent Hashing Ring with Virtual Nodes (Cassandra)
  Hash ring: 0 ─────────────────────────────── 2^64
  Physical nodes assigned random positions (vnodes): 256 per node.

         vnode A3   vnode B2   vnode C1   vnode A1   vnode D3
  ────────●──────────●──────────●──────────●──────────●─────────

  Key routing: hash(key) → walk clockwise → first vnode encountered = owner

  Adding Node E (gets 256 random vnode positions):
  - Only keys between E's new vnodes and their predecessors move to E
  - Roughly 1/(N+1) = 1/6 of data moves (not 90% like mod N)
  - All other keys: unchanged. Zero disruption to the majority.

  Vnode advantages:
  - Even load distribution without manual tuning
  - If Node A has 2× capacity → give it 512 vnodes
  - Failure: 256 vnodes redistributed to all remaining nodes → smooth

🧠 FAANG Interview Questions

Q1Design the partitioning strategy for a WhatsApp-scale messaging app: 2B users, 100B messages/day.
Answer

Partition messages by (chat_id, year_month) as the compound partition key. This co-locates all messages in a conversation on the same shard (efficient range scan for history), uses the month bucket to prevent unbounded partition growth in high-traffic group chats, and enables time-based tiering of old data to cheap storage.

Partitioning Design
  • Messages table partition key: (chat_id, year_month) → hash(chat_id) determines node, year_month buckets within that node. Clustering key: message_id (time-ordered UUID or Snowflake ID) → messages sorted by time within partition.
  • Why not just chat_id? A popular group chat accumulates unlimited messages → one partition grows forever → hot shard. Month bucketing caps partition size at ~1M messages/group/month ≈ manageable.
  • User's inbox index: Separate table partitioned by user_id: (user_id) → [(chat_id, last_message_ts, unread_count)]. Needed to answer "show all conversations for user X." Updated asynchronously via CDC when new messages arrive.
  • Read receipts / delivery status: Separate table partitioned by message_id (different write pattern, different retention policy — receipts can be deleted after 30 days; messages kept 1+ year).
Hot Spot: Large Group Chats
  • Group with 10,000 members at 1000 messages/min → all reads go to same partition. Mitigation: local read replicas for chats with >1000 members; pre-computed materialized view of the last 50 messages in Redis cache for sub-millisecond reads.
Scale Numbers
  • 100B messages/day = 1.15M writes/sec avg, ~5M/sec peak. At 1KB avg = 100TB/day ingested. Cassandra 100 nodes RF=3: ~3 PB/year. Tiered storage: messages >90 days → S3 (100× cheaper). Active hot tier ≈ 100TB on 100 nodes = 1TB/node. Very manageable.
Interview differentiator: Proactively mention the compound partition key insight: "The year_month component prevents unbounded partition growth — a common production failure mode in messaging systems. Without it, a group chat that's been active for 5 years has a single multi-gigabyte partition that causes read timeouts and hot-node issues. This is WhatsApp's actual design pattern."
Q2Elasticsearch has a hot shard receiving 80% of write traffic. Diagnose the cause and fix it without reindexing.
Answer

A hot shard is almost always caused by a low-cardinality or monotonically-increasing routing key. Documents with timestamps or sequential IDs as routing keys all hash to the same shard because recent documents consistently map to the same hash bucket. Fix by adding a random suffix to the routing key, using routing_partition_size, or adopting an ILM rollover policy for time-series data.

Root Cause Diagnosis
  • Detect the hot shard:
    GET _cat/shards?v&h=index,shard,prirep,docs,store&s=store:desc
    One shard with 10× the docs/indexing rate of others = confirmed hot shard.
  • Most common cause — timestamp in _id: If document _id = 2026-02-27-{uuid}, the lexicographic sort makes all February 2026 docs hash similarly → hot shard for current month. Sequential auto-increment IDs have the same problem.
  • Second cause — explicit low-cardinality routing: PUT /events/1?routing=sports where "sports" is 80% of traffic → all sports events on one shard.
Fixes
  • Fix 1 — routing_partition_size (best for custom routing): In index settings, set routing_partition_size: 3. Documents with the same routing key are now distributed across 3 shards instead of 1. Searches for a routing key must scatter to 3 shards but writes are balanced. No reindex required if set at index creation.
  • Fix 2 — ILM rollover (for time-series): Create an ILM policy: rollover to a new index when current index reaches 50GB or 30 days old. New index gets fresh shards. Write alias always points to the active index. Historical indices are read-only. This is the standard pattern for log/event data.
  • Fix 3 — Force merge on old shards: Once an index rolls over and becomes read-only, run POST /old-index/_forcemerge?max_num_segments=1 to collapse many small Lucene segments into one → halves read latency for historical queries.
Interview differentiator: Explain that hot shards in Elasticsearch are a form of the same problem as hot partitions in Cassandra — both caused by a skewed key distribution. The universal fix is to introduce entropy (randomness or time-based rotation) into the partition key. "You're trading write locality for write balance. The cost is scatter-gather reads; the benefit is no single shard being a bottleneck."
Q3Walk through adding a new node to a live Cassandra cluster and the risks to manage.
Answer

Cassandra's virtual node (vnode) design makes node addition essentially zero-downtime. The new node claims ownership of random vnode token ranges, triggers automatic data streaming from existing owners, and seamlessly joins the cluster while it's serving production traffic. The key risks to manage are: streaming I/O impact on existing nodes, and ensuring nodetool repair completes after bootstrap.

Step-by-Step Process
  • Step 1 — Configure and start: Set cassandra.yaml with seed node addresses. Start the Cassandra process. Node appears in nodetool status as UJ (Up/Joining).
  • Step 2 — Automatic bootstrap: New node selects 256 random vnode positions on the consistent hash ring. For each claimed range, it sends StreamRequest to the current owner. Current owners stream their data to the new node while continuing to serve read/write traffic. Cluster is fully operational throughout.
  • Step 3 — Node becomes UN (Up/Normal): Now fully participates: accepts writes for its vnodes, serves reads. Gossip state propagates to all nodes within seconds.
  • Step 4 — Run nodetool repair: Catches any writes that arrived during bootstrap for the new node's ranges that might have been written via hinted handoff. Critical for eventual consistency correctness.
Risks and Mitigations
  • Streaming I/O spike: Bootstrapping saturates the network and disk I/O of source nodes. Limit with: nodetool setstreamthroughput 200 (200 Mbps cap). Always add nodes during off-peak hours and monitor source node disk I/O and CPU during bootstrap.
  • Read availability during bootstrap: The replication factor (RF=3) ensures other replicas cover reads for the new node's ranges. Read availability is maintained. Never add more than 1 node at a time to a cluster with RF=3 — reduces the safety margin.
  • Decommission (opposite operation): nodetool decommission — node streams its data to remaining nodes first, then cleanly leaves. Never just kill a Cassandra node — it won't stream data out and will leave "dead" vnodes that other nodes won't automatically claim.
Interview differentiator: Compare to MongoDB/Redis resharding: "In MongoDB with manual sharding, adding a shard requires rebalancing the chunk distribution map and manually migrating chunks — much more operator-involved. Cassandra's vnode-based consistent hashing makes this automatic and proportional. The interviewer will appreciate that you know why Cassandra's design makes rebalancing elegant, not just that it works."
