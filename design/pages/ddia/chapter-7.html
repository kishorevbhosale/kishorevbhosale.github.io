Chapter 7 — Transactions

Big Picture: A transaction groups reads and writes into a logical unit — all succeed (commit) or all fail (rollback). The hardest problem is isolation: preventing concurrent transactions from interfering. This chapter covers all 6 race conditions, all 4 isolation levels, and 3 concurrency control mechanisms — the most-tested topic in database design interviews.

1 ACID Properties

  • Atomicity: "All or nothing." Either all operations in the transaction commit, or none do. If any step fails → rollback → DB as if nothing happened. Mechanism: WAL (Write-Ahead Log). Key capability is the ability to abort. Note: Atomicity ≠ atomic CPU operations.
  • Consistency: Application invariants always hold (sum of all account balances unchanged, FK constraints satisfied). The DB enforces column constraints (NOT NULL, UNIQUE, FK); the app ensures higher-level invariants. C is actually an application responsibility — the DB can't enforce domain logic.
  • Isolation: Concurrent transactions behave as if they executed serially. Full isolation (Serializable) is expensive — most DBs offer weaker defaults. The hardest property to get right.
  • Durability: Once committed, data survives crashes (power loss, OS crash, HW failure). Mechanism: WAL flushed to disk before commit ACK + replication to multiple nodes.

2 The Six Race Conditions — Memorize These

These 6 race conditions are the core of every isolation interview question. Map each to its isolation level fix.
Race 1: Dirty Read
Dirty Read — Reading Uncommitted Data
  t=1: Tx1 updates account_balance: 1000 → 900  (not yet committed)
  t=2: Tx2 reads account_balance → 900           ← reads uncommitted data!
  t=3: Tx1 ROLLBACK (hardware error) → balance reverts to 1000
  t=4: Tx2 uses 900 to make a decision — based on data that NEVER EXISTED!
  ──────────────────────────────────────────────────────────────────────
  Analogy: seeing a colleague's half-written email before they've finished it.
  Prevented by: Read Committed isolation level and above.
Race 2: Dirty Write
  • Tx1 starts updating row. Before Tx1 commits, Tx2 overwrites the same row. Final state mixes writes from two transactions.
  • Example: Car listing. Tx1 (buyer Alice): updates listing.buyer=Alice + inserts invoice for Alice. Tx2 (buyer Bob): updates listing.buyer=Bob + inserts invoice for Bob. Tx2 overwrites Tx1's listing update. Result: listing.buyer=Bob but invoice=Alice → inconsistent.
  • Prevented by: Read Committed (row-level write locks).
Race 3: Read Skew (Non-Repeatable Read)
Read Skew — Backup Sees Inconsistent Snapshot
  Invariant: Account1 + Account2 should always = $1000 (transfer, not creation).
  Account1 = $500, Account2 = $500. Total = $1000 ✓

  t=1: Backup/Audit process reads Account1 → $500
  t=2: Transfer: Account1 $500→$400; Account2 $500→$600 (both committed)
  t=3: Backup reads Account2 → $600

  Backup sees: $500 + $600 = $1100 — "money appeared from nowhere!"
  The read at t=1 and t=3 are in different consistent states.
  ──────────────────────────────────────────────────────────────────────
  Real impact: bank audits fail; backup restoration produces wrong state.
  Prevented by: Snapshot Isolation (Repeatable Read). DB takes a point-in-time snapshot.
Race 4: Lost Update
Lost Update — Read-Modify-Write Cycle Race
  Wikipedia page edit conflict example:

  Tx1: read(page_content) → "Hello World"   Tx2: read(page_content) → "Hello World"
  Tx1: append "!"                           Tx2: append "?"
  Tx1: write("Hello World!")                Tx2: write("Hello World?")
  Tx1: commit                               Tx2: commit (OVERWRITES Tx1!)
  Final: "Hello World?" — Tx1's edit is LOST!

  ── Solutions ──────────────────────────────────────────────────────────────────
  Atomic operation: UPDATE page SET content = content || '!' WHERE id=42
                    → DB does the read-modify-write internally (atomic)

  Explicit lock:    BEGIN; SELECT content FROM page WHERE id=42 FOR UPDATE;
                    -- other transactions BLOCKED here
                    UPDATE page SET content = ... WHERE id=42; COMMIT;

  Compare-and-set:  UPDATE page SET content='Hello World!'
                    WHERE id=42 AND content='Hello World'
                    → fails if concurrent update happened (check rows_affected=0)

  OCC (Optimistic): detect at commit time via conflict detection
                    → abort Tx1 or Tx2, retry the loser.
Race 5: Write Skew — The Sneaky One
Write Skew — Hospital On-Call Scheduling
  Invariant: At least 1 doctor must be on-call at all times.
  Current state: Alice = on_call=true, Bob = on_call=true. Count = 2.

  Tx1 (Alice requests day off):   SELECT count(*) WHERE on_call=true → 2
                                  "Safe to go off call (count > 1)"
                                  UPDATE SET on_call=false WHERE doctor='Alice'

  Tx2 (Bob requests day off):     SELECT count(*) WHERE on_call=true → 2 (same snapshot!)
                                  "Safe to go off call (count > 1)"
                                  UPDATE SET on_call=false WHERE doctor='Bob'

  Both transactions commit independently (they touch DIFFERENT rows — no row conflict!)
  Final: Alice = false, Bob = false. Count = 0. INVARIANT VIOLATED!

  ── Why Snapshot Isolation fails here ──────────────────────────────────────────
  Each transaction writes to a different row → no write-write conflict detected.
  The problem is each read a STALE view of the count (the premise becomes false).
  ── Only fix ────────────────────────────────────────────────────────────────────
  SERIALIZABLE isolation. Or: SELECT ... FOR UPDATE on the count check (predicate lock).
Race 6: Phantom Read
  • Tx1 reads "are there any bookings for room 101 at 2pm?" → 0 rows → inserts booking. Tx2 does the same → both get 0 rows → both insert → double booking.
  • Phantom = a row that APPEARS due to another transaction's INSERT matching your WHERE clause. No existing row to lock — the absence is what matters.
  • Hard to prevent: you can't lock a row that doesn't exist yet. Solutions: predicate locks (lock the WHERE condition), index-range locks (approximate predicate at index level), serializable isolation, or materialization (pre-create lock rows for every possible time slot).
  • Prevented by: Serializable isolation only.

4 Concurrency Control Mechanisms

MVCC — How Snapshot Isolation Works
  • Every write creates a new version of the row (never overwrite in place). Each version tagged with: created_by_txid and deleted_by_txid.
  • Transaction Tx100 sees: all versions created by transactions with txid ≤ 100 that were committed before Tx100 started, and not deleted before Tx100 started.
  • Reader-writer non-blocking: readers see an older snapshot; writers create new versions. No lock contention between readers and writers. Long-running backups don't block OLTP writes.
  • PostgreSQL VACUUM cleans up old versions; MySQL InnoDB purge thread does the same.
2PL vs SSI
Two-Phase Locking (Pessimistic) vs SSI (Optimistic)
  2PL (Two-Phase Locking) — Pessimistic:
  ───────────────────────────────────────────────────────────────
  Read:  acquire SHARED lock (multiple readers OK; blocks writers)
  Write: acquire EXCLUSIVE lock (blocks all readers AND writers)
  All locks held until COMMIT (two phases: growing + shrinking)
  Deadlock: Tx1 holds row A, wants row B. Tx2 holds row B, wants row A.
  → DB detects cycle → aborts one → client must retry.
  Predicate locks: lock ALL rows matching WHERE condition (including future rows!)
  Index-range locks: approximate predicate at index granularity. More practical.

  SSI (Serializable Snapshot Isolation) — Optimistic:
  ───────────────────────────────────────────────────────────────
  Transactions run without blocking (based on MVCC snapshots).
  DB tracks "anti-dependencies": what did each Tx READ that another Tx WROTE?
  At commit time: if anti-dependency creates a cycle → abort one → retry.
  Example: if Tx1's read result was invalidated by Tx2's write → Tx1 aborts.
  ✅ No blocking → high throughput. No deadlocks (only aborts).
  ✅ Much better performance than 2PL when conflict rate is low.
  ❌ Under high contention: many aborts → retry overhead. 2PL may outperform.
  Used by: PostgreSQL SERIALIZABLE (since 9.1), FoundationDB.

🧠 FAANG Interview Questions

Q1Design a concert seat reservation system that prevents double-booking. Which isolation level and locking strategy?
Answer

Use SELECT FOR UPDATE (pessimistic locking) for the check-and-book operation. This acquires a row lock on the specific seat before checking availability, ensuring no other transaction can book it until yours commits or rolls back. For Ticketmaster-scale, back it with a Redis reservation hold (SETNX + TTL) as a fast pre-check layer.

The Race Condition
  • Two users check seat 14C simultaneously → both read status='AVAILABLE' → both decide to book → both UPDATE → last writer wins → one booking silently overwrites the other. This is a write skew on a single row with a check-then-act pattern.
Solution 1 — SELECT FOR UPDATE (Production Standard)
BEGIN;
-- Lock the row BEFORE checking availability
SELECT id, status FROM seats
WHERE seat_number = '14C' FOR UPDATE;
-- Other transactions trying to FOR UPDATE on seat 14C now BLOCK here.
-- Once unblocked, they'll see the committed status.

UPDATE seats SET status='BOOKED', user_id=:uid
WHERE seat_number='14C' AND status='AVAILABLE';
-- If 0 rows affected: another transaction already booked it → rollback + show error
COMMIT;
Solution 2 — Unique Constraint (Cleaner Design)
  • Create a bookings table with UNIQUE(seat_id). Two concurrent INSERTs for the same seat: first succeeds, second gets a unique constraint violation → database-enforced, no application lock needed. Simpler and scales better — database engine optimizes constraint enforcement.
Solution 3 — Serializable SSI (PostgreSQL)
  • SET TRANSACTION ISOLATION LEVEL SERIALIZABLE. PostgreSQL Serializable Snapshot Isolation detects the check-then-act anti-dependency and aborts one of the concurrent transactions. Application retries. Cost: ~10–20% higher latency, occasional serialization failures to handle. Correct by default.
Ticketmaster Scale — Redis Pre-Reservation
  • SET seat:14C user_id NX PX 300000 — atomic compare-and-set with 5-minute TTL. If NX succeeds, user has a reservation hold. Then DB transaction confirms booking before TTL expires. Redis SETNX is ~microseconds vs DB SELECT FOR UPDATE which requires disk I/O. Redis TTL auto-releases abandoned holds.
Interview differentiator: Identify the exact anomaly name: "This is a write-skew with phantom reads. SELECT FOR UPDATE prevents it by converting the optimistic read into a pessimistic lock acquisition, forcing serialization of the check-then-book pattern." Then proactively mention: "For high-concurrency situations, I'd prefer the unique constraint approach — it pushes conflict detection to the DB engine which is more efficient than application-level locks."
Q2Your bank's transfer service occasionally loses money — debits happen but credits don't. Walk through every bug and fix.
Answer

Transfer from Account A to Account B has five distinct failure modes. Every single one has been hit in production at real financial companies. The correct implementation uses: single DB transaction (wraps both debit + credit), SELECT FOR UPDATE (prevents lost update), idempotency key (prevents duplicate on retry), and Saga pattern (for cross-service transfers).

Bug 1 — Lost Update (most common)
-- WRONG: Two concurrent transfers both read balance=500, both compute 500-100=400
SELECT balance FROM accounts WHERE id=A;  -- sees 500
SELECT balance FROM accounts WHERE id=A;  -- other txn also sees 500
UPDATE accounts SET balance=400 WHERE id=A;  -- first: ok
UPDATE accounts SET balance=400 WHERE id=A;  -- second: WRONG (should be 300)

-- CORRECT: Atomic SQL update + SELECT FOR UPDATE
UPDATE accounts SET balance = balance - 100
WHERE id=A AND balance >= 100;  -- 0 rows affected = insufficient funds
Bug 2 — Crash Between Debit and Credit
  • Debit account A succeeds. Application crashes. Credit never executes. Fix: wrap both in a single @Transactional. DB WAL rolls back the debit on crash recovery. Atomicity.
Bug 3 — Read Skew in Audit
  • Audit reads A=500 → transfer runs (A→400, B→600) → audit reads B=500 (old snapshot). Reports total = 1100 (should be 1100). Fix: use REPEATABLE READ for audit transactions → consistent snapshot across the entire audit run.
Bug 4 — Duplicate Transfer on Retry
// CORRECT: Idempotency key prevents double-processing
@Transactional(isolation = Isolation.REPEATABLE_READ)
public void transfer(String transferId, long fromId, long toId, BigDecimal amount) {
    if (transferRepo.exists(transferId)) return; // same result, no double-charge
    Account from = accountRepo.findByIdForUpdate(fromId); // SELECT FOR UPDATE
    Account to   = accountRepo.findByIdForUpdate(toId);   // consistent ordering avoids deadlock
    from.debit(amount);
    to.credit(amount);
    transferRepo.save(new Transfer(transferId, ...)); // idempotency record
} // autocommit on success, autorollback on exception
Bug 5 — Cross-Database Transfer (A in DB1, B in DB2)
  • No single ACID transaction can span two databases. Use Saga pattern: Debit A (DB1) → publish event → Credit B (DB2). If B credit fails → compensating transaction: Re-credit A. Use the outbox pattern to guarantee event delivery. This gives eventual consistency, not immediate consistency — acceptable for inter-bank transfers with audit trails.
Interview differentiator: Mention the deadlock risk in Bug 4: "I acquire locks on accounts in consistent order (lower ID first) to prevent deadlocks. If transfer T1 locks A then B, and transfer T2 locks B then A, they deadlock. Always establish a consistent lock ordering."
Q3What is MVCC? How does PostgreSQL implement it, and what operational costs does it introduce?
Answer

MVCC (Multi-Version Concurrency Control) provides non-blocking reads by maintaining multiple versions of each row — each transaction sees a consistent snapshot as of its start time. PostgreSQL implements this via hidden xmin/xmax columns on every tuple. The cost: dead tuples accumulate on disk, requiring VACUUM to reclaim space, and transaction ID wraparound must be managed at high TPS.

How PostgreSQL MVCC Works
  • Every heap tuple has: xmin (TX ID that created this version) and xmax (TX ID that deleted/superseded this version, 0 = current).
  • Transaction Tx100 sees a tuple if: xmin <= 100 AND (xmax = 0 OR xmax > 100) AND xmin's transaction is committed.
  • On UPDATE: old row gets xmax = current_txid; new row created with xmin = current_txid. Both exist on disk simultaneously during the transaction's lifetime.
  • Key benefit: Readers and writers never block each other. A long-running report sees a snapshot from its start time while OLTP writes proceed at full speed — no shared locks, no reader-writer deadlocks.
Operational Costs
  • Dead tuple bloat: Superseded row versions accumulate on disk until VACUUM removes them. A table updated 1000 times/second grows unboundedly without VACUUM. Monitor n_dead_tup in pg_stat_user_tables. Alert if dead tuples > 20% of live tuples.
  • Transaction ID wraparound: PostgreSQL uses 32-bit XIDs (4B values). At 10,000 TPS: 4B / 10K = ~115 hours = wraparound in 4.8 days without freezing! autovacuum_freeze_max_age (default: 200M txns) forces freezing of old tuples. High-TPS systems must aggressively tune autovacuum or run manual VACUUM FREEZE. Failure to do so causes database-wide downtime for emergency vacuuming.
  • Table bloat on write-heavy tables: Run VACUUM FULL (rewrites table, holds lock) or pg_repack (online, no downtime) to reclaim space after large bulk updates/deletes.
Interview differentiator: The XID wraparound point shows deep PostgreSQL knowledge. "At Cloudflare's scale, they ran into XID wraparound in a high-TPS Postgres cluster — it forced emergency vacuuming that took the DB offline for several hours. This is why monitoring age(datfrozenxid) from pg_database is a critical production alert: if it approaches 2B, you're in danger."
Q4Two doctors simultaneously book overlapping appointments. Snapshot Isolation doesn't prevent it — why not, and what's your fix?
Answer

This is a write skew anomaly. Under Snapshot Isolation (SI), each transaction reads a consistent snapshot, decides the slot is free, and writes to different rows. SI cannot detect that two independent writes together violate a global constraint. The fix is: either use Serializable isolation, pre-materialize lock rows, or use PostgreSQL's exclusion constraints.

Why Snapshot Isolation Fails
  • Transaction A: reads slots 2pm–3pm (free) → books 2pm–3pm row (inserts new row)
  • Transaction B: reads slots 2:30pm–3:30pm (free, A's booking not yet visible) → books 2:30pm–3:30pm row (inserts different row)
  • Both commit. Each read was consistent. Each write went to a different row. No conflict detected. Calendar now has two overlapping appointments.
Fix 1 — Materialization of Conflicts (No Serializable Needed)
-- Pre-create time slot rows for every 15-minute interval
CREATE TABLE appointment_slots (
  doctor_id INT, date DATE, slot_time TIME, is_booked BOOL,
  PRIMARY KEY(doctor_id, date, slot_time));

BEGIN;
-- Lock ALL slot rows that overlap with requested range
SELECT * FROM appointment_slots
  WHERE doctor_id=:doc AND date=:date
    AND slot_time BETWEEN '14:00' AND '15:00'
  FOR UPDATE;  -- blocks concurrent booking of any overlapping slot
UPDATE appointment_slots SET is_booked=true WHERE ...;
INSERT INTO appointments(...);
COMMIT;
Fix 2 — PostgreSQL Exclusion Constraint (Most Elegant)
ALTER TABLE appointments
ADD CONSTRAINT no_overlap
EXCLUDE USING GIST (
  doctor_id WITH =,
  tsrange(start_time, end_time) WITH &&
);
-- DB-enforced: any INSERT/UPDATE that creates an overlap for same doctor raises error
-- Application just catches the constraint violation and shows "slot taken"
Fix 3 — Serializable Isolation (Full Correctness)
  • SET TRANSACTION ISOLATION LEVEL SERIALIZABLE. PostgreSQL SSI detects the anti-dependency (A's read depends on data B didn't write, B's read depends on data A didn't write → cycle → conflict) and aborts one transaction. Application retries. ~15% latency overhead + occasional ERROR: could not serialize access to handle.
Interview differentiator: "The exclusion constraint solution is particularly satisfying — it's declarative (you describe the invariant, not the implementation), enforced by the database at all times regardless of application code, and based on the GIST index which handles interval overlap efficiently. This is the solution I'd recommend for a booking system because it eliminates an entire class of bugs permanently."
