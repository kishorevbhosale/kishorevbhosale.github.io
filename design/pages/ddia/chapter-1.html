Chapter 1 — Reliable, Scalable & Maintainable Applications

Big Picture: A data-intensive application is bottlenecked not by CPU but by volume, complexity, and speed of data. Modern apps combine databases, caches, search indexes, message queues, and batch processors. The developer's job is to compose them into a system that is reliable (correct despite faults), scalable (copes with growth), and maintainable (productive for future engineers).

1 Reliability

The system continues to work correctly at the desired performance level even when things go wrong (hardware faults, software bugs, human errors).

1.1 Fault vs. Failure
  • Fault: one component deviates from spec (e.g., a disk corrupts a byte, a network packet dropped, a process crashes).
  • Failure: the whole system stops providing required service to users.
  • Goal: build fault-tolerant systems — prevent faults from cascading into failures. Accept faults; prevent failures.
  • Netflix Chaos Monkey: deliberately kills random production instances to force teams to build resilient systems. "If it can fail, it will fail."
1.2 Hardware Faults
  • HDD MTTF = 10–50 years. In a 10,000-disk cluster → one disk dies every single day on average.
  • Traditional: RAID, dual PSU, ECC RAM — single machine reliability via redundant components.
  • Cloud-native: commodity VMs with higher individual failure rate. Software handles failures automatically (Auto Scaling Group replaces unhealthy instances).

1.3 Software Faults (Systematic)
  • Hardware faults are random and independent. Software faults are correlated — one bug crashes every node simultaneously.
  • Examples: runaway process consuming all RAM/CPU; slow dependent service cascading timeouts; leap-second kernel bug (crashed thousands of Linux servers, June 2012).
  • Mitigations: careful testing at every layer, process isolation (containers), restart on crash, sanity-check invariants in production (messages_in ≈ messages_out).
1.4 Human Errors — #1 Cause of Outages
Studies show operator configuration errors cause the majority of outages — more than hardware failures and software bugs combined.
  • Minimize error surface: well-designed APIs with sensible defaults, type safety, principle of least surprise.
  • Sandbox environments: staging that mirrors production with realistic data; test every deployment there first.
  • Fast rollback: feature flags + canary deploys → 1-click revert. Blue-green deployments.
  • Telemetry: metrics, logs, traces. Detect and diagnose problems within seconds of occurring.
  • Runbooks: documented on-call playbook for every known failure mode. No "call Dave."

2 Scalability

As the system grows in data volume, traffic volume, or complexity, there should be reasonable ways to deal with that growth. Scalability is not a yes/no binary — it is: "if we grow in this specific way, what options do we have?"

2.1 Describing Load — Load Parameters
  • Precisely describe current load before discussing performance changes. Use load parameters relevant to your system.
  • Web server → requests/second (RPS), peak concurrent connections.
  • Database → read/write ratio, working set size, cache hit rate.
  • Chat app → simultaneously online users, message fanout factor per user.
2.2 Twitter Timeline — The Classic Case Study
Twitter Home Timeline — 3 Architecture Approaches
  Approach 1: Pull-on-read (simple JOIN)
  ────────────────────────────────────────────────────────────
  SELECT t.* FROM tweets t
  JOIN follows f ON f.followee_id = t.user_id
  WHERE f.follower_id = :current_user
  ORDER BY t.created_at DESC LIMIT 100;
  Problem: 300K reads/sec × complex multi-million-row JOIN = DB overwhelmed.

  Approach 2: Fan-out on write (pre-computed caches)
  ────────────────────────────────────────────────────────────
  On new tweet: push tweet_id into each follower's timeline cache.
  Read = simple cache GET, O(1).
  Problem: user with 30M followers → 30M cache writes per tweet.
           Barack Obama tweets = 30M writes in seconds. Unacceptable spike.

  Approach 3: Hybrid (Twitter's actual production system)
  ────────────────────────────────────────────────────────────
  Normal users (< ~1M followers) → fan-out on write (pre-populated Redis lists)
  Celebrities (> 1M followers)   → pull-on-read merged at query time
  At timeline load: merge pre-computed cache + live celebrity tweets
  Key insight: architecture depends on the DISTRIBUTION of follower counts,
               not just the average. 99.9% of users can use fan-out safely.
2.3 Describing Performance — Always Use Percentiles
  • Response time is a distribution, not a single number. Report percentiles: p50, p95, p99, p999.
  • p50 (median): half of requests are faster than this. p99: 1 in 100 requests is slower. p999: 1 in 1,000.
  • Amazon found 100ms added latency costs ~1% in sales. Your most valuable customers tend to have the most data → they hit tail latency (p99+) most often.
Head-of-Line Blocking in Microservices
  Frontend makes 5 parallel backend calls:
    UserService:     12ms  ✓
    ProductService:  10ms  ✓
    PriceService:     8ms  ✓
    InventoryService:11ms  ✓
    ReviewService:  950ms  ← hit a p99 event (GC pause, slow DB query)

  User sees: 950ms total response time.
  Even though 4 out of 5 services responded in ~10ms.

  Lesson: fan-out × (backend p99 latency) = your user's experience.
  The more microservice calls you make, the more likely you hit a tail event.
  → Measure end-to-end p99 and p999, not just individual service p50.
2.4 Scale Up vs Scale Out
  • Vertical scaling (scale up): bigger machine, more cores/RAM. Simpler operationally; no distribution headaches. Good starting point. Has a hard upper limit and cost curve.
  • Horizontal scaling (scale out): add more machines. Elastic and theoretically unlimited. Required at internet scale.
  • Stateless services (API/web servers): trivial to scale out — add instances, put behind load balancer. Auto Scaling Groups.
  • Stateful systems (databases): much harder — data must be partitioned, replicas must stay consistent, re-sharding during scaling is disruptive.
Don't over-engineer for 100× scale before you have evidence. The architecture that works at 1× often looks completely different at 100×. Revisit and evolve as you grow with actual data to guide decisions.

3 Maintainability

Many different people will work on the system over time. They should all be able to work on it productively. Majority of software cost is ongoing maintenance, not initial development.

3.1 Operability
  • Good observability: dashboards, distributed traces, structured logs — show internal system behavior without requiring SSH access.
  • Standard tooling: use well-understood platforms (Kubernetes, Prometheus) rather than bespoke automation scripts.
  • Predictable behavior: system behaves consistently under load; no mysterious "restart it and it works" problems.
  • Self-healing: auto-restart crashed processes, retry transient failures, auto-scale. Operator can override when needed.
3.2 Simplicity — Kill Accidental Complexity
  • Essential complexity: irreducible, inherent in the problem (concurrent users in a booking system is genuinely hard).
  • Accidental complexity: added by bad implementation — poor abstractions, tight coupling, state explosion, inconsistent naming. Eliminate this!
  • Best tool: good abstraction hides complex implementation behind a clean interface. SQL hides B-Trees, WAL, buffer pools, lock managers. Every developer benefits without knowing internals.
3.3 Evolvability — Embrace Change
  • Requirements are never static: new features, new regulations (GDPR), new scale, new platforms, new business models.
  • Technical enablers: TDD, refactoring, feature flags, clean abstractions, modular architecture, CI/CD pipelines.
  • Warning sign of poor evolvability: "We can't change X because it'll break Y, Z, and W." = brittle architecture.

🧠 FAANG Interview Questions

Q1What is the difference between fault tolerance and high availability? How are they achieved in production systems?
Answer

Fault tolerance means the system keeps serving requests correctly with zero interruption even when a component fails. High Availability (HA) means the system recovers quickly from failures — typically within seconds to a few minutes. Fault tolerance is strictly harder and more expensive than HA.

Technical Mechanism
  • HA implementation: active-passive failover — a standby replica takes over when the primary fails. AWS Multi-AZ RDS detects primary failure via heartbeat, promotes the standby, updates the DNS CNAME, and your app reconnects — total downtime ~30–60 s. Your RTO = 1 min, RPO = seconds (async lag).
  • Fault tolerance implementation: active-active with synchronous replication. AWS Aurora writes to 6 copies across 3 AZs before acknowledging. You can lose a full AZ and keep writing with zero interruption. DynamoDB global tables replicate across regions in <1 s; a region failure is transparent to the client.
  • RTO vs RPO: HA targets RTO < 1 min, RPO < 1 min. Fault tolerance targets RTO = 0 s, RPO = 0. Every design decision on replication (sync vs async) is a trade-off between these two metrics and write latency.
  • Cost of synchronous replication: requires acknowledgement from all replicas before commit → adds network round-trip to every write. AWS Aurora solves this by acknowledging after 4-of-6 replicas respond, keeping write latency under 5 ms while still tolerating 2 AZ failures.
Real-World Example

Netflix runs active-active across 3 AWS regions. During a regional outage, traffic automatically routes to the two remaining regions with no customer-visible downtime. They achieve this using Eureka service registry, Zuul load balancer, and Cassandra's multi-region replication — all designed for fault tolerance, not just HA.

Interview differentiator: Always define RTO and RPO explicitly. Most engineers know "fault tolerance vs HA" — the ones who stand out quantify the trade-off: "synchronous replication eliminates RPO but adds 5–10 ms write latency; most systems accept async replication with a 1–5 s RPO to stay under 1 ms write overhead."
Q2Twitter has 300M active users. Designing the home timeline — which fan-out strategy would you use and why?
Answer

Use a hybrid fan-out: fan-out on write for regular users (pre-computed Redis timelines), pull-on-read for celebrities (>1M followers) merged at query time. This is exactly what Twitter's infrastructure evolved to after failing with pure push and pure pull.

Technical Mechanism
  • Fan-out on write (regular users): When Alice tweets, iterate her followers and write ZADD timeline:{follower_id} {tweet_timestamp} {tweet_id} to Redis sorted sets. Read = ZRANGE timeline:{user_id} 0 49 REV — O(log N) write-time work, O(1) read-time work. Works for 99.9% of users (avg ~200 followers → 200 Redis ops per tweet).
  • Pull-on-read (celebrities): Katy Perry has 109M followers — fan-out would queue 109M Redis writes per tweet. Instead, skip fan-out; at read time, fetch the user's normal timeline + query the last 50 tweets from each followed celebrity and merge by timestamp. The merge is O(C log C) where C = number of followed celebrities (typically <10).
  • Threshold: experimentally determined. Twitter set ~1M followers as the cutoff. Above that, the fan-out cost (latency spike, Redis memory) outweighs the merge cost at read time.
  • The key insight: the right load parameter is the distribution of follower counts, not the average. Designing for the 99th percentile user (heavy follower) is what makes the system work at scale.
Data Structure
ZADD timeline:{user_id} {unix_ms_timestamp} {tweet_id}  → write
ZRANGE timeline:{user_id} 0 49 REV WITHSCORES             → read (latest 50)
Interview differentiator: The interviewer expects you to identify the celebrity/hotspot problem. Go further — mention that celebrities must be identified dynamically (follower count can change), so use a background job that monitors follower counts and toggles the fan-out flag. Twitter calls these accounts "heavy hitters."
Q3Your service SLA requires p99 < 200ms. A latency regression is reported. Walk through measurement, alerting, and debugging.
Answer

Use Prometheus histograms (not summaries) to measure latency distribution per endpoint, alert at 150ms to get ahead of SLA breach, and debug via distributed traces when the alert fires. This three-layer observability stack — metrics → logs → traces — lets you go from "p99 is slow" to "this specific DB query on this specific host" in under 10 minutes.

Measurement
  • Instrument every HTTP handler with a Micrometer/Prometheus histogram: buckets at [5, 10, 25, 50, 100, 200, 500, 1000, +Inf] ms. Histograms (unlike summaries) can be aggregated across instances — critical when you have 50 pods.
  • Recording rule: histogram_quantile(0.99, sum(rate(http_duration_bucket[5m])) by (le, endpoint))
  • Alert at p99 > 150ms for >2 consecutive minutes — 50ms headroom gives you time to react before SLA breach.
Debugging Runbook
  • Step 1: Which endpoint is slow? (Grafana heatmap per endpoint)
  • Step 2: Correlate with error logs — is there a spike in DB timeouts, 503s from downstream, or GC pause logs?
  • Step 3: Pull a distributed trace (Jaeger / AWS X-Ray) for a slow request — see the waterfall. The slow span reveals the bottleneck: DB query? External API call? Serialization?
  • Common culprits: N+1 query (solve with JOIN or batch), missing DB index (EXPLAIN ANALYZE), lock contention (switch to MVCC or reduce txn scope), GC pause (tune heap or switch to G1GC), downstream service degradation (add circuit breaker + timeout).
  • Head-of-line blocking: if your service calls 10 downstream services serially, p99 ≈ sum of all p99s. Switch to parallel fanout with CompletableFuture.allOf() or reactive streams.
Interview differentiator: Most engineers stop at "add a Prometheus alert." The strong signal is mentioning that average latency is useless — 1% of requests can be 10× slower and the average barely moves. Always use percentiles. Also mention that tail latency gets amplified in fanout: with 10 downstream calls, if each has 1% chance of being slow, you have a ~10% chance of a slow composite request.
Q4You're on-call and your service is completely down. Walk through your incident response, step by step.
Answer

Follow Mitigate → Diagnose → Fix → Learn. The #1 mistake engineers make is spending 30 minutes debugging root cause while users are down — always try mitigation first (rollback, traffic shift, feature flag) before you start spelunking in logs.

Step-by-Step Playbook
  • 0–2 min — Triage: Scope the blast radius. What percentage of requests are failing? Which regions? What's the error code (500? 503? timeout?)? Declare severity (P0 = total outage, P1 = major degradation) and notify stakeholders immediately — don't wait until you know the cause.
  • 2–5 min — Fast mitigations (try in order): (1) Did a deploy happen in the last hour? → rollback. (2) Did a config change go out? → revert. (3) Is it overload? → enable rate limiting / shed load. (4) Can traffic route to another region? → shift 100% there. Mitigation in 5 minutes beats root cause analysis in 30.
  • 5–20 min — Diagnose: Once mitigated (or if no fast mitigation works), collect signals: error logs, DB connection pool exhaustion, downstream service health, CPU/memory anomalies. Form one hypothesis and test it. Never change two things simultaneously.
  • 20 min+ — Fix and verify: Apply targeted fix, monitor error rate returning to baseline. Keep stakeholders updated every 15 minutes even if just "still investigating."
  • Post-incident (within 48 h): Blameless post-mortem: 5-why analysis, timeline, action items with owners and dates. Share across teams — knowledge must transfer.
Interview differentiator: Explicitly say "mitigation before diagnosis" — this is the senior-engineer instinct. Also mention canary deployments and automated rollback triggers (e.g., if error rate > 1% for 2 min, auto-rollback) as prevention. This shows you've dealt with real incidents, not just read about them.
Q5What makes a system "maintainable"? How does good abstraction reduce long-term engineering cost?
Answer

Maintainability has three pillars: operability (system tells you what's wrong), simplicity (eliminate accidental complexity), and evolvability (changing one thing doesn't break five others). The ROI of investing in these compounds over time — a well-maintained codebase lets 10 engineers do the work of 100 in a poorly maintained one.

Operability
  • System must expose its own health: structured logs (JSON, not printf), metrics (RED: Rate, Errors, Duration per service), distributed traces, and runbooks linked from alerts. Operators should never need to SSH into a box to understand what's happening.
Simplicity — Eliminating Accidental Complexity
  • Essential complexity is irreducible (a booking system has genuinely complex concurrent-seat logic). Accidental complexity is self-inflicted: poor naming, shared mutable state, implicit dependencies, monolithic databases shared across teams. The goal is ruthlessly eliminating the second kind.
  • Test for accidental complexity: if onboarding a new engineer takes >2 weeks before they can make a safe production change, the system has too much accidental complexity.
Good Abstraction — Concrete Value
  • SQL hides B-Trees, WAL, buffer pools, MVCC, and lock managers behind a clean declarative interface. Every engineer who uses PostgreSQL benefits from 30 years of optimization without knowing a single detail of the storage engine. That's a force multiplier worth billions.
  • Signs of a leaky abstraction: "You need to use the NOLOCK hint on Tuesdays when batch jobs run" or "don't call this API more than 10 times per second or it'll crash." These reveal that implementation details have escaped the abstraction boundary.
  • Evolvability metrics: mean time to ship a feature end-to-end; number of teams that must approve a change; blast radius of a mistake. If these are high, the architecture is resisting change.
Interview differentiator: Frame abstraction as a business decision, not just a technical one. "A good abstraction lets 5 engineers build features that would otherwise require 50. The cost of maintaining bad abstractions is paid every sprint, in every standup, forever."
