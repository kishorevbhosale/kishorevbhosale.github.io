What is Spring Framework? How is it different from Spring Boot?
  • Spring Framework is a comprehensive Java application framework providing infrastructure support for building enterprise applications. It covers Dependency Injection (DI), Aspect-Oriented Programming (AOP), transaction management, MVC, security, data access, and more — but requires significant manual configuration.
  • Spring Boot is an opinionated extension of the Spring Framework that eliminates boilerplate setup. It provides auto-configuration, embedded web servers (Tomcat/Jetty), starter dependencies, and production-ready features out of the box.
AspectSpring FrameworkSpring Boot
ConfigurationRequires explicit XML or Java configAuto-configured based on classpath
Web ServerExternal server required (deploy WAR)Embedded server (Tomcat/Jetty/Undertow)
StartupMore boilerplate setupConvention over configuration; rapid start
Dependency MgmtManual version managementStarter POMs with tested, compatible versions
Production FeaturesMust integrate manuallyActuator built-in for health, metrics, info
Interview tip: Emphasize that Spring Boot doesn't replace Spring — it orchestrates it. All Spring modules remain available; Spring Boot just removes the configuration tax so developers focus on business logic.

Why was Spring Boot introduced? What problems does it solve?
  • Problem 1 — Configuration Overload: Traditional Spring required hundreds of lines of XML (beans, data sources, transaction managers, view resolvers). Spring Boot auto-configures all of these based on what's on the classpath.
  • Problem 2 — Dependency Hell: Choosing compatible versions of Spring, Hibernate, Jackson, etc. was error-prone. Spring Boot Starters provide curated, tested dependency sets.
  • Problem 3 — Deployment Complexity: WAR files required an external application server. Spring Boot bundles an embedded server, producing a self-contained executable JAR.
  • Problem 4 — Production Readiness: Adding health checks, metrics, and monitoring required separate tooling. Spring Boot Actuator provides these out of the box.
  • Problem 5 — Slow Developer Feedback Loop: Spring Boot DevTools enables automatic restarts on code changes, significantly improving developer productivity.
Interview tip: Summarize with: "Spring Boot was born to make the 80% use case zero-config while keeping full Spring flexibility for the 20% requiring customization."

What are the advantages of using Spring Boot in enterprise applications?
  • Rapid Development: Minimal configuration; a production-ready REST API can be running in minutes with Spring Initializr.
  • Auto-Configuration: Intelligently configures beans (DataSource, JPA, Security, MVC) based on classpath dependencies and application properties.
  • Embedded Server: No need to install and manage external servers. JARs are self-contained and easy to containerize (Docker).
  • Starter Dependencies: Pre-tested, compatible dependency bundles eliminate version conflicts and reduce Maven/Gradle complexity.
  • Actuator: Built-in health checks, metrics, environment info, and thread dumps — critical for production operations.
  • Microservices-Ready: Spring Cloud integrates seamlessly, enabling service discovery, config server, circuit breakers, and API gateways.
  • Testability: First-class testing support via @SpringBootTest, @DataJpaTest, MockMvc, and Testcontainers.
  • Active Ecosystem: Broad community, frequent releases, extensive documentation, and enterprise adoption.
Interview tip: Relate advantages to real scenarios — e.g., "In my last project, Spring Boot DevTools reduced restart time from 30 seconds to under 3 seconds, improving daily developer productivity significantly."

Explain the differences between Spring, Spring MVC, and Spring Boot.
  • Spring (Core Framework): The foundation — provides the IoC container, Dependency Injection, AOP, core utilities, and integration modules (Spring Data, Spring Security, Spring Batch, etc.). You use Spring when you need any of these capabilities.
  • Spring MVC: A web framework built on top of Spring Core. Implements the Model-View-Controller pattern for building web apps and REST APIs. Uses DispatcherServlet as the front controller that routes requests to @Controller handlers. Requires manual configuration of DispatcherServlet, view resolvers, and message converters.
  • Spring Boot: A bootstrap layer on top of both Spring Core and Spring MVC (and all other Spring projects). Provides auto-configuration, starter dependencies, and embedded servers so developers get a working Spring MVC (or WebFlux) application without any configuration files.
Analogy: Spring Core is the engine, Spring MVC is the car body for web apps, and Spring Boot is the fully assembled car with GPS and auto-everything — ready to drive.
5
How does Spring Boot simplify dependency management?
  • Spring Boot Parent POM: Inheriting from spring-boot-starter-parent brings a curated Bill of Materials (BOM) that pins compatible versions for hundreds of libraries — Hibernate, Jackson, SLF4J, Lettuce, etc. Developers declare dependencies without specifying versions.
  • Starter Dependencies: Each spring-boot-starter-* is a single dependency that transitively pulls all required libraries. For example, spring-boot-starter-web includes Spring MVC, Jackson, Tomcat, and Validation — no manual wiring.
  • BOM Import: Non-Boot projects can import spring-boot-dependencies BOM to get the same version management without the parent POM.
  • No Version Conflicts: Since all versions are tested together by the Spring team, you rarely encounter ClassNotFoundException or binary incompatibility issues.
<!-- Just declare the feature, not the version -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
</dependency>
Interview tip: Mention that you can still override a specific version in properties (e.g., <jackson.version>2.16.0</jackson.version>) when you need a particular bug fix or feature.

What is an embedded server in Spring Boot? Which servers are supported?
  • An embedded server is a web server packaged inside the application's JAR file. Spring Boot starts this server programmatically at application startup — no external Tomcat/JBoss/WebLogic installation is required.
  • Supported servers:
    • Apache Tomcat (default) — included via spring-boot-starter-web. Battle-tested, thread-per-request model.
    • Jetty — lightweight, good for embedded/IoT scenarios. Exclude Tomcat and add spring-boot-starter-jetty.
    • Undertow (JBoss) — high performance, non-blocking I/O, lower memory footprint. Used heavily in reactive apps.
    • Reactor Netty — used automatically with spring-boot-starter-webflux for reactive (non-blocking) applications.
  • Configuration: server.port, server.servlet.context-path, SSL via server.ssl.* in application.properties.
  • Switching servers: Exclude the default Tomcat dependency and add the desired server starter.
Interview tip: The embedded server model is a key enabler of containerization — a Spring Boot JAR inside a Docker container is completely self-sufficient, which is why it maps so well to Kubernetes deployments.

Explain the concept of convention over configuration in Spring Boot.
  • Core Idea: Spring Boot applies sensible, tested defaults so developers only need to configure things when they deviate from the convention. The framework "guesses" what you need based on available libraries on the classpath.
  • Examples:
    • If spring-boot-starter-web is on the classpath → Spring Boot configures a DispatcherServlet, MVC, Jackson, and Tomcat on port 8080.
    • If spring-boot-starter-data-jpa is present and H2 is on the classpath → an in-memory DataSource and EntityManagerFactory are configured automatically.
    • If spring-security is on the classpath → all endpoints are secured with Basic Auth by default.
  • Overriding conventions: Any auto-configured bean can be overridden by declaring your own bean of the same type. @ConditionalOnMissingBean ensures auto-config backs off when a user-defined bean exists.
Interview tip: This principle dramatically reduces the barrier to entry for new projects. The key insight is that conventions are not restrictions — they are simply defaults that can always be overridden with a single property or bean declaration.

What are Spring Boot Starters? Give examples of commonly used ones.
  • Definition: Spring Boot Starters are pre-packaged dependency descriptors. Each starter pulls in all the libraries needed for a specific functionality — tested and version-aligned by the Spring team.
  • Common starters and what they include:
    • spring-boot-starter-web — Spring MVC, Tomcat, Jackson, Validation
    • spring-boot-starter-data-jpa — Hibernate, Spring Data JPA, transaction management
    • spring-boot-starter-security — Spring Security (authentication, authorization)
    • spring-boot-starter-test — JUnit 5, Mockito, AssertJ, Spring Test utilities
    • spring-boot-starter-actuator — Health, metrics, info, env endpoints
    • spring-boot-starter-cache — Spring Cache abstraction with ConcurrentHashMap by default
    • spring-boot-starter-webflux — Project Reactor, Netty (reactive non-blocking)
    • spring-boot-starter-amqp — RabbitMQ integration
    • spring-boot-starter-data-redis — Lettuce Redis client
Interview tip: Starters follow the naming convention spring-boot-starter-{feature}. Third-party starters use {name}-spring-boot-starter (e.g., mybatis-spring-boot-starter).

What are Spring Boot DevTools? Why are they used?
  • Spring Boot DevTools (spring-boot-devtools) is a development-only module that enhances developer productivity.
  • Key features:
    • Automatic Restart: Monitors the classpath for changes. When a class changes and is compiled, DevTools restarts the application using two classloaders — the "base" classloader (third-party JARs, rarely changed) and the "restart" classloader (your code). Only the restart classloader is reloaded, making restarts ~5-10x faster than a cold start.
    • LiveReload: Triggers browser refresh on resource changes (HTML, CSS, JS) via an embedded LiveReload server.
    • Disabled Caches: Thymeleaf template caching, Spring MVC resource caching are disabled — changes reflect immediately without restart.
    • Remote DevTools: Enables remote restart for apps deployed on remote servers (used cautiously).
  • Automatically disabled in production: DevTools is not activated when running a fully packaged JAR or when launched via a production JVM agent flag.
Interview tip: DevTools never belongs in a production deployment — the spring-boot-devtools dependency is automatically excluded when you package a fat JAR with mvn package.

What is Spring Initializr and how does it help in project setup?
  • Spring Initializr (start.spring.io) is a web-based tool that generates a ready-to-run Spring Boot project skeleton.
  • What it generates:
    • A valid Maven or Gradle build file with selected dependencies
    • Correct Spring Boot parent version
    • Main application class with @SpringBootApplication
    • Empty test class
    • application.properties placeholder
    • Correct directory structure following Maven conventions
  • Inputs: Project type (Maven/Gradle), language (Java/Kotlin/Groovy), Spring Boot version, Group/Artifact/Name, Java version, and dependencies.
  • IDE Integration: IntelliJ IDEA, Eclipse (STS), and VS Code all have built-in Spring Initializr wizards — no browser required.
  • API: Can be used programmatically (e.g., in custom project generators or CI pipelines).
Interview tip: Spring Initializr eliminates the "blank page" problem — it ensures the project structure is correct from day one, reducing early configuration errors common in manual setup.

What is the role of Spring Boot CLI? When is it useful?
  • Spring Boot CLI (Command Line Interface) is a command-line tool that enables rapid development of Spring Boot applications using Groovy scripts — without requiring a Maven/Gradle build setup.
  • Key capabilities:
    • spring run app.groovy — runs a Groovy-based Spring Boot app directly
    • spring init — calls Spring Initializr from the command line
    • spring test — runs tests in Groovy scripts
  • Use cases:
    • Rapid prototyping: Write a few lines of Groovy to prototype a REST endpoint without any build files
    • Demos and proof-of-concepts: Show a working API in a single file
    • Scripting: Automate Spring Boot project creation in CI pipelines
  • Not for production: CLI-based apps are not standard for production use — they lack the structure and build lifecycle of Maven/Gradle projects.
Interview tip: Most teams don't use the CLI day-to-day, but it demonstrates the depth of Spring Boot's tooling ecosystem. Knowing it exists shows breadth of knowledge.

How does Spring Boot achieve auto-configuration?
  • Trigger: @EnableAutoConfiguration (included in @SpringBootApplication) activates auto-configuration at startup.
  • Discovery mechanism:
    • Spring Boot 2.x: Reads all auto-configuration class names from META-INF/spring.factories under the key org.springframework.boot.autoconfigure.EnableAutoConfiguration.
    • Spring Boot 3.x: Uses META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports (more performant, explicit imports).
  • Conditional application: Each auto-configuration class is annotated with conditional annotations that control whether it activates:
    • @ConditionalOnClass — applies only if a specific class is on the classpath
    • @ConditionalOnMissingBean — backs off if a user-defined bean already exists
    • @ConditionalOnProperty — activates based on a property value
  • Example — DataSource auto-config: If HikariCP and spring.datasource.url are present, Spring Boot creates a HikariDataSource bean automatically.
  • Debugging: Run with --debug flag or set logging.level.org.springframework.boot.autoconfigure=DEBUG to see the auto-configuration report.
Interview tip: Understanding the conditional mechanism is key. The "backs off" behavior (@ConditionalOnMissingBean) is what allows users to override any auto-configuration by simply declaring their own bean.

Explain Spring Boot's layered architecture.
  • Spring Boot applications typically follow a multi-tier layered architecture:
    • Presentation Layer (Controller): @RestController / @Controller — handles HTTP requests, input validation, serialization/deserialization. Delegates to service layer.
    • Service Layer: @Service — contains business logic, orchestrates multiple repository calls, manages transactions. Decoupled from HTTP and persistence concerns.
    • Repository / Data Access Layer: @Repository — interacts with the database via JPA repositories, JDBC, or other ORMs. Translates persistence exceptions to Spring's DataAccessException hierarchy.
    • Domain / Model Layer: Plain Java objects (POJOs), JPA entities, DTOs — represent business domain data.
  • Layered JAR (Spring Boot 2.3+): Spring Boot also introduced a "layered JAR" concept for Docker optimization — the fat JAR is split into layers (dependencies, snapshot-dependencies, resources, application) enabling incremental Docker image builds.
Interview tip: Always separate concerns between layers. A common interview scenario: "What if business logic ends up in the controller?" Answer: it breaks testability, reusability, and violates SRP — service layer is the correct place for business logic.

What is the default logging framework used in Spring Boot?
  • Spring Boot uses SLF4J (Simple Logging Facade for Java) as the logging API, backed by Logback as the default implementation — both included transitively via spring-boot-starter-logging.
  • Default log format: Timestamp, log level, PID, thread name, logger name, and message.
  • Switching to Log4j2: Exclude spring-boot-starter-logging and add spring-boot-starter-log4j2.
  • Configuration: Simple settings via application.properties: 
    logging.level.com.myapp=DEBUG
logging.file.name=application.log
logging.pattern.console=%d{yyyy-MM-dd} %-5level %logger{36} - %msg%n
  • Structured logging (Spring Boot 3.4+): Native support for JSON-structured logs for ELK/Loki pipelines.
Interview tip: Always code against SLF4J (LoggerFactory.getLogger(...)), never directly against Logback or Log4j — this decouples your code from the logging implementation and enables painless switching.

What are the limitations of Spring Boot?
  • Opinionated defaults can be limiting: When you need non-standard configurations (e.g., custom Hibernate session factory, custom security chain), fighting Spring Boot's defaults can be complex and confusing.
  • Large artifact size: Fat JARs bundle all dependencies — a simple REST service can be 50–80 MB. This impacts cold start times (especially for serverless/Lambda) and network transfer in CI/CD.
  • Startup time: Full Spring Boot applications with many beans can take 10–30 seconds to start. Spring Native (GraalVM AOT compilation) partially addresses this.
  • Auto-configuration magic: Can be hard to debug — unexpected beans may be configured, leading to hard-to-trace behavior. Requires understanding of conditional configuration.
  • Memory footprint: Higher baseline memory compared to micro-frameworks like Quarkus or Micronaut — relevant for constrained environments.
  • Over-engineering risk: For simple use cases, Spring Boot introduces unnecessary complexity; a simple web server might not need a full IoC container.
Interview tip: This is a maturity question — interviewers want to know you understand trade-offs, not just capabilities. Acknowledge that Spring Boot is excellent for the vast majority of enterprise use cases while being honest about its limitations.

How do profiles work in Spring Boot?
  • Profiles enable environment-specific configuration and bean registration in Spring Boot.
  • Profile-specific property files: application-{profile}.properties or application-{profile}.yml are loaded when that profile is active. Properties override the base application.properties.
  • Activating profiles:
    • spring.profiles.active=dev in application.properties
    • --spring.profiles.active=prod as a command-line argument
    • SPRING_PROFILES_ACTIVE=staging environment variable (recommended in containers)
    • @ActiveProfiles("test") in test classes
  • Profile-conditional beans: @Profile("dev") on a @Bean or @Configuration class — that bean is only registered when the named profile is active.
  • Profile groups (Spring Boot 2.4+): spring.profiles.group.prod=proddb,prodmq — activating "prod" activates all grouped profiles.
# application-dev.yml
spring:
  datasource:
    url: jdbc:h2:mem:devdb
  jpa:
    show-sql: true

# application-prod.yml
spring:
  datasource:
    url: jdbc:postgresql://prod-host:5432/appdb
Interview tip: In production, always activate profiles via environment variables — never hardcode spring.profiles.active=prod in the deployed artifact, as it reduces portability.

How do you override default properties in Spring Boot?
  • Spring Boot uses a property source priority chain (highest priority first):
    1. Command-line arguments: --server.port=9090
    2. Java System Properties: -Dserver.port=9090
    3. OS environment variables: SERVER_PORT=9090
    4. Profile-specific properties: application-prod.yml
    5. Application properties: application.yml / application.properties
    6. Default values in @Value and @ConfigurationProperties
  • Overriding a Spring Boot default: Set the property with the same key at a higher-priority source. E.g., to change the default Tomcat port from 8080: add server.port=8085 in application.properties.
  • Overriding via custom bean: Define your own @Bean of the auto-configured type — Spring Boot backs off due to @ConditionalOnMissingBean.
  • Excluding auto-config: @SpringBootApplication(exclude = {DataSourceAutoConfiguration.class})
Interview tip: Understanding this priority chain is essential for debugging — "why is my app still using the wrong database URL?" is usually caused by a lower-priority property file being overridden by an environment variable.
