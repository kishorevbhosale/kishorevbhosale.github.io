What is Dependency Injection (DI) in Spring? Why is it important?
  • Dependency Injection is a design pattern where an object's dependencies are provided externally (by the IoC container) rather than created by the object itself. Spring's IoC (Inversion of Control) container manages this process.
  • Why it's important:
    • Loose coupling: Objects depend on abstractions (interfaces), not concrete implementations — enabling easy substitution and testing.
    • Testability: Dependencies can be mocked in unit tests without a running Spring context.
    • Maintainability: Changing an implementation only requires a change in one place (the configuration), not in all classes that use it.
    • Single Responsibility: Objects focus on their business logic, not on constructing their dependencies.
// Without DI — tightly coupled, untestable
public class OrderService {
    private PaymentService paymentService = new StripePaymentService(); // hardcoded
}

// With DI — loosely coupled, testable
@Service
public class OrderService {
    private final PaymentService paymentService; // injected by Spring
    public OrderService(PaymentService paymentService) {
        this.paymentService = paymentService;
    }
}
Interview tip: DI is the enabler of SOLID principles — particularly Dependency Inversion (D in SOLID). Always frame DI as "what you depend on should be an abstraction, not a concrete class."

What are the different types of Dependency Injection in Spring?
  • Spring supports three types of DI:
    • Constructor Injection: Dependencies are provided through the class constructor. Marked with @Autowired (optional in Spring 4.3+ if there is only one constructor).
    • Setter Injection: Dependencies are injected through setter methods after object construction. Useful for optional dependencies.
    • Field Injection: Dependencies are directly injected into fields using @Autowired. Convenient but discouraged for production code.
// Constructor Injection (preferred)
@Service
public class OrderService {
    private final PaymentService paymentService;
    public OrderService(PaymentService paymentService) {
        this.paymentService = paymentService;
    }
}

// Setter Injection
@Service
public class NotificationService {
    private EmailService emailService;
    @Autowired
    public void setEmailService(EmailService emailService) {
        this.emailService = emailService;
    }
}

// Field Injection (avoid in production)
@Service
public class ReportService {
    @Autowired
    private DataService dataService;
}
Interview tip: Constructor injection is the Spring team's official recommendation. It enforces immutability (final fields), makes dependencies explicit, and enables easy unit testing without a Spring context.

Which injection method is preferred in Spring and why?
  • Constructor injection is the recommended approach by the Spring team and the broader Java community. Reasons:
  • Immutability: final fields prevent accidental reassignment after construction.
  • Mandatory dependency enforcement: If a required dependency isn't available, the container fails fast at startup rather than producing a NullPointerException later.
  • Circular dependency detection: Spring immediately throws BeanCurrentlyInCreationException for circular dependencies between constructor-injected beans — revealing design flaws early.
  • Pure unit testing: No Spring context needed — simply call new MyService(mockDependency) in tests.
  • Avoids reflection: Unlike field injection, constructor injection works without breaking encapsulation.
  • Lombok integration: @RequiredArgsConstructor generates a constructor for all final fields, eliminating boilerplate.
@Service
@RequiredArgsConstructor  // Lombok generates constructor
public class OrderService {
    private final PaymentService paymentService;
    private final InventoryService inventoryService;
    private final NotificationService notificationService;
}
Interview tip: Field injection is often seen in tutorials for brevity, but it's an anti-pattern in production code. Always advocate for constructor injection and explain why — this shows senior-level understanding.

How does Spring manage the lifecycle of a bean?
  • Spring beans go through a well-defined lifecycle managed by the ApplicationContext:
    1. Instantiation: Spring creates the bean instance via the constructor.
    2. Property Population: Dependencies are injected via setter injection or field injection.
    3. BeanNameAware / BeanFactoryAware / ApplicationContextAware: If the bean implements these interfaces, Spring invokes the corresponding setter methods.
    4. BeanPostProcessor — postProcessBeforeInitialization(): Custom pre-init processing (e.g., AOP proxying begins here).
    5. @PostConstruct / InitializingBean.afterPropertiesSet() / init-method: Custom initialization logic runs.
    6. BeanPostProcessor — postProcessAfterInitialization(): Custom post-init processing (AOP proxy wrapping completes here).
    7. Bean is ready for use.
    8. @PreDestroy / DisposableBean.destroy() / destroy-method: On context shutdown, cleanup runs.
Interview tip: Prefer @PostConstruct and @PreDestroy over implementing InitializingBean/DisposableBean — the annotation approach is less invasive and doesn't couple your bean to Spring API interfaces.

What is the role of the BeanFactory interface?
  • BeanFactory is the root interface for the Spring IoC container. It provides the basic contract for bean access: getBean(), containsBean(), isSingleton(), isPrototype().
  • Characteristics:
    • Lazy initialization — beans are created only when requested (unless configured otherwise)
    • Lightweight — suitable for resource-constrained environments
    • No built-in AOP, event publishing, or internationalization support
  • Common implementation: DefaultListableBeanFactory (the actual container used internally by ApplicationContext).
  • In practice, you almost never use BeanFactory directly — use ApplicationContext, which extends it with additional capabilities.
Interview tip: BeanFactory is the contract; ApplicationContext is the full implementation. Think of BeanFactory as a bare-bones vending machine and ApplicationContext as a full grocery store with AOP, events, and messages.

What are the different bean scopes available in Spring?
  • singleton (default) — one shared instance per ApplicationContext. Stateless beans (services, repositories, utilities) should use this scope.
  • prototype — a new instance is created every time the bean is requested from the container. Used for stateful beans (form beans, command objects).
  • request — one instance per HTTP request; the instance is created at the start of a request and destroyed at its end. Web-aware ApplicationContext required.
  • session — one instance per HTTP session; tied to the user's session lifecycle. Used for user-specific state.
  • application — one instance per ServletContext (similar to singleton but shared across all Spring contexts in the same web app).
  • websocket — one instance per WebSocket session.
  • Custom scopes — implement org.springframework.beans.factory.config.Scope and register it with the container (e.g., thread scope, tenant scope).
Interview tip: The most common gotcha: injecting a prototype bean into a singleton bean. After the first injection, the singleton holds the same prototype reference forever. Use @Lookup or ObjectProvider<T> to get a new prototype instance on each call.

How do you create a prototype-scoped bean?
// Option 1: @Scope annotation
@Component
@Scope("prototype")
public class PdfGenerator {
    public byte[] generate(ReportData data) { ... }
}

// Option 2: @Bean with scope
@Configuration
public class AppConfig {
    @Bean
    @Scope("prototype")
    public PdfGenerator pdfGenerator() {
        return new PdfGenerator();
    }
}

// Correctly consuming a prototype from a singleton bean:
@Service
public class ReportService {
    // ObjectProvider is the recommended way — gets new instance on every call
    private final ObjectProvider<PdfGenerator> pdfGeneratorProvider;

    public ReportService(ObjectProvider<PdfGenerator> pdfGeneratorProvider) {
        this.pdfGeneratorProvider = pdfGeneratorProvider;
    }

    public byte[] createReport(ReportData data) {
        PdfGenerator generator = pdfGeneratorProvider.getObject(); // new instance each call
        return generator.generate(data);
    }
}
Interview tip: Direct @Autowired injection of a prototype into a singleton gives you a stale singleton-equivalent prototype. The correct patterns are ObjectProvider, @Lookup method injection, or ApplicationContext.getBean().

What is the default bean scope in Spring Boot?
  • The default bean scope in Spring (and Spring Boot) is singleton.
  • This means a single instance of the bean is created at application startup and shared across the entire ApplicationContext. All requests for that bean return the same instance.
  • Implications for thread safety: Singleton beans shared across multiple threads must be stateless or thread-safe. Storing mutable state (like a user-specific field) in a singleton service is a critical bug — it causes data leakage between concurrent requests.
  • Why singleton by default? It's memory-efficient, provides faster response (no object creation overhead), and encourages stateless design.
Interview tip: "Are Spring singleton beans thread-safe?" — No, not inherently. Thread safety depends on whether the bean holds mutable state. Spring services and repositories should be stateless (no instance fields storing request-scoped data), making them naturally thread-safe.

What is the role of @PostConstruct and @PreDestroy annotations?
  • @PostConstruct marks a method to be called after dependency injection is complete and before the bean is put into service. Used for initialization logic that requires injected dependencies to be available.
  • @PreDestroy marks a method to be called just before the bean is removed from the container (on context close). Used for cleanup — releasing connections, closing streams, etc.
@Component
public class CacheManager {

    @Autowired
    private RedisTemplate<String, Object> redisTemplate;

    private Map<String, Object> localCache;

    @PostConstruct
    public void initCache() {
        // Safe to use redisTemplate here — it's already injected
        localCache = new HashMap<>();
        // Pre-load frequently accessed data
    }

    @PreDestroy
    public void evictCache() {
        // Clean up before shutdown
        localCache.clear();
    }
}
  • Execution order: Constructor → DI → @PostConstruct → bean is in use → @PreDestroy → bean destroyed.
  • These are JSR-250 annotations (jakarta.annotation package) — not Spring-specific, so they work with any JSR-250-compliant container.
Interview tip: @PostConstruct is a safer initialization point than constructor — at constructor time, dependencies may not be set yet (for setter/field injection). Use constructor + final fields to avoid this ambiguity entirely.

What are BeanPostProcessor and BeanFactoryPostProcessor?
  • BeanFactoryPostProcessor: Runs before any bean is instantiated. It modifies the bean definitions (metadata) in the container. Examples: PropertySourcesPlaceholderConfigurer resolves ${} property placeholders in bean definitions.
  • BeanPostProcessor: Runs after each bean is instantiated but before it's returned to the application. It modifies bean instances. This is how Spring applies AOP proxies, @Autowired injection, and @PostConstruct/@PreDestroy callbacks.

What is circular dependency in Spring? How is it handled?
  • Circular dependency occurs when Bean A depends on Bean B, and Bean B depends on Bean A (directly or transitively). Spring cannot create either bean first.
  • With constructor injection: Spring throws BeanCurrentlyInCreationException at startup — it cannot resolve the cycle. This is the desired behavior as it surfaces the design flaw.
  • With setter/field injection: Spring Boot 2.6+ also throws an error by default (spring.main.allow-circular-references is false). Older versions handled it by using partially-constructed beans.
  • How to fix:
    • Refactor: Extract shared functionality into a third bean that neither depends on the other (preferred).
    • @Lazy on one injection point — breaks the cycle by using a proxy for the lazily-initialized bean.
    • Use an event/observer pattern to decouple the two services.
Interview tip: Circular dependencies are a code smell indicating poor separation of concerns. The correct answer is always "refactor to eliminate the circular dependency" rather than "configure Spring to allow it."

What is @DependsOn annotation in Spring?
  • @DependsOn forces Spring to initialize the specified beans before the current bean, even if there's no direct dependency relationship through injection.
  • Use case: When a bean has an implicit startup ordering requirement — e.g., a database migration bean must run before any JPA repository beans are initialized.
@Component
@DependsOn("flywayMigration")
public class ProductRepository { ... }

@Bean(name = "flywayMigration")
public Flyway flyway(DataSource dataSource) {
    Flyway flyway = Flyway.configure().dataSource(dataSource).load();
    flyway.migrate();
    return flyway;
}
  • Note: This only guarantees initialization order — it does not inject the dependency. It's for cases where Bean A's constructor has a side effect (like running migrations) that must complete before Bean B starts.
Interview tip: @DependsOn should be used sparingly. If you find yourself using it frequently, it often indicates a design issue. Explicit dependency injection is always preferred over implicit ordering.

How do you create lazy-initialized beans in Spring Boot?
  • Annotation-level: @Lazy on a @Component or @Bean method — that specific bean is created on first request.
  • Injection-point level: @Autowired @Lazy private ExpensiveService service; — Spring injects a CGLIB proxy; actual bean creation deferred to first method call.
  • Global lazy initialization: spring.main.lazy-initialization=true — all beans lazy by default. Override individual critical beans with @Lazy(false).
// Individual lazy bean
@Service
@Lazy
public class HeavyReportService {
    public HeavyReportService() {
        // Expensive initialization — only runs when first used
    }
}

// Global + specific override
// application.properties:
// spring.main.lazy-initialization=true

@Component
@Lazy(false)  // This critical bean must be eager
public class DatabaseHealthChecker { ... }
  • Benefit in tests: Significantly reduces context startup time in @SpringBootTest tests by not instantiating all beans.
Interview tip: Global lazy initialization can mask startup configuration errors — a misconfigured bean won't fail until first use. Use it in development/testing, but consider eager initialization in production for fail-fast behavior.

Explain the lifecycle phases of a Spring bean with example.
@Component
public class AuditService implements InitializingBean, DisposableBean,
        BeanNameAware, ApplicationContextAware {

    @Autowired
    private AuditRepository repository; // Step 2: DI

    // Step 3: Aware interfaces
    @Override
    public void setBeanName(String name) {
        System.out.println("Bean name: " + name);
    }

    // Step 5: InitializingBean
    @Override
    public void afterPropertiesSet() {
        System.out.println("afterPropertiesSet called");
    }

    // Step 5: @PostConstruct (runs before afterPropertiesSet in same order as in spec)
    @PostConstruct
    public void init() {
        System.out.println("@PostConstruct - verifying DB connection");
        repository.count(); // safe — repository is injected
    }

    // Step 8: @PreDestroy
    @PreDestroy
    public void cleanup() {
        System.out.println("@PreDestroy - flushing audit queue");
    }

    // Step 8: DisposableBean
    @Override
    public void destroy() {
        System.out.println("destroy() called");
    }
}
  • Full lifecycle sequence: Instantiate → Set properties/DI → *Aware callbacks → BeanPostProcessor.before → @PostConstruct → afterPropertiesSet → custom init-method → BeanPostProcessor.after → Ready → @PreDestroy → destroy() → custom destroy-method
Interview tip: Know the order: @PostConstruct runs before InitializingBean.afterPropertiesSet() which runs before initMethod. This ordering is deterministic and important when multiple initialization mechanisms are used simultaneously.
