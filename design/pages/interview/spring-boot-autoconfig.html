What is auto-configuration in Spring Boot? How does it work?
  • Auto-configuration is Spring Boot's mechanism to automatically configure the application context based on the libraries on the classpath and the properties set — without requiring any explicit bean definitions from the developer.
  • How it works (step by step):
    1. @SpringBootApplication includes @EnableAutoConfiguration.
    2. Spring Boot loads all auto-configuration candidates from META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports (Boot 3.x) or META-INF/spring.factories (Boot 2.x).
    3. Each auto-configuration class is annotated with @AutoConfiguration or @Configuration and conditional annotations (@ConditionalOnClass, @ConditionalOnMissingBean, etc.).
    4. Only configurations whose conditions are satisfied are applied — the rest are silently skipped.
  • Example: If DataSource and JPA are on the classpath and spring.datasource.url is set, DataSourceAutoConfiguration creates a DataSource bean, HibernateJpaAutoConfiguration creates an EntityManagerFactory, and TransactionAutoConfiguration sets up transaction management.
Interview tip: Auto-configuration is not magic — it's conditional @Configuration classes loaded via the service loader pattern. Running with --debug prints the auto-configuration report showing what was applied and what was skipped and why.

What is the role of spring.factories in auto-configuration?
  • META-INF/spring.factories is a properties file inside each Spring Boot JAR (including spring-boot-autoconfigure.jar) that registers implementations of various Spring extension interfaces.
  • In the context of auto-configuration: It maps the key org.springframework.boot.autoconfigure.EnableAutoConfiguration to a comma-separated list of @Configuration classes to be considered for auto-configuration.
  • Spring Boot 3.x change: Auto-configuration class registration was moved to a dedicated file META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports for better performance and clarity. spring.factories still exists for other extension points (like ApplicationListener, EnvironmentPostProcessor).
# META-INF/spring.factories (Spring Boot 2.x)
org.springframework.boot.autoconfigure.EnableAutoConfiguration=\
  org.springframework.boot.autoconfigure.jdbc.DataSourceAutoConfiguration,\
  org.springframework.boot.autoconfigure.orm.jpa.HibernateJpaAutoConfiguration,\
  org.springframework.boot.autoconfigure.security.servlet.SecurityAutoConfiguration
Interview tip: The service loader pattern via spring.factories is what makes Spring Boot's auto-configuration pluggable — any third-party library can add its own auto-configuration by simply including a spring.factories file in its JAR.

What is the difference between @EnableAutoConfiguration and @SpringBootApplication?
  • @EnableAutoConfiguration is a focused annotation that solely activates Spring Boot's auto-configuration mechanism. It triggers the loading and conditional application of auto-configuration classes.
  • @SpringBootApplication is a convenience meta-annotation that combines:
    • @SpringBootConfiguration (marks the class as a Spring config source)
    • @EnableAutoConfiguration (triggers auto-config)
    • @ComponentScan (scans the application's packages for components)
  • Using @SpringBootApplication is equivalent to using all three, with the advantage of being a single annotation.
  • When to use @EnableAutoConfiguration directly: When you need fine-grained control — e.g., separating the main application class from the component scan root, or in non-standard configurations.
Interview tip: Always use @SpringBootApplication on the main class. Use the individual annotations only in specialized scenarios where the combined behavior needs to be split across multiple classes.

How do you exclude an auto-configuration class in Spring Boot?
  • Three approaches:
// Option 1: exclude attribute on @SpringBootApplication
@SpringBootApplication(exclude = {
    DataSourceAutoConfiguration.class,
    SecurityAutoConfiguration.class
})
public class MyApp { ... }

// Option 2: excludeName for classes not on the classpath
@SpringBootApplication(excludeName = {
    "org.springframework.boot.autoconfigure.jdbc.DataSourceAutoConfiguration"
})
public class MyApp { ... }
# Option 3: application.properties
spring.autoconfigure.exclude=\
  org.springframework.boot.autoconfigure.jdbc.DataSourceAutoConfiguration,\
  org.springframework.boot.autoconfigure.security.servlet.SecurityAutoConfiguration
  • Common use cases:
    • Exclude DataSourceAutoConfiguration when managing DataSource manually
    • Exclude SecurityAutoConfiguration to configure security from scratch
    • Exclude JmxAutoConfiguration in containers where JMX is not needed
Interview tip: Excluding auto-configuration should be a deliberate decision — always explain why. Simply excluding to suppress startup errors usually masks an underlying configuration problem that should be fixed properly.

What is the purpose of META-INF/spring.factories file?
  • META-INF/spring.factories is Spring Boot's extension registry — a properties file that maps extension point interfaces to their implementations, discovered via the Java service loader mechanism.
  • Extension points registered here:
    • EnableAutoConfiguration — auto-configuration classes
    • ApplicationContextInitializer — customize context before refresh
    • ApplicationListener — listen to application events
    • EnvironmentPostProcessor — modify the Environment before context creation
    • SpringApplicationRunListener — hook into the application run lifecycle
    • FailureAnalyzer — provide human-readable failure messages
  • Any library that wants to integrate with Spring Boot adds its own spring.factories to its JAR — this is how third-party auto-configurations (Flyway, Micrometer, Spring Data) are discovered.
Interview tip: Understanding spring.factories is key to writing custom Spring Boot Starters. Your starter's auto-configuration class must be registered here (or in AutoConfiguration.imports for Boot 3.x) to be picked up by the container.

How do you create a custom auto-configuration in Spring Boot?
// Step 1: Create the auto-configuration class
@AutoConfiguration  // Spring Boot 3.x (@Configuration for 2.x)
@ConditionalOnClass(AuditClient.class)
@ConditionalOnMissingBean(AuditService.class)
@EnableConfigurationProperties(AuditProperties.class)
public class AuditAutoConfiguration {

    @Bean
    @ConditionalOnProperty(name = "audit.enabled", havingValue = "true", matchIfMissing = true)
    public AuditService auditService(AuditProperties properties) {
        return new DefaultAuditService(properties.getEndpoint());
    }
}

// Step 2: Register in auto-configuration imports
// For Spring Boot 3.x:
// META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports
// com.mylib.audit.AuditAutoConfiguration

// For Spring Boot 2.x:
// META-INF/spring.factories
// org.springframework.boot.autoconfigure.EnableAutoConfiguration=\
//   com.mylib.audit.AuditAutoConfiguration
  • Key principles: Make your auto-configuration fully conditional — always use @ConditionalOnMissingBean to back off when users define their own implementation. This is the "opinionated but overridable" contract.
Interview tip: The most common mistake in custom auto-configuration is forgetting @ConditionalOnMissingBean. Without it, your auto-configured bean cannot be overridden, violating the Spring Boot contract.

What is the difference between @ConditionalOnClass and @ConditionalOnMissingBean?
  • @ConditionalOnClass — applies the configuration only if the specified class(es) are present on the classpath. Used to activate configuration for a library that may or may not be a dependency.
    Example: @ConditionalOnClass(HikariDataSource.class) — HikariCP auto-configuration only activates if HikariCP JAR is on the classpath.
  • @ConditionalOnMissingBean — applies the configuration only if the specified bean type is not already registered in the application context. Enables user override — if you define your own DataSource bean, Spring Boot's auto-configured one backs off.
    Example: @ConditionalOnMissingBean(DataSource.class) — auto-configured DataSource only registers if no DataSource bean exists.
AnnotationConditionPurpose
@ConditionalOnClassClass on classpathLibrary presence detection
@ConditionalOnMissingClassClass NOT on classpathActivate when library absent
@ConditionalOnBeanBean already existsActivate when dependency bean is present
@ConditionalOnMissingBeanBean does NOT existUser override support
Interview tip: @ConditionalOnClass and @ConditionalOnMissingBean are the two most important conditional annotations. Every auto-configuration class uses them to ensure it only activates when appropriate and always yields to user-defined beans.

What is @ConditionalOnProperty and when would you use it?
  • @ConditionalOnProperty conditionally activates a bean or configuration based on the presence and value of a property in the Spring Environment.
  • Parameters:
    • name/value — property key
    • havingValue — expected value (default: non-false, non-empty string)
    • matchIfMissing — if true, condition passes even if the property is absent (default: false)
    • prefix — property prefix (combined with name)
// Feature toggle for Stripe payment
@Configuration
@ConditionalOnProperty(prefix = "payment", name = "provider", havingValue = "stripe")
public class StripePaymentConfig {
    @Bean
    public PaymentGateway stripeGateway() { return new StripeGateway(); }
}

// Activate if property is absent OR set to true
@Bean
@ConditionalOnProperty(name = "cache.enabled", havingValue = "true", matchIfMissing = true)
public CacheManager cacheManager() { return new RedisCacheManager(); }
  • Use cases: Feature flags, environment-specific integrations, enabling optional modules, A/B configuration testing.
Interview tip: @ConditionalOnProperty is the Spring Boot way to implement feature toggles. Combined with Spring Cloud Config Server, you can change feature flags at runtime across all microservice instances without redeployment.

Explain @ConditionalOnResource and its use case.
  • @ConditionalOnResource applies configuration only if a specific resource file exists in the classpath or file system.
  • Syntax: @ConditionalOnResource(resources = "classpath:mybatis-config.xml")
  • Use cases:
    • Auto-configure MyBatis only if a mybatis-config.xml is present
    • Enable HTTPS-specific config only if a keystore file exists
    • Activate legacy configuration only if a specific properties file is present
@Configuration
@ConditionalOnResource(resources = "classpath:logback-custom.xml")
public class CustomLoggingConfig {
    // Applied only if logback-custom.xml exists in classpath
    @Bean
    public LoggingAspect loggingAspect() {
        return new LoggingAspect();
    }
}
Interview tip: @ConditionalOnResource is less commonly used than @ConditionalOnClass or @ConditionalOnProperty, but it's valuable for detecting optional configuration files. It checks for resource existence, not content.

What are Spring Boot Starter dependencies?
  • Spring Boot Starters are curated, pre-packaged dependency descriptors that aggregate all the libraries needed for a specific feature or technology into a single Maven/Gradle dependency.
  • What each starter provides:
    • Transitive dependencies — all required libraries, version-aligned
    • Auto-configuration classes — registered in spring.factories
    • Default configuration — via auto-configuration and property defaults
  • Benefits:
    • No more "which version of Hibernate works with this version of Spring?"
    • One line in pom.xml instead of 10+ individual dependencies
    • Tested, known-good dependency combinations
Interview tip: When asked about dependency management best practices, mention that Spring Boot Starters implement the "Transitive Dependencies" pattern — they abstract away the details of what libraries are needed, letting developers focus on features rather than classpath management.

How do Spring Boot Starters simplify dependency management?
  • Single dependency declaration: Instead of individually adding Spring MVC, Tomcat, Jackson, and Validation with specific compatible versions, developers add one spring-boot-starter-web entry.
  • BOM-managed versions: The parent POM (spring-boot-starter-parent) pins all dependency versions. No version conflicts, no ClassNotFoundException from mismatched versions.
  • Automatic transitive dependency resolution: Maven/Gradle resolves all transitive dependencies declared by the starter — developers don't need to know the full dependency tree.
  • Easy upgrades: Upgrading Spring Boot version in the parent POM upgrades all transitive dependencies together, maintaining compatibility.
  • Exclusion mechanism: Default dependencies can be excluded and replaced: exclude Tomcat from spring-boot-starter-web, add spring-boot-starter-undertow.
Interview tip: Before Spring Boot Starters, setting up a Spring MVC + JPA application required 15-20 manually-versioned dependencies. With starters, it's 3-4 entries. This is a huge productivity gain in enterprise environments where teams manage hundreds of services.

What is spring-boot-starter-data-jpa and what does it provide?
  • spring-boot-starter-data-jpa provides everything needed for JPA-based database persistence:
  • Transitive dependencies included:
    • Hibernate ORM — the JPA implementation (provider)
    • Spring Data JPA — repository abstraction, query methods, JpaRepository
    • Spring ORM — Spring's transaction management integration with JPA
    • HikariCP — high-performance JDBC connection pool (included transitively via spring-boot-starter-jdbc)
  • Auto-configuration provided:
    • DataSourceAutoConfiguration — configures DataSource based on spring.datasource.*
    • HibernateJpaAutoConfiguration — configures EntityManagerFactory
    • JpaRepositoriesAutoConfiguration — enables repository scanning
    • TransactionAutoConfiguration — enables transaction management
  • Requires a JDBC driver dependency (e.g., postgresql, h2) to be added separately.
Interview tip: Know that spring.jpa.hibernate.ddl-auto=validate is the recommended production setting — it validates the schema matches the entities without modifying it. Never use create-drop in production.

Explain the role of spring-boot-starter-security.
  • spring-boot-starter-security includes Spring Security and enables automatic security hardening:
  • What happens when you add this starter:
    • All HTTP endpoints are secured — require authentication by default
    • A default user is created (user/randomly-generated password printed in logs)
    • Login form is provided at /login
    • CSRF protection is enabled for state-changing operations
    • HTTP Basic Auth is enabled
    • Security headers (X-Frame-Options, X-XSS-Protection, HSTS) are added to all responses
  • Customization: Create a @Configuration class with a SecurityFilterChain bean to override defaults — configure custom authentication, authorization rules, JWT, OAuth2, etc.
Interview tip: Adding spring-boot-starter-security to an existing application is often a breaking change because all endpoints suddenly require authentication. Always plan security integration early in the project.

What is the purpose of spring-boot-starter-test?
  • spring-boot-starter-test is a test-scope starter that bundles everything needed for writing Spring Boot tests:
  • Included libraries:
    • JUnit 5 (Jupiter) — test framework with @Test, @ExtendWith, parameterized tests
    • Mockito — mocking framework (@Mock, @Spy, when(...).thenReturn(...))
    • AssertJ — fluent assertion library (assertThat(result).isEqualTo(expected))
    • Spring Test + Spring Boot Test — @SpringBootTest, @DataJpaTest, @WebMvcTest, MockMvc
    • Hamcrest — matcher-based assertions (for legacy compatibility)
    • JSONPath — JSON response testing
  • Scope: Always declared with <scope>test</scope> — not included in production artifacts.
Interview tip: Spring Boot 2.2+ uses JUnit 5 by default. If you need JUnit 4, you must add the vintage engine: junit-vintage-engine. Most modern projects should use JUnit 5 exclusively.

How do you create a custom Spring Boot Starter?
  • A custom Spring Boot Starter consists of two modules (by convention):
    1. {name}-spring-boot-autoconfigure — contains the auto-configuration class and @ConfigurationProperties.
    2. {name}-spring-boot-starter — a thin wrapper that depends on the autoconfigure module and any required third-party libraries.
// autoconfigure module
@AutoConfiguration
@ConditionalOnClass(AuditClient.class)
@EnableConfigurationProperties(AuditProperties.class)
public class AuditAutoConfiguration {

    @Bean
    @ConditionalOnMissingBean
    public AuditService auditService(AuditProperties props) {
        return new AuditServiceImpl(props.getServerUrl());
    }
}

// META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports:
// com.company.audit.AuditAutoConfiguration
  • Naming convention: Third-party starters use {name}-spring-boot-starter (e.g., mybatis-spring-boot-starter), not spring-boot-starter-{name} (that prefix is reserved for official Spring projects).
Interview tip: Custom starters are the right pattern for shared infrastructure in large organizations — common audit logging, company-wide security configuration, standard metrics setup. One team builds the starter; all teams consume it with zero configuration.

What are the advantages of custom Spring Boot Starters?
  • DRY principle at the infrastructure level: Common cross-cutting concerns (security, logging, metrics, audit) are implemented once in a starter and consumed by all services.
  • Enforced organizational standards: Security policies, logging formats, and monitoring configuration are baked into the starter — developers can't accidentally skip them.
  • Zero-configuration for consumers: Services that add the starter get all functionality out of the box — same opinionated-but-overridable contract as official Spring Boot Starters.
  • Centralized upgrades: When the organization upgrades its standard logging framework, updating the starter version in all services' pom.xml is sufficient.
  • Testability: The starter can include test utilities that all consuming services can use (e.g., @EnableTestAuditService).
Interview tip: Custom starters are a hallmark of mature microservice platforms. Mentioning that you've created internal starters for shared infrastructure shows platform engineering experience — a strong differentiator in senior interviews.

How do you debug issues with auto-configuration in Spring Boot?
  • Auto-configuration report: Run with --debug flag or set logging.level.org.springframework.boot.autoconfigure=DEBUG. Spring Boot prints a detailed report showing:
    • Positive matches: Auto-configurations that applied and why
    • Negative matches: Auto-configurations that were skipped and why (missing class, existing bean, property not set)
    • Exclusions: Explicitly excluded auto-configurations
    • Unconditional classes: Always applied
  • Actuator conditions endpoint: GET /actuator/conditions returns the auto-configuration report as JSON (requires Actuator dependency).
  • Common debugging patterns:
    • Bean unexpectedly created → check if you have both the auto-config class AND a @ConditionalOnMissingBean on it
    • Expected auto-config not applied → verify the required class is on the classpath or property is set
    • Use ConditionEvaluationReport programmatically in tests
Interview tip: The auto-configuration report is your first tool when Spring Boot "doesn't do what you expect." Knowing to check it immediately demonstrates experienced troubleshooting skills.

What is the role of spring-boot-autoconfigure module?
  • spring-boot-autoconfigure is the core Spring Boot module that contains all of Spring Boot's built-in auto-configuration classes — over 130+ configurations covering web, data, security, messaging, caching, and more.
  • Contents:
    • Auto-configuration classes for every Spring technology (JPA, Security, Web, Cache, Actuator, etc.)
    • All @ConditionalOn* annotations are defined here
    • The AutoConfiguration.imports file registering all auto-config classes
    • @ConfigurationProperties classes for all Spring Boot configurable properties
  • This module is included transitively via any Spring Boot Starter — you don't add it directly.
  • Source of truth: When you need to understand exactly what Spring Boot configures for a feature, reading the corresponding class in spring-boot-autoconfigure is the definitive answer.
Interview tip: If you want to understand why a Spring Boot feature behaves a certain way, the source code of spring-boot-autoconfigure is the ground truth. Senior engineers reference it — it's publicly available on GitHub.
