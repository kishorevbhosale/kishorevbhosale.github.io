What is Spring Security and why is it used?
  • Spring Security is the de facto security framework for Spring Boot applications. It provides comprehensive authentication, authorization, and protection against common security vulnerabilities.
  • Core capabilities:
    • Authentication: Who are you? — Form login, HTTP Basic, JWT, OAuth2/OIDC, LDAP, SAML
    • Authorization: What can you do? — Role-based access control, method-level security, expression-based security
    • Security protections: CSRF, clickjacking, session fixation, XSS headers, HSTS, content sniffing prevention
  • How it works: Spring Security integrates with Spring MVC via a chain of servlet filters (SecurityFilterChain). Every incoming HTTP request passes through this chain before reaching the controller.
  • Auto-configuration: Adding spring-boot-starter-security automatically secures all endpoints and provides a default login form.
Interview tip: Spring Security is not just for web apps — it also provides method-level security (@PreAuthorize) for service-layer operations, making it suitable for both REST APIs and traditional web apps.

Difference between authentication and authorization.
  • Authentication — verifies who you are. The process of confirming identity through credentials (username/password, token, certificate). Establishes the security principal (the authenticated user/system).
  • Authorization — determines what you can do. Checks whether the authenticated principal has permission to perform the requested action on the requested resource.

How do you configure basic authentication in Spring Boot?
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        return http
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/public/**").permitAll()
                .anyRequest().authenticated()
            )
            .httpBasic(Customizer.withDefaults())  // Enable HTTP Basic Auth
            .build();
    }

    @Bean
    public UserDetailsService userDetailsService() {
        UserDetails user = User.builder()
            .username("admin")
            .password(passwordEncoder().encode("secret"))
            .roles("ADMIN")
            .build();
        return new InMemoryUserDetailsManager(user);
    }

    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
    }
}
  • In production: Use a database-backed UserDetailsService — never store users in-memory. Always encode passwords with BCryptPasswordEncoder.
  • Properties-based default user: Set spring.security.user.name and spring.security.user.password for simple configuration.
Interview tip: HTTP Basic Auth sends credentials (Base64-encoded) with every request — it's only secure over HTTPS. Never use it over plain HTTP in production. For REST APIs, JWT or OAuth2 tokens are strongly preferred.

What is the role of UserDetailsService in Spring Security?
  • UserDetailsService is a core Spring Security interface with a single method: UserDetails loadUserByUsername(String username). It is the bridge between Spring Security and your user store (database, LDAP, external service).
  • Spring Security calls loadUserByUsername() during authentication to retrieve the user's credentials, roles, and account status.
  • UserDetails provides: getUsername(), getPassword() (encoded), getAuthorities() (roles/permissions), and account status flags (isEnabled, isAccountNonExpired, etc.).
@Service
@RequiredArgsConstructor
public class CustomUserDetailsService implements UserDetailsService {

    private final UserRepository userRepository;

    @Override
    public UserDetails loadUserByUsername(String email) throws UsernameNotFoundException {
        User user = userRepository.findByEmail(email)
            .orElseThrow(() -> new UsernameNotFoundException("User not found: " + email));

        return org.springframework.security.core.userdetails.User.builder()
            .username(user.getEmail())
            .password(user.getPasswordHash())
            .roles(user.getRole().name())
            .accountExpired(!user.isActive())
            .build();
    }
}
Interview tip: The method is called loadUserByUsername but you can look up by email, phone, or any identifier — username is just the parameter name. Implement it to query whatever unique field your system uses for authentication.

What is the difference between GrantedAuthority and Role?
  • GrantedAuthority is the Spring Security interface representing a permission or authority granted to a principal. It's a string-based representation of what the user can do.
  • Role is a special type of GrantedAuthority with the prefix ROLE_. Spring Security automatically adds this prefix when you use the roles() method, and checks for it in hasRole() expressions.
  • Difference:
    • hasAuthority("READ_PRODUCT") — checks for the exact string "READ_PRODUCT"
    • hasRole("ADMIN") — checks for the authority "ROLE_ADMIN" (prefix added automatically)
// Using roles (Spring adds ROLE_ prefix automatically)
User.builder()
    .username("admin")
    .roles("ADMIN", "USER")  // stored as ROLE_ADMIN, ROLE_USER
    .build();

// Using authorities (no prefix added)
User.builder()
    .username("editor")
    .authorities("READ_PRODUCT", "WRITE_PRODUCT", "ROLE_EDITOR")
    .build();

// In security rules
.requestMatchers("/admin/**").hasRole("ADMIN")           // checks ROLE_ADMIN
.requestMatchers("/products/write").hasAuthority("WRITE_PRODUCT") // checks exact string
Interview tip: For fine-grained authorization, use GrantedAuthority with specific permission strings (READ_ORDER, WRITE_ORDER, DELETE_ORDER) rather than coarse-grained roles. This gives you precise control over what each user can do.

How do you secure REST APIs with role-based access in Spring Boot?
// URL-based authorization in SecurityFilterChain
@Bean
public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    return http
        .authorizeHttpRequests(auth -> auth
            .requestMatchers(HttpMethod.GET, "/api/products/**").hasAnyRole("USER", "ADMIN")
            .requestMatchers(HttpMethod.POST, "/api/products").hasRole("ADMIN")
            .requestMatchers(HttpMethod.DELETE, "/api/products/**").hasRole("ADMIN")
            .requestMatchers("/api/admin/**").hasRole("ADMIN")

            .anyRequest().authenticated()
        )
        .build();
}

@RequestMapping("/api/orders")
public class OrderController {

What is CSRF protection? How does Spring Security handle it?
  • CSRF (Cross-Site Request Forgery) is an attack where a malicious site tricks an authenticated user's browser into making unwanted requests to a trusted site. Because the browser automatically includes session cookies, the server accepts the forged request.
  • Spring Security's CSRF protection: Generates a unique, unpredictable token (CSRF token) per session. State-changing requests (POST, PUT, DELETE) must include this token. The browser-based form or AJAX client includes it; a forged request from another site cannot obtain it.
  • For REST APIs (JWT/stateless): CSRF is not needed — REST APIs use token-based authentication, not browser session cookies. Always disable CSRF for stateless REST APIs: http.csrf(AbstractHttpConfigurer::disable)
  • For form-based apps: Keep CSRF enabled. Use Thymeleaf's th:action or include the _csrf hidden field.
Interview tip: The question "why do you disable CSRF for REST APIs?" is a classic. The correct answer: REST APIs use Bearer tokens (JWT) in the Authorization header — not cookies — so CSRF attacks don't apply. Cookies are the attack vector, not Authorization headers.

What is JWT (JSON Web Token)? How is it used in Spring Security?
  • JWT is a compact, self-contained token that encodes identity and claims as a signed JSON payload. Structure: header.payload.signature
  • Header: Algorithm and token type (e.g., HS256, RS256)
  • Payload: Claims — sub (subject/userId), iat (issued at), exp (expiry), custom claims (roles, email)
  • Signature: Ensures the token hasn't been tampered with. Created by signing header + payload with a secret key (HMAC) or private key (RSA).
  • Flow in Spring Security:
    1. Client sends credentials to /auth/login
    2. Server validates credentials, generates JWT signed with secret key
    3. Client stores JWT (memory or localStorage) and sends Authorization: Bearer {token} on every request
    4. Spring Security's JWT filter validates the token, extracts claims, and sets the SecurityContext
@Component
public class JwtTokenProvider {
    @Value("${jwt.secret}") private String secret;
    @Value("${jwt.expiration}") private long expiration; // milliseconds

    public String generateToken(UserDetails user) {
        return Jwts.builder()
            .setSubject(user.getUsername())
            .claim("roles", user.getAuthorities().stream()
                .map(GrantedAuthority::getAuthority).collect(Collectors.toList()))
            .setIssuedAt(new Date())
            .setExpiration(new Date(System.currentTimeMillis() + expiration))
            .signWith(Keys.hmacShaKeyFor(secret.getBytes()), SignatureAlgorithm.HS256)
            .compact();
    }
}
Interview tip: JWT is self-contained and stateless — the server doesn't need to look up a session in a database on every request. This makes JWTs ideal for microservices where shared session state is difficult. The trade-off: you cannot immediately revoke a JWT before it expires.

How do you implement JWT authentication in Spring Boot?
// Step 1: JWT filter
@Component
public class JwtAuthenticationFilter extends OncePerRequestFilter {

    @Autowired private JwtTokenProvider jwtProvider;
    @Autowired private UserDetailsService userDetailsService;

    @Override
    protected void doFilterInternal(HttpServletRequest req, HttpServletResponse res,
            FilterChain chain) throws ServletException, IOException {

        String authHeader = req.getHeader("Authorization");
        if (authHeader == null || !authHeader.startsWith("Bearer ")) {
            chain.doFilter(req, res);
            return;
        }

        String token = authHeader.substring(7);
        if (jwtProvider.isValid(token)) {
            String username = jwtProvider.getUsername(token);
            UserDetails user = userDetailsService.loadUserByUsername(username);
            UsernamePasswordAuthenticationToken auth =
                new UsernamePasswordAuthenticationToken(user, null, user.getAuthorities());
            auth.setDetails(new WebAuthenticationDetailsSource().buildDetails(req));
            SecurityContextHolder.getContext().setAuthentication(auth);
        }
        chain.doFilter(req, res);
    }
}

// Step 2: Register filter in security config
http.addFilterBefore(jwtAuthFilter, UsernamePasswordAuthenticationFilter.class);
Interview tip: Always use OncePerRequestFilter for JWT filters — it guarantees the filter executes exactly once per request regardless of any filter chain forwarding. Also ensure the filter sets the SecurityContext before calling chain.doFilter().

What is the role of filters in Spring Security?
  • Spring Security processes every HTTP request through a filter chain (SecurityFilterChain) — a sequence of servlet filters applied before the request reaches the controller.
  • Key built-in filters (in order):
    • DisableEncodeUrlFilter — prevents session IDs in URLs
    • SecurityContextPersistenceFilter — restores SecurityContext from session (stateful apps)
    • HeaderWriterFilter — adds security headers (HSTS, X-Frame-Options)
    • CsrfFilter — validates CSRF tokens
    • LogoutFilter — handles logout requests
    • UsernamePasswordAuthenticationFilter — processes form-based login
    • BasicAuthenticationFilter — processes HTTP Basic Auth
    • ExceptionTranslationFilter — converts security exceptions to HTTP responses
    • AuthorizationFilter — enforces access control rules
  • Custom filters (like JWT filter) are inserted at specific positions using addFilterBefore(), addFilterAfter(), or addFilterAt().
Interview tip: The filter chain is the foundation of Spring Security. Understanding the order matters — your JWT filter must execute before UsernamePasswordAuthenticationFilter to set the authentication before Spring Security's default authentication checks run.

What is OAuth2 and how does it work with Spring Security?
  • OAuth2 is an open authorization framework that enables applications to obtain limited access to user accounts on a third-party service (Google, GitHub, Keycloak) without exposing the user's credentials.
  • Roles in OAuth2:
    • Resource Owner: The user who owns the data
    • Client: Your Spring Boot application requesting access
    • Authorization Server: Issues tokens (Keycloak, Auth0, AWS Cognito, Google)
    • Resource Server: API protected by OAuth2 tokens (your Spring Boot API)
  • Spring Security OAuth2 roles:
    • spring-boot-starter-oauth2-client — for apps that log users in via OAuth2 (client role)
    • spring-boot-starter-oauth2-resource-server — for APIs that accept and validate JWT/opaque tokens
# Resource server configuration (Spring Boot 3.x)
spring:
  security:
    oauth2:
      resourceserver:
        jwt:
          issuer-uri: https://auth.mycompany.com/realms/myapp
          # Spring Security auto-discovers JWKS endpoint from issuer URI
Interview tip: In microservice architectures, each service is an OAuth2 Resource Server. Services don't manage authentication themselves — they simply validate the JWT access token from the Authorization Server. This centralizes identity management.

What is the difference between @EnableWebSecurity and WebSecurityConfigurerAdapter?
  • @EnableWebSecurity enables Spring Security's web security support. In Spring Boot, it's automatically triggered by auto-configuration when Spring Security is on the classpath — you still add it on your custom @Configuration class to signal customization intent.
  • WebSecurityConfigurerAdapter was the traditional approach (Spring Security 5.x and earlier) for security configuration. You extended it and overrode configure(HttpSecurity), configure(AuthenticationManagerBuilder), etc.
  • Deprecated in Spring Security 5.7, removed in 6.x: WebSecurityConfigurerAdapter is no longer available in Spring Security 6 / Spring Boot 3. The new approach uses a SecurityFilterChain bean returned from a @Configuration class.
// Old approach (Spring Boot 2.x, deprecated)
@Configuration
public class OldSecurityConfig extends WebSecurityConfigurerAdapter {
    @Override
    protected void configure(HttpSecurity http) throws Exception { ... }
}

// New approach (Spring Boot 3.x / Spring Security 6)
@Configuration
@EnableWebSecurity
public class SecurityConfig {
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception { ... }
}
Interview tip: If someone asks about WebSecurityConfigurerAdapter in 2025, explain it's deprecated and removed. The modern bean-based approach is cleaner — multiple SecurityFilterChain beans can coexist with different @Order values (e.g., API security vs admin security).

How do you customize Spring Security configuration in Spring Boot 3.x (no WebSecurityConfigurerAdapter)?
@Configuration
@EnableWebSecurity
@EnableMethodSecurity  // Enable @PreAuthorize, @PostAuthorize, @Secured
public class SecurityConfig {

    private final JwtAuthenticationFilter jwtFilter;
    private final UserDetailsService userDetailsService;

    @Bean
    public SecurityFilterChain apiFilterChain(HttpSecurity http) throws Exception {
        return http
            .securityMatcher("/api/**")  // Apply only to /api/** routes
            .csrf(AbstractHttpConfigurer::disable)
            .sessionManagement(s -> s.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/auth/**").permitAll()
                .requestMatchers("/api/admin/**").hasRole("ADMIN")
                .anyRequest().authenticated()
            )
            .authenticationProvider(authenticationProvider())
            .addFilterBefore(jwtFilter, UsernamePasswordAuthenticationFilter.class)
            .exceptionHandling(e -> e
                .authenticationEntryPoint((req, res, ex) ->
                    res.sendError(HttpServletResponse.SC_UNAUTHORIZED, "Unauthorized"))
                .accessDeniedHandler((req, res, ex) ->
                    res.sendError(HttpServletResponse.SC_FORBIDDEN, "Forbidden"))
            )
            .build();
    }

    @Bean
    public AuthenticationProvider authenticationProvider() {
        DaoAuthenticationProvider provider = new DaoAuthenticationProvider();
        provider.setUserDetailsService(userDetailsService);
        provider.setPasswordEncoder(passwordEncoder());
        return provider;
    }

    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
    }

    @Bean
    public AuthenticationManager authenticationManager(AuthenticationConfiguration config) throws Exception {
        return config.getAuthenticationManager();
    }
}
Interview tip: The lambda-based DSL in Spring Boot 3.x is more readable and type-safe than the old method-chaining style. Multiple SecurityFilterChain beans with different securityMatcher values allow fine-grained, modular security configuration.

What is stateless authentication and why is it important for REST APIs?
  • Stateless authentication means the server does not store any session state between requests. Each request must carry all the authentication information needed for the server to validate it independently.
  • Why it matters for REST APIs:
    • Scalability: Any server instance can handle any request — no "sticky sessions" or shared session store needed. Perfect for horizontally scaled microservices.
    • Simplicity: No session management infrastructure (Redis session store, replication) required.
    • Cloud-native: Containers and serverless functions can be scaled up/down freely — no session affinity needed.
    • REST principle compliance: REST requires statelessness — each request must be self-contained.
  • Implementation: SessionCreationPolicy.STATELESS in Spring Security prevents session creation/use. JWT tokens carry all auth info in each request.
Interview tip: Setting SessionCreationPolicy.STATELESS alone doesn't make your app stateless — you must also ensure your controllers don't use HttpSession. In stateless JWT-based apps, state lives in the token and the database, not in server memory.

How do you secure endpoints with method-level security (@PreAuthorize, @Secured)?
// Enable with @EnableMethodSecurity (Spring Boot 3.x)
// or @EnableGlobalMethodSecurity (Spring Boot 2.x)

@Service
public class OrderService {

    // SpEL expression — fine-grained control
    @PreAuthorize("hasRole('ADMIN') or (hasRole('USER') and #userId == authentication.principal.id)")
    public List<Order> getOrdersByUser(Long userId) { ... }

    // Check permission AFTER method execution
    @PostAuthorize("returnObject.customerId == authentication.principal.id or hasRole('ADMIN')")
    public Order getOrder(Long orderId) { ... }

    // Filter collection — only return items user can access
    @PostFilter("filterObject.customerId == authentication.principal.id")
    public List<Order> getAllOrders() { ... }

    // Simple role-based (less flexible)
    @Secured({"ROLE_ADMIN", "ROLE_MANAGER"})
    public void deleteOrder(Long orderId) { ... }
}
  • @PreAuthorize — evaluated before method execution; blocks if false.
  • @PostAuthorize — evaluated after; useful when access depends on return value.
  • @PreFilter / @PostFilter — filters collection inputs/outputs.
  • @Secured — simpler, only checks roles, no SpEL.
Interview tip: Method-level security is the most powerful pattern for complex authorization logic. The SpEL expressions can access Spring beans (@myBean.canAccess(...)), parameters (#paramName), and authentication details. This enables domain-driven authorization policies.
