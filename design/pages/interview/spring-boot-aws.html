How do you deploy a Spring Boot application to AWS?
  • Option 1 — AWS Elastic Beanstalk: PaaS — upload the Spring Boot JAR/WAR, EB manages EC2, load balancer, auto-scaling, and deployment. Simplest path for managed deployments.
  • Option 2 — Amazon ECS (Elastic Container Service): Containerize the app (Docker), push image to ECR, deploy via ECS tasks on EC2 or Fargate (serverless containers). Ideal for microservices.
  • Option 3 — Amazon EKS (Elastic Kubernetes Service): Deploy Spring Boot containers on a managed Kubernetes cluster. Best for complex microservice deployments with advanced orchestration needs.
  • Option 4 — AWS Lambda: Serverless deployment — Spring Boot as a Lambda function (using Spring Cloud Function or AWS Lambda Adapter). Best for event-driven, low-traffic use cases. Cold start is the main challenge.
  • Option 5 — EC2 directly: Run JAR on EC2 with a systemd service. Manual but full control.
  • Typical CI/CD pipeline: GitHub Actions/CodePipeline → Build JAR → Docker build + push to ECR → ECS/EKS deployment via rolling update.
Interview tip: Know the trade-offs: Elastic Beanstalk is fastest to get started but less control; ECS/Fargate is the sweet spot for most microservices (containerized, managed infra, pay-per-use); EKS is for organizations already invested in Kubernetes.

What are the best practices for securing AWS RDS credentials in Spring Boot?
  • Never hardcode credentials in application.properties or commit them to version control — this is a critical security violation.
  • Best practices (in order of preference):
    1. AWS Secrets Manager: Store credentials as JSON. Spring Cloud AWS auto-integrates: spring.config.import=aws-secretsmanager:/myapp/db-credentials
    2. AWS Systems Manager Parameter Store: Store individual parameters (/myapp/prod/DB_PASSWORD) and reference via spring.config.import=aws-parameterstore:/myapp/prod/
    3. IAM Database Authentication: Use temporary tokens — no passwords at all. Requires the RDS instance to have IAM authentication enabled and the application's IAM role granted access.
    4. Environment variables: Inject at runtime by ECS task definition or Kubernetes Secrets — never bake into the Docker image.
  • Rotation: Enable automatic credential rotation in Secrets Manager (30-day rotation) — Spring Cloud AWS can handle rotation transparently.
Interview tip: The gold standard is IAM Database Authentication for RDS — zero static credentials, short-lived tokens, and full CloudTrail audit of every connection. If IAM auth is not feasible, AWS Secrets Manager with automatic rotation is the next best option.

How do you integrate Spring Boot with AWS S3?
// Spring Cloud AWS S3 integration
@Service
public class S3FileService {

    private final S3Client s3Client;
    private final String bucketName;

    public S3FileService(S3Client s3Client, @Value("${aws.s3.bucket}") String bucketName) {
        this.s3Client = s3Client;
        this.bucketName = bucketName;
    }

    public String uploadFile(String key, InputStream content, long contentLength, String contentType) {
        s3Client.putObject(PutObjectRequest.builder()
            .bucket(bucketName)
            .key(key)
            .contentType(contentType)
            .contentLength(contentLength)
            .serverSideEncryption(ServerSideEncryption.AES256)
            .build(),
            RequestBody.fromInputStream(content, contentLength));

        return "https://" + bucketName + ".s3.amazonaws.com/" + key;
    }

    public InputStream downloadFile(String key) {
        return s3Client.getObject(GetObjectRequest.builder()
            .bucket(bucketName).key(key).build());
    }

    // Pre-signed URL for direct browser download (avoids proxying through backend)
    public String generatePresignedUrl(String key, Duration expiry) {
        S3Presigner presigner = S3Presigner.create();
        return presigner.presignGetObject(r -> r
            .getObjectRequest(g -> g.bucket(bucketName).key(key))
            .signatureDuration(expiry)).url().toString();
    }
}
Interview tip: For user file downloads, generate pre-signed S3 URLs instead of streaming through the backend — it reduces application load, avoids bandwidth costs, and leverages S3's CDN capabilities. Pre-signed URLs expire after a set duration, preventing unauthorized access.

How do you upload and download files to S3 using Spring Boot?
@RestController
@RequestMapping("/api/files")
public class FileController {

    private final S3FileService s3Service;

    // Upload via multipart form
    @PostMapping("/upload")
    public ResponseEntity<UploadResponse> upload(
            @RequestParam("file") MultipartFile file,
            @RequestParam("category") String category) throws IOException {

        String key = category + "/" + UUID.randomUUID() + "-" + file.getOriginalFilename();
        String url = s3Service.uploadFile(key, file.getInputStream(),
                                          file.getSize(), file.getContentType());
        return ResponseEntity.ok(new UploadResponse(key, url));
    }

    // Download file directly (small files)
    @GetMapping("/download/{key}")
    public ResponseEntity<InputStreamResource> download(@PathVariable String key) {
        InputStream stream = s3Service.downloadFile(key);
        return ResponseEntity.ok()
            .contentType(MediaType.APPLICATION_OCTET_STREAM)
            .header(HttpHeaders.CONTENT_DISPOSITION, "attachment; filename=" + key)
            .body(new InputStreamResource(stream));
    }

    // Generate pre-signed download URL (for large files)
    @GetMapping("/presigned/{key}")
    public ResponseEntity<String> presignedUrl(@PathVariable String key) {
        String url = s3Service.generatePresignedUrl(key, Duration.ofMinutes(15));
        return ResponseEntity.ok(url);
    }
}
Interview tip: For large file uploads (>100MB), use S3 Multipart Upload API — the AWS SDK handles chunking automatically. For very large files, consider using pre-signed POST/PUT URLs for direct client-to-S3 upload, completely bypassing your backend server.

How do you integrate Spring Boot with AWS SNS?
@Service
public class NotificationService {

    private final SnsClient snsClient;

    @Value("${aws.sns.user-events-topic-arn}")
    private String userEventsTopicArn;

    // Publish message to SNS topic
    public void publishUserRegistered(UserDto user) {
        String message = objectMapper.writeValueAsString(
            Map.of("eventType", "USER_REGISTERED", "userId", user.getId(), "email", user.getEmail()));

        PublishRequest request = PublishRequest.builder()
            .topicArn(userEventsTopicArn)
            .message(message)
            .messageAttributes(Map.of(
                "eventType", MessageAttributeValue.builder()
                    .dataType("String").stringValue("USER_REGISTERED").build()))
            .build();

        snsClient.publish(request);
    }
}
// SNS fan-out pattern:
// SNS Topic "user-events"
//   → SQS queue (email-service subscription)
//   → SQS queue (analytics-service subscription)
//   → HTTP endpoint (webhook subscription)
//   → Email/SMS (direct SNS delivery)
Interview tip: SNS + SQS fan-out is the most powerful messaging pattern in AWS — publish once to SNS, N services receive via their own SQS queues independently. Each service processes at its own pace, can replay from SQS, and failures in one service don't affect others.

What are the best practices for building serverless applications with Spring Boot on AWS?
  • Minimize package size: Use Spring Native (GraalVM) for native image compilation — reduces JAR from 50MB to 10MB and startup from 3s to 100ms.
  • Use Spring Cloud Function: Write business logic as Function<Input, Output> — portable across Lambda, Azure Functions, and local execution.
  • Handle cold starts: Use Lambda SnapStart, Provisioned Concurrency for latency-sensitive functions, or warming strategies (scheduled ping every 5 minutes).
  • Stateless design: Each Lambda invocation is independent. Don't maintain in-memory state — use DynamoDB, ElastiCache, or S3 for state.
  • Connection pooling caution: Lambda functions share database connections poorly — use RDS Proxy to pool connections and prevent DB connection exhaustion from concurrent Lambda invocations.
  • Observability: Use AWS X-Ray for distributed tracing, CloudWatch for logs and metrics, with structured logging for searchability.
  • Timeout and memory tuning: Profile and set appropriate timeout (default: 3s, max: 15min). More memory = more CPU allocation in Lambda.
Interview tip: RDS Proxy is critical for Lambda + RDS setups. Without it, each Lambda invocation opens a DB connection, and with 1000 concurrent invocations, you'll exhaust PostgreSQL's connection limit (default: 100). RDS Proxy multiplexes connections between thousands of Lambda functions.

@Service

    @Cacheable(value = "products", key = "#id")
    public ProductDto getProduct(Long id) {
        return productRepo.findById(id).map(mapper::toDto)
            .orElseThrow(() -> new ResourceNotFoundException("Product", id));
    }

How do you secure Spring Boot applications running on AWS (IAM roles, VPCs, security groups)?
  • IAM Roles (no static credentials): Assign IAM roles to EC2 instances, ECS tasks, or Lambda functions. The AWS SDK automatically uses the instance/task role — no access keys in code or config.
  • Principle of least privilege: Each service role has only the permissions it needs. Order Service role: SQS SendMessage + S3 PutObject only. Not *.
  • VPC isolation:
    • Application servers in private subnets — not directly internet-accessible
    • RDS, ElastiCache in private subnets — only reachable from application subnet
    • Public-facing load balancer in public subnet
    • NAT Gateway for outbound internet from private subnets
  • Security Groups (stateful firewall):
    • App SG: Allow inbound 8080 from ALB SG only
    • RDS SG: Allow inbound 5432 from App SG only
    • ALB SG: Allow inbound 443 from 0.0.0.0/0
  • Encryption at rest and in transit: Enable RDS encryption, S3 server-side encryption (SSE-S3 or SSE-KMS), ElastiCache encryption, TLS for all service-to-service communication.
Interview tip: The defense-in-depth model: VPC isolates the network perimeter, Security Groups restrict inbound/outbound at the instance level, IAM controls API access, and Secrets Manager/KMS protects credentials and data. No single layer is sufficient alone.

What is AWS API Gateway and how do you integrate it with Spring Boot microservices?
  • AWS API Gateway is a fully managed API proxy service that sits between clients and backend services. It handles request routing, authentication, rate limiting, caching, and CORS — without running a separate load balancer.
  • Integration options:
    • HTTP Integration: Routes to HTTP endpoints (Spring Boot on ECS/EC2)
    • Lambda Integration: Routes to Lambda functions (Spring Cloud Function apps)
    • VPC Link: Routes to private VPC resources via NLB
  • Features used with Spring Boot microservices:
    • JWT Authorizer — validates tokens before forwarding to Spring Boot
    • Request/response transformation
    • Rate limiting per API key or per user
    • Stage variables for environment-specific routing
Interview tip: AWS API Gateway + Lambda is optimal for serverless microservices. For containerized microservices on ECS, consider ALB (Application Load Balancer) + Spring Cloud Gateway — it's more cost-effective at high traffic volumes (API Gateway charges per million requests).
