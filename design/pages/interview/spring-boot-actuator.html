What is Spring Boot Actuator and why is it used?
  • Spring Boot Actuator is a production-ready sub-project that adds operational monitoring and management capabilities to Spring Boot applications. It exposes endpoints that provide insights into the application's health, metrics, configuration, and runtime state.
  • Why it's used:
    • Health checks: Kubernetes liveness/readiness probes, load balancer health checks, monitoring alerts
    • Metrics: JVM memory, thread pools, HTTP request latency, custom business metrics — feeds into Prometheus/Grafana
    • Configuration visibility: Which properties are active, which beans are registered, which auto-configurations applied
    • Diagnostics: Thread dumps, heap dumps, HTTP trace logs for production debugging
    • Application management: Graceful shutdown, loggers level adjustment at runtime
  • Dependency: spring-boot-starter-actuator
Interview tip: Actuator is a must-have in production Spring Boot applications. Any deployment without health checks is operationally blind — orchestrators (Kubernetes, ECS) cannot replace failing instances without health probe endpoints.

// Secure actuator endpoints with Spring Security
@Bean
public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    return http
        .securityMatcher("/actuator/**")
        .authorizeHttpRequests(auth -> auth
            .requestMatchers("/actuator/health", "/actuator/info").permitAll()
            .requestMatchers("/actuator/**").hasRole("ACTUATOR_ADMIN")
        )
        .httpBasic(Customizer.withDefaults())
        .build();
}
  • Production pattern: Run actuator on a separate internal port (management.server.port=8081) accessible only within the VPC/service mesh — not exposed to external clients.
Interview tip: The safest production setup: separate management port (8081) accessible only internally, expose only health, info, metrics (for Prometheus scraping), and loggers (for runtime log level adjustment). Secure with network policy and Basic Auth.

How do you create custom Actuator endpoints?
@Component
@Endpoint(id = "application-status")  // Accessible at /actuator/application-status
public class ApplicationStatusEndpoint {

    private final List<ExternalService> externalServices;

    @ReadOperation  // HTTP GET
    public ApplicationStatusDto getStatus() {
        Map<String, String> serviceStatuses = externalServices.stream()
            .collect(Collectors.toMap(
                ExternalService::getName,
                s -> s.isAvailable() ? "UP" : "DOWN"
            ));

        return new ApplicationStatusDto(
            "order-service",
            "1.2.3",
            serviceStatuses,
            Runtime.getRuntime().freeMemory()
        );
    }

    @WriteOperation  // HTTP POST
    public void triggerMaintenanceMode(@Selector String action) {
        if ("enable".equals(action)) {
            maintenanceModeService.enable();
        }
    }

    @DeleteOperation  // HTTP DELETE
    public void clearApplicationCache() {
        cacheManager.clearAll();
    }
}
  • @ReadOperation → GET, @WriteOperation → POST, @DeleteOperation → DELETE
  • Use @Selector for path segments: /actuator/application-status/enable
  • Can also be exposed as JMX beans via @JmxEndpoint or @WebEndpoint for HTTP-only.
Interview tip: Custom Actuator endpoints are ideal for exposing business-level health indicators — "are all payment gateways reachable?", "is the feature flag cache populated?", "how many orders are in PENDING state?" — operational insights beyond technical JVM metrics.

What is Micrometer in Spring Boot?
  • Micrometer is a vendor-neutral application metrics instrumentation library — the SLF4J of metrics. It provides a common API for recording metrics that can be exported to any monitoring backend (Prometheus, Datadog, CloudWatch, InfluxDB, Graphite, etc.).
  • Core metric types:
    • Counter — monotonically increasing count (HTTP requests, errors, events)
    • Gauge — current value (active connections, queue size, cache entries)
    • Timer — measures latency and throughput (method execution time, HTTP response time)
    • DistributionSummary — records distribution of values (response sizes, order amounts)
    • LongTaskTimer — tracks long-running operations
  • Spring Boot auto-configures Micrometer with instrumentation for JVM, Tomcat, Spring MVC, Spring Data, Spring Cloud, and more.
@Service
public class OrderService {

    private final Counter ordersCreated;
    private final Timer orderProcessingTime;

    public OrderService(MeterRegistry registry) {
        ordersCreated = Counter.builder("orders.created")
            .tag("env", "prod")
            .description("Total orders created")
            .register(registry);

        orderProcessingTime = Timer.builder("order.processing.duration")
            .percentiles(0.5, 0.95, 0.99)
            .register(registry);
    }

    public OrderDto createOrder(CreateOrderRequest request) {
        return orderProcessingTime.record(() -> {
            OrderDto order = processOrder(request);
            ordersCreated.increment();
            return order;
        });
    }
}
Interview tip: Micrometer's tag-based dimensionality is powerful — add tags for environment, service, region, and user tier. This enables Prometheus/Grafana queries like "p95 latency for premium users in us-east-1 in the last 5 minutes."

What is Zipkin and how does it integrate with Spring Boot?
  • Zipkin is an open-source distributed tracing system. It collects timing data from microservices, correlates spans by Trace ID, and provides a UI to visualize:
    • End-to-end request flow across services (Gantt chart view)
    • Latency breakdown per service and operation
    • Service dependency diagram
    • Error annotations and exception messages
  • Integration flow:
    1. Spring Boot sends span data to Zipkin via HTTP POST to /api/v2/spans
    2. Zipkin stores spans in-memory (dev) or Elasticsearch/Cassandra (production)
    3. Zipkin UI correlates spans by Trace ID and renders the waterfall view
  • Running Zipkin locally: docker run -p 9411:9411 openzipkin/zipkin
Interview tip: For production, use Elasticsearch as Zipkin's storage backend — in-memory is for development only. In AWS, consider AWS X-Ray instead of self-hosted Zipkin — it integrates with the AWS ecosystem and doesn't require managing another service.

What are the best practices for monitoring microservices in production?
  • Three pillars of observability:
    • Metrics: Micrometer + Prometheus + Grafana — what is happening (request rate, latency, error rate, resource utilization)
    • Logs: Structured JSON logs → ELK/Loki — what happened (detailed event records)
    • Traces: Micrometer Tracing/Zipkin/Jaeger/X-Ray — why it happened (request flow across services)
  • Alert on SLOs: Error rate > 1%, P99 latency > 2s, DB connection pool exhaustion > 80%
  • Health checks: Liveness probe (is the JVM alive?), Readiness probe (is the service ready to handle traffic?)
  • Capacity metrics: CPU, memory, connection pool usage, queue depth — detect saturation before it impacts users
  • Business metrics: Orders per second, payment success rate, user registrations — detect business anomalies alongside technical ones
  • Runbook automation: Correlate alerts to runbooks so on-call engineers know exactly what to do
Interview tip: Monitoring is not just about technical metrics. Business metrics (orders created, payments processed, cart abandonment rate) are equally important — a technical health check passing while business metrics drop signals a logic bug, not an infrastructure issue.

What is the health check endpoint and why is it important in cloud deployments?
  • /actuator/health reports the application's operational status. Spring Boot auto-creates health indicators for all configured components: database, Redis, Kafka, RabbitMQ, disk space, etc.
  • Response: {"status": "UP"} or {"status": "DOWN", "components": {"db": {"status": "DOWN"}}}
  • Kubernetes probes (Spring Boot 2.3+):
    • /actuator/health/liveness — is the JVM alive and not deadlocked? Restart if DOWN.
    • /actuator/health/readiness — is the service ready to receive traffic? Remove from load balancer if DOWN.
# Kubernetes deployment probe configuration
livenessProbe:
  httpGet:
    path: /actuator/health/liveness
    port: 8080
  initialDelaySeconds: 30
  periodSeconds: 10
  failureThreshold: 3

readinessProbe:
  httpGet:
    path: /actuator/health/readiness
    port: 8080
  initialDelaySeconds: 15
  periodSeconds: 5
  failureThreshold: 3
Interview tip: The distinction between liveness and readiness is critical: liveness restarts pods when the JVM is in a bad state (deadlock, OOM). Readiness keeps pods in the load balancer rotation only when all dependencies (DB, cache) are reachable. Failing readiness doesn't restart the pod; failing liveness does.

How do you enable request/response tracing in Spring Boot?
// Option 1: Spring Boot Actuator httptrace (Spring Boot 2.x)
// Stores last 100 HTTP requests/responses in memory
@Bean
public HttpTraceRepository httpTraceRepository() {
    return new InMemoryHttpTraceRepository();
}
// Access via: GET /actuator/httptrace
// Option 2: Custom logging filter (recommended for production)
@Component
@Order(Ordered.HIGHEST_PRECEDENCE)
public class RequestResponseLoggingFilter extends OncePerRequestFilter {

    @Override
    protected void doFilterInternal(HttpServletRequest request, HttpServletResponse response,
            FilterChain chain) throws ServletException, IOException {

        long startTime = System.currentTimeMillis();
        String requestId = UUID.randomUUID().toString().substring(0, 8);

        MDC.put("requestId", requestId);
        MDC.put("httpMethod", request.getMethod());
        MDC.put("requestUri", request.getRequestURI());

        try {
            chain.doFilter(request, response);
        } finally {
            long duration = System.currentTimeMillis() - startTime;
            log.info("Request completed: method={}, uri={}, status={}, duration={}ms",
                request.getMethod(), request.getRequestURI(), response.getStatus(), duration);
            MDC.clear();
        }
    }
}
Interview tip: Use MDC (Mapped Diagnostic Context) to add contextual information (request ID, user ID, trace ID) to all log statements within the request thread. MDC is thread-local — it propagates through the entire request handling chain automatically.

How do you integrate Spring Boot with centralized logging frameworks?

  • Integration options:
    • ELK Stack: Logstash encoder → Filebeat → Elasticsearch → Kibana

What is OpenTelemetry and how does it relate to Spring Boot observability?
  • OpenTelemetry (OTel) is a CNCF (Cloud Native Computing Foundation) open standard for observability instrumentation — it provides vendor-neutral APIs and SDKs for collecting metrics, logs, and traces (the three observability pillars) from applications.
  • Key components:
    • OTLP (OpenTelemetry Protocol): Standard format for transmitting telemetry data
    • OTel Collector: Receives, processes, and exports telemetry to multiple backends
    • Auto-instrumentation: Java agent that automatically instruments popular frameworks without code changes
  • Spring Boot integration: Micrometer Tracing supports OTel Bridge (micrometer-tracing-bridge-otel). Traces can be exported via OTLP to Jaeger, Tempo (Grafana), Datadog, New Relic, Honeycomb.
<dependency>
    <groupId>io.micrometer</groupId>
    <artifactId>micrometer-tracing-bridge-otel</artifactId>
</dependency>
<dependency>
    <groupId>io.opentelemetry</groupId>
    <artifactId>opentelemetry-exporter-otlp</artifactId>
</dependency>
Interview tip: OpenTelemetry is the future of observability — it prevents vendor lock-in. By instrumenting once with OTel, you can switch between Datadog, Grafana Tempo, Jaeger, and Honeycomb by simply changing the exporter configuration. This is increasingly important in enterprise observability strategies.

How do you configure graceful shutdown in Spring Boot applications?
server:
  shutdown: graceful  # Default is immediate

spring:
  lifecycle:
    timeout-per-shutdown-phase: 30s  # Wait up to 30s for in-flight requests
  • What graceful shutdown does:
    1. Marks the readiness probe as DOWN — Kubernetes stops routing new traffic to this pod
    2. Waits for in-flight requests to complete (up to the timeout)
    3. Closes database connections, Kafka consumers, thread pools cleanly
    4. Runs @PreDestroy methods and Spring context shutdown hooks
    5. JVM exits with code 0
  • Kubernetes grace period alignment: Set Kubernetes terminationGracePeriodSeconds > Spring Boot's timeout + buffer (e.g., Spring Boot 30s timeout → Kubernetes 40s termination period).
# Kubernetes deployment
spec:
  template:
    spec:
      terminationGracePeriodSeconds: 40  # Must exceed Spring Boot's shutdown timeout
      containers:
        - lifecycle:
            preStop:
              exec:
                command: ["/bin/sh", "-c", "sleep 5"]  # Small delay for load balancer routing update
Interview tip: Graceful shutdown is non-negotiable for zero-downtime deployments. Without it, rolling updates kill pods mid-request, causing 5xx errors. The preStop hook adds a brief sleep to allow the load balancer to remove the pod before the process starts draining.

What are performance tuning tips for Spring Boot in production?
  • JVM tuning:
    • Use G1GC (default in Java 11+) or ZGC for low-latency applications
    • Set -Xms = -Xmx to avoid GC overhead from heap resizing
    • Enable JVM ergonomics: -XX:+UseContainerSupport for Docker/Kubernetes CPU/memory awareness
  • Spring Boot:
    • Enable lazy initialization in dev: spring.main.lazy-initialization=true
    • Use @Transactional(readOnly = true) — skips dirty checking for read queries
    • Enable Spring Boot's layered JAR for faster Docker image builds
  • Database:
    • Tune HikariCP pool size: maximumPoolSize = (cpu_cores * 2) + spindle_count
    • Enable second-level cache (Ehcache/Redis) for frequently-read, rarely-changed data
    • Use @BatchSize or JOIN FETCH to avoid N+1 query problems
  • Web layer:
    • Set appropriate Tomcat thread pool: server.tomcat.threads.max=200
    • Enable HTTP/2: server.http2.enabled=true
    • Use @Cacheable for expensive, stable computations
    • Enable response compression: server.compression.enabled=true
Interview tip: Always profile before optimizing. Use Micrometer metrics + Grafana to identify actual bottlenecks (is it GC, DB queries, external API latency?). Premature optimization wastes time — measure, identify, fix, measure again.

How do you handle thread pool tuning in Spring Boot?
# Tomcat thread pool (synchronous web)
server:
  tomcat:
    threads:
      max: 200          # Max concurrent request handler threads
      min-spare: 10     # Min always-alive threads
    connection-timeout: 20000  # 20s connection timeout
    max-connections: 8192      # Max simultaneous connections
    accept-count: 100          # Queue size when all threads busy

# Async executor
spring:
  task:
    execution:
      pool:
        core-size: 5
        max-size: 20
        queue-capacity: 500
        keep-alive: 60s
      thread-name-prefix: "async-"
// HikariCP connection pool tuning
spring:
  datasource:
    hikari:
      maximum-pool-size: 10   # (cpu_cores * 2) + effective_spindle_count
      minimum-idle: 5
      idle-timeout: 600000    # 10 minutes
      max-lifetime: 1800000   # 30 minutes (less than DB connection timeout)
      connection-timeout: 30000
  • Guideline for Tomcat threads: Too many threads = context switching overhead + excessive memory. Too few = request queuing under load. Profile under realistic load with Gatling/JMeter to find the right size.
Interview tip: For I/O-bound applications, more threads are beneficial (threads wait on I/O, not CPU). For CPU-bound applications, more threads than CPU cores causes context switching overhead. Use Reactive (WebFlux) for highly concurrent I/O-bound scenarios instead of increasing thread pool size.

What are the common production pitfalls in Spring Boot and how to avoid them?
  • N+1 query problem: @OneToMany with LAZY fetch + loop → 1 query for list + N queries for each item. Fix: JOIN FETCH or @EntityGraph. Detect: spring.jpa.show-sql=true + Hibernate statistics.
  • Thread pool exhaustion: All Tomcat threads blocked waiting on slow external calls → 503 errors. Fix: Set timeouts on all external calls, use circuit breakers, or switch to reactive stack.
  • Memory leaks: Static collection holding references to request-scoped objects, ThreadLocal variables not cleaned up. Fix: Review static fields, always clean up ThreadLocal in finally blocks.
  • Connection pool exhaustion: Too many concurrent DB operations, or connections not returned due to exceptions. Fix: Tune HikariCP pool size, set connection timeout, use try-with-resources, ensure transactions are bounded.
  • Unprotected Actuator endpoints: Leaking config, thread dumps, bean definitions. Fix: Secure actuator with security config or separate management port on private network.
  • Swallowed exceptions in @Transactional: Catching and logging without rethrowing — transaction commits with partial data. Fix: Rethrow or use TransactionAspectSupport.currentTransactionStatus().setRollbackOnly().
  • Missing @Async proxy bypass: Calling @Async from the same class — proxy not invoked, runs synchronously. Fix: Call from different bean, or use @Autowired self injection (with caution).
  • Database schema drift in production: Using ddl-auto=update in production. Fix: Always use ddl-auto=validate or none in production; use Flyway/Liquibase for migrations.
Interview tip: Knowing these pitfalls and their root causes demonstrates production experience. The best interviewers ask "have you ever encountered X in production?" — having concrete examples of these issues, their symptoms, and how you fixed them makes a very strong impression.
