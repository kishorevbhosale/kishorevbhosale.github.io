How do you create a REST API in Spring Boot?
// 1. Add spring-boot-starter-web dependency
// 2. Create a @RestController
@RestController
@RequestMapping("/api/v1/products")
public class ProductController {

    private final ProductService productService;

    public ProductController(ProductService productService) {
        this.productService = productService;
    }

    @GetMapping
    public ResponseEntity<Page<ProductDto>> getAllProducts(
            @RequestParam(defaultValue = "0") int page,
            @RequestParam(defaultValue = "20") int size) {
        return ResponseEntity.ok(productService.findAll(PageRequest.of(page, size)));
    }

    @GetMapping("/{id}")
    public ResponseEntity<ProductDto> getProduct(@PathVariable Long id) {
        return ResponseEntity.ok(productService.findById(id));
    }

    @PostMapping
    public ResponseEntity<ProductDto> createProduct(@Valid @RequestBody CreateProductRequest request) {
        ProductDto created = productService.create(request);
        URI location = URI.create("/api/v1/products/" + created.getId());
        return ResponseEntity.created(location).body(created);
    }

    @PutMapping("/{id}")
    public ResponseEntity<ProductDto> updateProduct(
            @PathVariable Long id,
            @Valid @RequestBody UpdateProductRequest request) {
        return ResponseEntity.ok(productService.update(id, request));
    }

    @DeleteMapping("/{id}")
    public ResponseEntity<Void> deleteProduct(@PathVariable Long id) {
        productService.delete(id);
        return ResponseEntity.noContent().build();
    }
}
Interview tip: Follow REST conventions: use nouns for resources (not verbs), return correct HTTP status codes (201 Created, 204 No Content, 404 Not Found), use ResponseEntity for precise response control, and version your API from day one.

Difference between @Controller and @RestController.
  • @Controller is a Spring MVC annotation that marks the class as a web controller. Handler methods return a view name (String), which is resolved to an HTML template by a ViewResolver (Thymeleaf, JSP).
  • @RestController = @Controller + @ResponseBody. Every handler method's return value is directly serialized (via Jackson) to the HTTP response body as JSON/XML — no view resolution occurs.
  • Adding @ResponseBody to individual methods in a @Controller is equivalent to using @RestController for those methods.
Aspect@Controller@RestController
Return typeView name (String) → HTMLObject → JSON/XML
@ResponseBodyMust add per method or classImplicit on all methods
Use caseServer-side rendered web appsREST APIs, microservices
Interview tip: In microservice architectures, @RestController is the standard. @Controller is still relevant for Spring MVC apps with Thymeleaf or other template engines.

What is the role of @RequestMapping annotation?
  • @RequestMapping maps HTTP requests to handler methods (or controller classes). It's the foundational URL mapping annotation in Spring MVC.
  • Attributes:
    • value/path — URL pattern(s)
    • method — HTTP method (GET, POST, PUT, DELETE, PATCH)
    • consumes — accepted request content types (application/json)
    • produces — response content types (application/json)
    • params — required/excluded request parameters
    • headers — required request headers
  • Modern usage: @GetMapping, @PostMapping, etc. are composed annotations that include @RequestMapping(method = RequestMethod.GET). They're more concise and expressive.
// Class-level: base path for all methods
@RestController
@RequestMapping(path = "/api/v1/users", produces = MediaType.APPLICATION_JSON_VALUE)
public class UserController {

    // Method-level: combined with class-level = /api/v1/users/{id}
    @RequestMapping(value = "/{id}", method = RequestMethod.GET)
    public UserDto getUser(@PathVariable Long id) { ... }
}
Interview tip: Always define the base path at the class level with @RequestMapping and use @GetMapping, @PostMapping, etc. for specific methods. This keeps paths DRY and makes the controller's URL structure clear at a glance.

Difference between @GetMapping, @PostMapping, @PutMapping, @DeleteMapping, and @PatchMapping.
AnnotationHTTP MethodSemanticsIdempotent?
@GetMappingGETRetrieve resource — safe, no side effectsYes
@PostMappingPOSTCreate new resource; response includes locationNo
@PutMappingPUTReplace entire resource (full update)Yes
@PatchMappingPATCHPartial update — only provided fields changeConditionally
@DeleteMappingDELETEDelete resourceYes
  • All are shorthand for @RequestMapping(method = RequestMethod.XXX).
  • PUT vs PATCH: PUT requires the full resource in the body; missing fields are set to null/default. PATCH allows partial updates — only provided fields are changed. Use PATCH when clients send only the changed fields.
Interview tip: HTTP method semantics matter for API design and caching. GET responses can be cached by browsers/CDNs. POST creates a new resource and returns 201 Created with a Location header. PATCH is becoming preferred over PUT for partial updates in modern APIs.

// Basic path variable

// Named path variable (when parameter name differs)
@GetMapping("/users/{userId}/orders/{orderId}")
public ResponseEntity<OrderDto> getUserOrder(
        @PathVariable("userId") Long userId,

// Optional path variable (Spring 4.3.3+)
@GetMapping({"/users/{id}", "/users"})
public ResponseEntity<?> getUsers(
        @PathVariable(value = "id", required = false) Long id) {
    if (id != null) {
        return ResponseEntity.ok(userService.findById(id));
    }
    return ResponseEntity.ok(userService.findAll());
}

// Pattern constraints in path variable
@GetMapping("/orders/{id:\\d+}")  // id must be digits only
public ResponseEntity<OrderDto> getOrder(@PathVariable Long id) { ... }
Interview tip: Spring Boot automatically converts path variable strings to the declared parameter type (Long, UUID, Enum, etc.) using ConversionService. If conversion fails (e.g., passing "abc" for a Long), Spring returns a 400 Bad Request automatically.

How do you send and receive JSON in Spring Boot?
  • Spring Boot's spring-boot-starter-web includes Jackson — the default JSON serializer/deserializer. Jackson is auto-configured as a MappingJackson2HttpMessageConverter.
  • Receiving JSON: Use @RequestBody on the method parameter — Jackson deserializes the request body JSON into a Java object.
  • Sending JSON: Return a Java object from a @RestController method — Jackson serializes it to JSON in the response body.
  • Content negotiation: Client sends Content-Type: application/json for requests with body; Spring returns JSON when Accept: application/json is set.
// Receive JSON
@PostMapping("/orders")
public ResponseEntity<OrderDto> createOrder(@RequestBody CreateOrderRequest request) {
    // request is automatically deserialized from JSON
    return ResponseEntity.ok(orderService.create(request));
}

// Jackson customization
@Bean
public ObjectMapper objectMapper() {
    return new ObjectMapper()
        .configure(DeserializationFeature.FAIL_ON_UNKNOWN_PROPERTIES, false)
        .registerModule(new JavaTimeModule())
        .disable(SerializationFeature.WRITE_DATES_AS_TIMESTAMPS);
}
Interview tip: Register JavaTimeModule to properly serialize/deserialize Java 8 date/time types (LocalDate, ZonedDateTime). Without it, Jackson serializes dates as arrays by default — a common issue in new Spring Boot projects.

What is the role of @RequestBody annotation?
  • @RequestBody maps the HTTP request body to a Java object. Spring uses the appropriate HttpMessageConverter (Jackson for JSON, JAXB for XML) to deserialize the body.
  • It reads the raw bytes from the request stream, applies content type negotiation via Content-Type header, and converts to the target type.
  • With validation: Combine with @Valid to trigger Bean Validation on the deserialized object.
  • Immutable DTOs: Jackson works with records and Lombok @Value/@Builder classes — use @JsonCreator for constructor-based deserialization.
public record CreateUserRequest(
    @NotBlank String name,
    @Email String email,
    @NotNull @Min(18) Integer age
) {}

@PostMapping("/users")
public ResponseEntity<UserDto> createUser(@Valid @RequestBody CreateUserRequest request) {
    // If validation fails, Spring returns 400 Bad Request with error details
    return ResponseEntity.status(201).body(userService.create(request));
}
Interview tip: @RequestBody reads the request body exactly once from the input stream. If you need to read it multiple times (e.g., for logging), use a ContentCachingRequestWrapper in a filter to buffer the body.

What is the role of @ResponseEntity?
  • ResponseEntity<T> represents the entire HTTP response — status code, headers, and body — giving you full control over what Spring sends back to the client.
  • Without ResponseEntity: Returning an object from a method uses default status 200 OK with no custom headers.
  • With ResponseEntity: You control the status code, add headers (Location, custom headers), and optionally include or exclude the body.
// 201 Created with Location header
@PostMapping
public ResponseEntity<ProductDto> create(@Valid @RequestBody CreateProductRequest req) {
    ProductDto product = productService.create(req);
    return ResponseEntity
        .created(URI.create("/api/products/" + product.getId()))
        .body(product);
}

// 204 No Content
@DeleteMapping("/{id}")
public ResponseEntity<Void> delete(@PathVariable Long id) {
    productService.delete(id);
    return ResponseEntity.noContent().build();
}

// 200 with custom headers
@GetMapping("/{id}")
public ResponseEntity<ProductDto> get(@PathVariable Long id) {
    return ResponseEntity.ok()
        .header("X-Product-Source", "database")
        .cacheControl(CacheControl.maxAge(1, TimeUnit.HOURS))
        .body(productService.findById(id));
}
Interview tip: Use ResponseEntity consistently in REST controllers to communicate intent clearly. Returning a bare object with a 200 is fine for GET, but 201 for POST, 204 for DELETE, and 202 Accepted for async operations are important REST semantic signals.

How do you implement global exception handling in Spring Boot?
@RestControllerAdvice  // = @ControllerAdvice + @ResponseBody
public class GlobalExceptionHandler {

    // Handle custom business exception
    @ExceptionHandler(ResourceNotFoundException.class)
    @ResponseStatus(HttpStatus.NOT_FOUND)
    public ErrorResponse handleNotFound(ResourceNotFoundException ex) {
        return new ErrorResponse("NOT_FOUND", ex.getMessage(), LocalDateTime.now());
    }

    // Handle validation failures
    @ExceptionHandler(MethodArgumentNotValidException.class)
    @ResponseStatus(HttpStatus.BAD_REQUEST)
    public ValidationErrorResponse handleValidation(MethodArgumentNotValidException ex) {
        Map<String, String> fieldErrors = ex.getBindingResult().getFieldErrors().stream()
            .collect(Collectors.toMap(FieldError::getField, FieldError::getDefaultMessage));
        return new ValidationErrorResponse("VALIDATION_FAILED", fieldErrors);
    }

    // Handle unexpected errors
    @ExceptionHandler(Exception.class)
    @ResponseStatus(HttpStatus.INTERNAL_SERVER_ERROR)
    public ErrorResponse handleGeneric(Exception ex, HttpServletRequest request) {
        log.error("Unhandled exception for request {}", request.getRequestURI(), ex);
        return new ErrorResponse("INTERNAL_ERROR", "An unexpected error occurred", LocalDateTime.now());
    }
}

public record ErrorResponse(String code, String message, LocalDateTime timestamp) {}
Interview tip: Never expose stack traces or internal implementation details (class names, SQL errors) in API error responses — this is a security vulnerability. Log internally with full details; return structured, sanitized error codes and messages to clients.

What is the use of @ControllerAdvice?
  • @ControllerAdvice is a specialization of @Component that provides cross-cutting concerns for all controllers in the application — or a subset specified by package, annotation, or type.
  • Capabilities:
    • @ExceptionHandler methods — global exception handling across all controllers
    • @ModelAttribute methods — add model attributes to all controllers (for view-based apps)
    • @InitBinder methods — configure data binding and validation globally
  • Scoping: Can be limited to specific controllers via @ControllerAdvice(basePackages = "com.app.api") or @ControllerAdvice(assignableTypes = OrderController.class).
  • For REST APIs: Use @RestControllerAdvice (= @ControllerAdvice + @ResponseBody) so exception handler methods return JSON automatically.
Interview tip: A well-designed global exception handler with @RestControllerAdvice is a requirement in any production REST API. It ensures consistent error response format, centralized logging, and prevents leaking implementation details.

What is the difference between @ExceptionHandler and @ControllerAdvice?
  • @ExceptionHandler — annotates a method that handles a specific exception. When placed inside a controller, it handles exceptions thrown only by that controller.
  • @ControllerAdvice — a class-level annotation that makes exception handlers globally applicable across all controllers. Methods inside it annotated with @ExceptionHandler intercept matching exceptions from any controller.
  • Resolution order:
    • Controller-local @ExceptionHandler takes priority over global @ControllerAdvice
    • More specific exception types match before their parent classes
// Local handler — only for UserController
@RestController
public class UserController {
    @ExceptionHandler(UserNotFoundException.class)
    public ResponseEntity<?> handleUserNotFound(UserNotFoundException ex) {
        return ResponseEntity.status(404).body(ex.getMessage());
    }
}

// Global handler — applies to ALL controllers
@RestControllerAdvice
public class GlobalHandler {
    @ExceptionHandler(ResourceNotFoundException.class)
    public ResponseEntity<?> handleNotFound(ResourceNotFoundException ex) {
        return ResponseEntity.status(404).body(new ErrorResponse(...));
    }
}
Interview tip: Best practice is to use a single @RestControllerAdvice for global exception handling. Controller-local @ExceptionHandler is only appropriate when a controller-specific exception needs unique handling not applicable elsewhere.

How do you perform input validation in Spring Boot?
// 1. Add spring-boot-starter-validation dependency

public class CreateUserRequest {
    @NotBlank(message = "Name is required")
    @Size(min = 2, max = 100)
    private String name;

    @Email(message = "Invalid email format")
    @NotNull
    private String email;

    @NotNull @Min(value = 18, message = "Must be 18 or older")
    private Integer age;

    @Pattern(regexp = "^[0-9]{10}$", message = "Phone must be 10 digits")
    private String phone;
}

// 3. Trigger validation with @Valid in controller
@PostMapping("/users")
public ResponseEntity<UserDto> createUser(@Valid @RequestBody CreateUserRequest request) {
    return ResponseEntity.ok(userService.create(request));
}

// 4. Handle MethodArgumentNotValidException in @ControllerAdvice
@ExceptionHandler(MethodArgumentNotValidException.class)
public ResponseEntity<Map<String, String>> handleValidation(MethodArgumentNotValidException ex) {
    Map<String, String> errors = ex.getBindingResult().getFieldErrors().stream()
        .collect(Collectors.toMap(FieldError::getField, FieldError::getDefaultMessage));
    return ResponseEntity.badRequest().body(errors);
}
Interview tip: Validation should happen at the REST layer (controller) — not inside service methods. Service methods should trust that inputs are valid. This separation keeps services clean and testable without HTTP context.

How do you define custom validation constraints?
// Step 1: Define the annotation
@Documented
@Constraint(validatedBy = ValidPasswordValidator.class)
@Target({ElementType.FIELD, ElementType.PARAMETER})
@Retention(RetentionPolicy.RUNTIME)
public @interface ValidPassword {
    String message() default "Password must be at least 8 chars with uppercase, lowercase, and digit";
    Class<?>[] groups() default {};
    Class<? extends Payload>[] payload() default {};
}

// Step 2: Implement the validator
public class ValidPasswordValidator implements ConstraintValidator<ValidPassword, String> {

    private static final Pattern PASSWORD_PATTERN =
        Pattern.compile("^(?=.*[A-Z])(?=.*[a-z])(?=.*\\d).{8,}$");

    @Override
    public boolean isValid(String value, ConstraintValidatorContext context) {
        if (value == null) return false;
        return PASSWORD_PATTERN.matcher(value).matches();
    }
}

// Step 3: Use it
public class RegistrationRequest {
    @ValidPassword
    private String password;
}
Interview tip: Custom validators can inject Spring beans (e.g., UserRepository) to check uniqueness in the database. Spring automatically detects this and makes the validator Spring-managed when it's annotated with @Component.

What is the difference between checked and unchecked exceptions in REST APIs?
  • Checked exceptions (Exception subclasses, not RuntimeException) — compiler requires them to be caught or declared in method signature. In REST APIs, they represent recoverable errors that callers should handle.
  • Unchecked exceptions (RuntimeException subclasses) — no compiler enforcement. Represent programming errors or unexpected conditions. Spring rolls back transactions on these by default.
  • REST API exception strategy:
    • Create custom unchecked business exceptions (extend RuntimeException): ResourceNotFoundException, ConflictException, ValidationException.
    • Handle all exceptions in @RestControllerAdvice and map to appropriate HTTP status codes.
    • Never propagate infrastructure exceptions (JPA, SQL, IO) to the API layer — wrap them in business exceptions.
// Custom unchecked business exceptions
public class ResourceNotFoundException extends RuntimeException {
    public ResourceNotFoundException(String resource, Long id) {
        super(resource + " with id " + id + " not found");
    }
}
// Maps to 404 in GlobalExceptionHandler
Interview tip: In modern Spring Boot REST APIs, the best practice is to use unchecked exceptions for business errors and handle them centrally. This avoids polluting service method signatures with checked exception declarations and keeps the API clean.

How do you implement versioning in REST APIs (URI, Query Params, Headers)?
  • URI versioning (most common): /api/v1/users, /api/v2/users
  • Query parameter versioning: /api/users?version=1
  • Header versioning: Accept: application/vnd.company.v1+json or custom X-API-Version: 1 header
  • Media type (content negotiation): Accept: application/vnd.company.users.v2+json
// URI versioning — most readable, cacheable
@RestController
@RequestMapping("/api/v1/users")
public class UserV1Controller { ... }

@RestController
@RequestMapping("/api/v2/users")
public class UserV2Controller { ... }

// Header versioning
@GetMapping(value = "/users", headers = "X-API-Version=1")
public UserV1Dto getUsersV1() { ... }

@GetMapping(value = "/users", headers = "X-API-Version=2")
public UserV2Dto getUsersV2() { ... }
StrategyProsCons
URIVisible, cacheable, easy to test in browserPollutes URL space
Query paramSimpleNot RESTful, can conflict with filters
HeaderClean URLs, RESTfulHarder to test, not cacheable by standard CDNs
Interview tip: URI versioning is the pragmatic choice for most teams — it's explicit, easy to document, and works well with CDN caching. Header versioning is more "pure" REST but adds friction in client development and testing.

How do you implement HATEOAS in Spring Boot?
  • HATEOAS (Hypermedia as the Engine of Application State) enriches REST responses with links that describe available actions and related resources — making APIs self-discoverable.
  • Spring Boot provides spring-boot-starter-hateoas for HATEOAS support.
// Add spring-boot-starter-hateoas dependency

@RestController
@RequestMapping("/api/orders")
public class OrderController {

    @GetMapping("/{id}")
    public EntityModel<OrderDto> getOrder(@PathVariable Long id) {
        OrderDto order = orderService.findById(id);

        return EntityModel.of(order,
            linkTo(methodOn(OrderController.class).getOrder(id)).withSelfRel(),
            linkTo(methodOn(OrderController.class).cancelOrder(id)).withRel("cancel"),
            linkTo(methodOn(CustomerController.class).getCustomer(order.getCustomerId())).withRel("customer")
        );
    }
}

// Response JSON:
// {
//   "id": 123, "status": "PENDING", "total": 99.99,
//   "_links": {
//     "self": { "href": "/api/orders/123" },
//     "cancel": { "href": "/api/orders/123/cancel" },
//     "customer": { "href": "/api/customers/456" }
//   }
// }
Interview tip: HATEOAS is the Level 3 of Richardson Maturity Model. While it's rarely required, knowing it demonstrates thorough REST knowledge. Most practical APIs implement Level 2 (Resources + HTTP Verbs) and that's sufficient for most enterprise applications.

How do you implement pagination and sorting in REST APIs?
// Repository
public interface ProductRepository extends JpaRepository<Product, Long> {
    Page<Product> findByCategory(String category, Pageable pageable);
}

// Controller
@GetMapping
public ResponseEntity<Page<ProductDto>> getProducts(
        @RequestParam(required = false) String category,
        @RequestParam(defaultValue = "0") int page,
        @RequestParam(defaultValue = "20") int size,
        @RequestParam(defaultValue = "createdAt") String sortBy,
        @RequestParam(defaultValue = "desc") String sortDir) {

    Sort sort = sortDir.equalsIgnoreCase("asc")
        ? Sort.by(sortBy).ascending()
        : Sort.by(sortBy).descending();

    Pageable pageable = PageRequest.of(page, size, sort);
    Page<Product> products = category != null
        ? productRepo.findByCategory(category, pageable)
        : productRepo.findAll(pageable);

    return ResponseEntity.ok(products.map(productMapper::toDto));
}

// Response includes pagination metadata:
// { "content": [...], "totalElements": 150, "totalPages": 8,
//   "number": 0, "size": 20, "last": false }
  • Alternatively, use @PageableDefault(size = 20, sort = "createdAt", direction = DESC) Pageable pageable in the controller parameter — Spring MVC handles deserialization from query params automatically.
Interview tip: Always set a maximum page size to prevent clients requesting size=10000 and crashing your database. Use a custom HandlerMethodArgumentResolver or validation to cap the size parameter.

How do you secure REST APIs in Spring Boot?
  • Authentication approaches:
    • JWT (JSON Web Tokens): Stateless tokens; client sends Authorization: Bearer {token} header. Server validates signature — no session lookup needed. Ideal for microservices.
    • OAuth2 / OpenID Connect: Delegate authentication to an identity provider (Keycloak, AWS Cognito, Auth0). Client gets access tokens from the provider.
    • API Keys: Simple, suitable for server-to-server communication.
  • Spring Security configuration (Spring Boot 3.x):
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        return http
            .csrf(AbstractHttpConfigurer::disable)  // stateless API — no CSRF needed
            .sessionManagement(s -> s.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/auth/**", "/actuator/health").permitAll()
                .requestMatchers("/api/admin/**").hasRole("ADMIN")
                .anyRequest().authenticated()
            )
            .addFilterBefore(jwtAuthFilter, UsernamePasswordAuthenticationFilter.class)
            .build();
    }
}
  • Additional security measures: HTTPS only, rate limiting, input sanitization, CORS configuration, security headers (Spring Security adds them automatically).
Interview tip: A complete REST API security strategy includes: JWT/OAuth2 for authentication, RBAC for authorization, HTTPS in transit, input validation to prevent injection, rate limiting to prevent abuse, and Actuator endpoints secured separately.
