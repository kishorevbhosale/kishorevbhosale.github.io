What is Spring Data JPA and how is it different from Hibernate?
  • Hibernate is a full-featured ORM (Object-Relational Mapping) framework and the most popular JPA (Java Persistence API) implementation. It handles CRUD operations, caching, lazy loading, and schema generation.
  • Spring Data JPA is an abstraction layer built on top of JPA (typically Hibernate). It provides repository interfaces (JpaRepository, CrudRepository) that auto-generate boilerplate CRUD and query method implementations at runtime.

LevelJPA Provider (persistence engine)Repository abstraction on top of JPA

Query derivationManual HQL/JPQLDerived from method names (findByEmailAndActive)
PaginationManual offset/limitBuilt-in Pageable support
ReplaceabilityCan switch to EclipseLinkAbstracts the provider — code doesn't change
Interview tip: Spring Data JPA uses Hibernate as the default JPA provider but abstracts it behind the JPA specification. You can swap Hibernate for EclipseLink without changing application code — a key benefit of the abstraction.

What are the advantages of using Spring Data JPA?
  • Eliminates boilerplate: No more manual EntityManager code for standard CRUD. Extending JpaRepository<T, ID> gives you 17+ built-in methods instantly.
  • Query derivation from method names: findByLastNameAndAgeGreaterThan(String lastName, int age) — Spring Data parses the method name and generates the JPQL query automatically.
  • Pagination and sorting: Built-in Page<T> findAll(Pageable pageable) — no manual offset/limit calculation.
  • Custom queries: @Query annotation for complex JPQL/native SQL queries when derivation is insufficient.
  • Auditing: @EnableJpaAuditing, @CreatedDate, @LastModifiedDate for automatic timestamp management.
  • Projections and DTOs: Interface-based and DTO projections to fetch only required columns.
  • Specifications: Dynamic query composition using the Criteria API via JpaSpecificationExecutor.
Interview tip: Know when NOT to use Spring Data JPA — for complex reporting queries with many JOINs and aggregations, native SQL or JOOQ often produces cleaner, better-performing code. Spring Data JPA excels at standard entity CRUD operations.

What is the role of JpaRepository interface?
  • JpaRepository<T, ID> is the top-level Spring Data JPA repository interface combining all repository capabilities:
  • Inherited capabilities:
    • From CrudRepository: save(), findById(), findAll(), deleteById(), count(), existsById()
    • From PagingAndSortingRepository: findAll(Pageable), findAll(Sort)
    • JPA-specific: saveAll(), flush(), saveAndFlush(), deleteInBatch(), getOne() (deprecated, use getReferenceById())
public interface UserRepository extends JpaRepository<User, Long> {
    // Derived query — Spring generates: SELECT u FROM User u WHERE u.email = ?1
    Optional<User> findByEmail(String email);

    // Derived query with multiple conditions
    List<User> findByActiveAndRoleOrderByCreatedAtDesc(boolean active, String role);

    // Custom JPQL
    @Query("SELECT u FROM User u WHERE u.department.id = :deptId AND u.active = true")
    List<User> findActiveMembersByDepartment(@Param("deptId") Long deptId);
}
Interview tip: For most applications, JpaRepository is the right choice. If you need to limit the surface area of the repository (e.g., only reads, no deletes), use CrudRepository or a custom repository interface — following the Interface Segregation Principle.

How do you define custom query methods in Spring Data JPA?
  • Method name derivation: Spring Data parses the method name to generate the query. Uses keywords: findBy, countBy, existsBy, deleteBy, And, Or, Between, GreaterThan, Like, OrderBy, etc.
  • @Query annotation: For complex queries that can't be derived from method names.
  • Named queries: @NamedQuery on entities or in orm.xml.
  • Native queries: @Query(value = "...", nativeQuery = true) for database-specific SQL.
public interface OrderRepository extends JpaRepository<Order, Long> {

    // Derived — Spring generates JPQL
    List<Order> findByStatusAndCustomerEmailOrderByCreatedAtDesc(
            OrderStatus status, String email);

    // JPQL @Query
    @Query("SELECT o FROM Order o WHERE o.total > :minAmount AND o.status = :status")
    Page<Order> findHighValueOrders(
            @Param("minAmount") BigDecimal minAmount,
            @Param("status") OrderStatus status,
            Pageable pageable);

    // Native query
    @Query(value = "SELECT * FROM orders WHERE MONTH(created_at) = :month",
           nativeQuery = true)
    List<Order> findByMonth(@Param("month") int month);

    // Modifying query
    @Modifying
    @Transactional
    @Query("UPDATE Order o SET o.status = :status WHERE o.id IN :ids")
    int bulkUpdateStatus(@Param("ids") List<Long> ids, @Param("status") OrderStatus status);
}
Interview tip: Method name derivation is convenient for simple queries but becomes unreadable for complex ones. Use @Query for anything with 3+ conditions. Always use @Modifying + @Transactional for UPDATE/DELETE queries.

How do you use @Query annotation in Spring Data JPA?
public interface ProductRepository extends JpaRepository<Product, Long> {

    // JPQL with named parameters
    @Query("SELECT p FROM Product p WHERE p.category.name = :category AND p.price <= :maxPrice")
    List<Product> findByCategoryAndMaxPrice(
            @Param("category") String category,
            @Param("maxPrice") BigDecimal maxPrice);

    // Pagination with @Query
    @Query("SELECT p FROM Product p WHERE p.stock > 0 ORDER BY p.rating DESC")
    Page<Product> findInStockByRating(Pageable pageable);

    // DTO projection (constructor expression)
    @Query("SELECT new com.app.dto.ProductSummary(p.id, p.name, p.price) FROM Product p WHERE p.active = true")
    List<ProductSummary> findActiveSummaries();

    // Modifying query (UPDATE/DELETE must use @Modifying)
    @Modifying
    @Transactional
    @Query("UPDATE Product p SET p.stock = p.stock - :qty WHERE p.id = :id AND p.stock >= :qty")
    int decrementStock(@Param("id") Long id, @Param("qty") int qty);
}
  • Position parameters: @Query("... WHERE p.name = ?1") — ordinal, less readable.
  • Named parameters (recommended): @Param("name") — explicit and refactoring-safe.
  • @Modifying: Required for UPDATE/DELETE queries. Without it, Spring Data throws an exception. Optionally use clearAutomatically = true to clear the persistence context after modification.
Interview tip: @Modifying queries require a transaction. Either annotate the repository method with @Transactional or ensure the calling service method is transactional. Without a transaction, an InvalidDataAccessApiUsageException is thrown.

Difference between save() and saveAndFlush().
  • save() — persists or merges the entity in the JPA persistence context (first-level cache). The actual SQL INSERT/UPDATE may be deferred until the transaction commits or the persistence context is flushed. Hibernate batches SQL operations for efficiency.
  • saveAndFlush() — persists/merges the entity AND immediately flushes the persistence context to the database. The SQL is executed right away, within the current transaction.
Aspectsave()saveAndFlush()
SQL timingDeferred (on flush/commit)Immediate
PerformanceBetter (batch friendly)Less optimal (forces flush)
Use caseStandard persistenceWhen immediately visible DB state needed (e.g., before a native query in same transaction)
Interview tip: saveAndFlush() is needed when you execute a native query in the same transaction that must see the saved entity's data. Flush sends pending SQL but does NOT commit — the transaction boundary remains unchanged.

What are derived query methods in Spring Data JPA?
  • Derived query methods allow Spring Data JPA to automatically generate a query by parsing the repository method name. No @Query annotation is needed.
  • Naming convention: {Subject}By{Predicate}
    • Subject: find, get, read, query, count, exists, delete
    • Predicate: field names connected by And, Or, with optional keywords
public interface EmployeeRepository extends JpaRepository<Employee, Long> {
    // WHERE email = ?
    Optional<Employee> findByEmail(String email);

    // WHERE department = ? AND active = ?
    List<Employee> findByDepartmentAndActive(String dept, boolean active);

    // WHERE salary BETWEEN ? AND ?
    List<Employee> findBySalaryBetween(BigDecimal min, BigDecimal max);

    // WHERE name LIKE '%?%'
    List<Employee> findByNameContainingIgnoreCase(String keyword);

    // WHERE join_date > ?
    List<Employee> findByJoinDateAfterOrderBySalaryDesc(LocalDate date);

    // COUNT WHERE department = ?
    long countByDepartment(String department);

    // EXISTS WHERE email = ?
    boolean existsByEmail(String email);
}
Interview tip: Spring Data validates derived query method names at application startup — a typo like findByEmaail will fail immediately with a PropertyReferenceException, which is far better than a runtime failure.

What is optimistic locking vs pessimistic locking in JPA?
  • Optimistic Locking: Assumes conflicts are rare. Multiple transactions read the same data. Before updating, the version is checked — if it changed since the read, an OptimisticLockException is thrown (the transaction retries or the user is notified). No database locks held during the transaction.
  • Pessimistic Locking: Assumes conflicts are frequent. Acquires a database-level lock (shared or exclusive) when data is read. Other transactions are blocked until the lock is released. Higher contention but guaranteed consistency.
// Optimistic Locking — version field
@Entity
public class Account {
    @Id Long id;
    BigDecimal balance;

    @Version  // Incremented on each update; checked on save
    Long version;
}

// Pessimistic Locking — lock at read time
public interface AccountRepository extends JpaRepository<Account, Long> {

    @Lock(LockModeType.PESSIMISTIC_WRITE)  // SELECT ... FOR UPDATE
    @Query("SELECT a FROM Account a WHERE a.id = :id")
    Optional<Account> findByIdWithLock(@Param("id") Long id);
}
AspectOptimisticPessimistic
DB locksNoYes
ConcurrencyHighLower
Deadlock riskNonePossible
Best forRead-heavy, low-conflict dataWrite-heavy, high-conflict critical data
Interview tip: Optimistic locking with @Version is the recommended approach for most web applications. Pessimistic locking is appropriate for financial transactions (debit/credit) where you cannot afford any lost updates.

What is the role of @Transactional annotation in Spring Boot?
  • @Transactional declares a method (or class) as transactional. Spring wraps the method in a transaction proxy that:
    • Starts a transaction before the method executes
    • Commits the transaction on successful return
    • Rolls back the transaction on an unchecked exception (or checked exceptions specified in rollbackFor)
  • Class-level: Applies to all public methods in the class.
  • Method-level: Overrides class-level settings for that specific method.
  • Key attributes: propagation, isolation, readOnly, timeout, rollbackFor, noRollbackFor.
@Service
@Transactional(readOnly = true)  // Class-level: all methods read-only by default
public class OrderService {

    @Transactional  // Override: this method needs a write transaction
    public Order createOrder(OrderDto dto) {
        Order order = orderMapper.toEntity(dto);
        return orderRepository.save(order);
    }

    public Order getOrder(Long id) {
        return orderRepository.findById(id)
            .orElseThrow(() -> new OrderNotFoundException(id));
    }
}
Interview tip: Always use @Transactional(readOnly = true) for read-only methods — Hibernate skips dirty checking (comparing entity snapshots), which can significantly improve performance in read-heavy applications.

Difference between declarative and programmatic transaction management.
  • Declarative transaction management — uses @Transactional annotation (or XML <tx:advice>). Spring's AOP proxy handles transaction demarcation transparently. The business method contains only business logic.
  • Programmatic transaction management — manually uses TransactionTemplate or PlatformTransactionManager to control transaction boundaries in code. Gives fine-grained control at the cost of mixing transaction logic with business logic.
// Declarative (preferred)
@Service
public class PaymentService {
    @Transactional
    public void processPayment(Long orderId, BigDecimal amount) {
        // Spring handles begin/commit/rollback
        orderRepo.debit(orderId, amount);
        paymentRepo.record(orderId, amount);
    }
}

// Programmatic (use when dynamic control needed)
@Service
public class BatchService {
    @Autowired TransactionTemplate transactionTemplate;

    public void processBatch(List<Item> items) {
        items.forEach(item -> transactionTemplate.execute(status -> {
            try {
                itemRepo.save(item);
                return null;
            } catch (Exception e) {
                status.setRollbackOnly(); // fine-grained control
                return null;
            }
        }));
    }
}
Interview tip: Declarative is preferred for 95% of cases — it's cleaner and follows SRP. Use programmatic when transaction boundaries need to change dynamically at runtime, or when you need partial rollback within a loop.

How do you configure transaction propagation in Spring?
  • Propagation defines how a transaction behaves when a transactional method calls another transactional method.
@Transactional(propagation = Propagation.REQUIRES_NEW)
public void auditLog(String action) {
    auditRepo.save(new AuditEntry(action));
}

@Transactional(propagation = Propagation.REQUIRED) // default
public void createOrder(OrderDto dto) {
    // Runs in current transaction, or creates one if none exists
    Order order = orderRepo.save(dto.toEntity());
    auditLog("ORDER_CREATED"); // If auditLog is REQUIRED, it joins this transaction
                               // If REQUIRES_NEW, it runs in separate transaction
}
Interview tip: The most important propagation levels to know: REQUIRED (default — join or create), REQUIRES_NEW (always create new — used for audit logging that should persist even if main transaction rolls back), and NESTED (savepoint within current transaction).

What is the role of isolation levels in transactions?
  • Isolation levels define how concurrent transactions interact — specifically, which anomalies are permitted to occur in exchange for higher concurrency (lower isolation = more concurrency + more anomalies).

97
Explain READ_COMMITTED, REPEATABLE_READ, and SERIALIZABLE isolation levels.
  • READ_COMMITTED: A transaction only reads data that has been committed. Prevents dirty reads (reading another transaction's uncommitted changes). However, if you read the same row twice in the same transaction, you may get different values if another transaction committed a change between reads (non-repeatable read).
  • REPEATABLE_READ: All reads within a transaction return the same data for rows already read — prevents non-repeatable reads. However, new rows inserted by other transactions can appear in subsequent range queries within the same transaction (phantom reads). This is the default in MySQL InnoDB.
  • SERIALIZABLE: Full isolation — transactions are completely isolated from each other, as if executed serially. Prevents all anomalies. Implemented via range locks or MVCC snapshot isolation. Highest correctness, lowest concurrency.
Interview tip: Most enterprise applications use READ_COMMITTED because it provides a good balance of correctness and performance. Upgrade to REPEATABLE_READ or SERIALIZABLE only for specific critical sections that require stronger guarantees.

What is dirty read, non-repeatable read, and phantom read?
  • Dirty Read: Transaction A reads data written by Transaction B, which hasn't committed yet. If B rolls back, A has read invalid data.
    Example: T1 updates salary to 10000 (not yet committed); T2 reads salary = 10000; T1 rolls back; T2 used wrong value.
  • Non-Repeatable Read: Transaction A reads the same row twice and gets different values because Transaction B committed an update between A's two reads.
    Example: T1 reads balance = 500; T2 updates balance to 300 and commits; T1 reads balance again = 300. Two reads within T1 returned different values.
  • Phantom Read: Transaction A executes the same query twice and gets different sets of rows because Transaction B inserted or deleted rows that match A's query between A's two reads.
    Example: T1 counts orders where status='PENDING' = 5; T2 inserts 2 new pending orders; T1 counts again = 7. The "phantom" rows appeared.
Interview tip: These three anomalies form a hierarchy: dirty read ⊃ non-repeatable read ⊃ phantom read. Each isolation level prevents them from the most severe (dirty) up to the most subtle (phantom). Know a concrete real-world scenario for each.

How do you roll back transactions in Spring Boot?
  • Default rollback behavior: Spring automatically rolls back a transaction when an unchecked exception (RuntimeException or Error) propagates out of a @Transactional method. Checked exceptions do NOT trigger rollback by default.
  • Rollback for checked exceptions: Use @Transactional(rollbackFor = Exception.class) or rollbackFor = {IOException.class, SQLException.class}.
  • Programmatic rollback: In programmatic transactions, call TransactionStatus.setRollbackOnly().
  • Manual rollback via TransactionSynchronizationManager: Less common.
@Transactional(rollbackFor = Exception.class)
public void transferFunds(Long fromId, Long toId, BigDecimal amount) throws InsufficientFundsException {
    Account from = accountRepo.findById(fromId).orElseThrow();
    if (from.getBalance().compareTo(amount) < 0) {
        throw new InsufficientFundsException("Insufficient balance"); // checked — triggers rollback
    }
    from.debit(amount);
    accountRepo.findById(toId).orElseThrow().credit(amount);
}
Interview tip: A very common bug: catching an exception inside a @Transactional method and swallowing it — the transaction commits despite the error because the exception never propagated out. Always either rethrow the exception or explicitly mark for rollback via TransactionAspectSupport.currentTransactionStatus().setRollbackOnly().

What is the difference between rollbackFor and noRollbackFor in @Transactional?
  • rollbackFor — specifies exception types (including checked exceptions) that should trigger a rollback. Without this, Spring only rolls back for RuntimeException and Error.
    @Transactional(rollbackFor = {IOException.class, BusinessException.class})
  • noRollbackFor — specifies exception types that should not trigger a rollback, even though they are RuntimeExceptions. Useful when you want to handle certain exceptions gracefully without undoing the transaction.
    @Transactional(noRollbackFor = OptimisticLockingFailureException.class)
  • Combined example:
@Transactional(
    rollbackFor = {IOException.class, ServiceException.class},
    noRollbackFor = OptimisticLockingFailureException.class
)
public void processOrder(OrderDto dto) throws IOException {
    // IOException triggers rollback (checked exception)
    // OptimisticLockingFailureException doesn't rollback — caller handles retry
}
Interview tip: The default Spring rule is: rollback on RuntimeException and Error; commit on checked exceptions. rollbackFor = Exception.class is a common pattern for financial operations where any failure must undo all changes.
