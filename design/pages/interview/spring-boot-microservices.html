What are microservices and how are they different from monoliths?
  • Microservices architecture decomposes an application into small, independently deployable services — each owning its own data store, business logic, and deployment lifecycle. Services communicate over network APIs (REST, gRPC, events).
  • Monolith: A single deployable unit containing all functionality — presentation, business logic, and data access tightly coupled in one codebase and process.
AspectMonolithMicroservices
DeploymentOne artifact (WAR/JAR)Independent per-service deployment
ScalingEntire app scales togetherScale individual bottleneck services
TechnologySingle tech stackPolyglot — best tool per service
Failure isolationOne bug can bring down everythingFailures contained to one service
Development speedFast initially, slows at scaleComplex from day one, faster for large teams
Operational complexitySimpleHigh — needs CI/CD, service mesh, tracing
Interview tip: Microservices trade deployment simplicity for independent scalability and team autonomy. Don't recommend microservices as the default — start with a well-structured monolith and extract services when team size and scaling requirements justify it.

What are the main advantages of microservices architecture?
  • Independent deployability: Teams deploy features without coordinating across the entire codebase — enabling continuous delivery pipelines per service.
  • Targeted scalability: Scale only the services under load (e.g., scale the order service during peak shopping) — more cost-efficient than scaling the entire monolith.
  • Technology freedom: Each service chooses the best language, framework, and database for its requirements — Python for ML, Go for performance-critical paths, Kafka for event streaming.
  • Fault isolation: A failure in the payment service doesn't crash the catalog service. Combined with circuit breakers, overall system resilience increases.
  • Team autonomy: Small teams own services end-to-end (develop, deploy, operate) — aligned with Conway's Law: system design mirrors team communication structure.
  • Evolutionary architecture: Services can be rewritten in isolation — you can replace a legacy service without touching others.
Interview tip: The business case for microservices is organizational scalability — it enables multiple teams to work independently. If you have a 5-person team, microservices add overhead without benefit. The right size is typically 50+ developers before microservices provide net positive value.

What is the role of Netflix Eureka in Spring Cloud?
  • Eureka is a service registry and discovery server — a central directory where all microservice instances register themselves and discover other services.
  • How it works:
    • Each service instance registers at startup with its name, IP, port, and health check URL
    • Services send heartbeats every 30 seconds to confirm they're alive
    • If a service stops sending heartbeats, Eureka removes it from the registry after a timeout
    • Clients fetch the registry and cache it locally — they can route to instances without contacting Eureka for every call
  • Self-preservation mode: If Eureka loses too many heartbeats (network partition), it enters self-preservation mode and doesn't evict instances — preventing mass deregistration due to network issues.
  • High availability: Run multiple Eureka instances that peer-replicate their registries.
Interview tip: Eureka's client-side caching makes it resilient — even if the Eureka server is temporarily down, clients continue routing to the last-known addresses. This is a key architectural advantage over DNS-only service discovery.

How do you configure a Spring Boot application as a Eureka client?

<dependency>

Interview tip: Always set spring.application.name — it's the logical service name used for discovery. Use consistent naming conventions: lowercase, hyphenated (e.g., order-service, not OrderService or order_service).

What is Spring Cloud Config Server? How is it used?
  • Spring Cloud Config Server provides centralized, externalized configuration management for distributed systems. All microservices fetch their configuration from the Config Server at startup instead of managing their own property files.
  • Typical setup: Config Server reads properties from a Git repository. When services start, they call the Config Server to fetch their configuration.
// Config Server
@SpringBootApplication
@EnableConfigServer
public class ConfigServerApp { ... }
# Config Server application.yml
spring:
  cloud:
    config:
      server:
        git:
          uri: https://github.com/myorg/config-repo
          default-label: main
          search-paths: '{application}'  # subfolder per app
# Client (microservice) bootstrap.yml or application.yml
spring:
  config:
    import: configserver:http://config-server:8888
  application:
    name: order-service
  profiles:
    active: prod
# Fetches: order-service-prod.yml from Git
  • Config refresh: Use @RefreshScope on beans and call /actuator/refresh to apply config changes without restart. For automatic refresh, integrate Spring Cloud Bus (Kafka/RabbitMQ) to broadcast refresh events.
Interview tip: Config Server enables the 12-Factor App "Config" principle — configuration stored in the environment, not the code. A key question: "How do you handle sensitive config?" — Use Spring Cloud Config with Vault backend or AWS Secrets Manager, not plain Git.

How do you secure communication between Config Server and microservices?
  • Basic Auth on Config Server: Add spring-boot-starter-security to Config Server and configure credentials. Clients provide them in the spring.config.import URL: configserver:http://user:pass@config-server:8888.
  • TLS/HTTPS: Configure SSL on the Config Server to encrypt communication in transit.
  • Property encryption: Spring Cloud Config supports symmetric and asymmetric encryption. Encrypt sensitive values with /encrypt endpoint — stored as {cipher}ABC123 in Git, decrypted on the fly when served to clients.
  • Spring Security on Config Server:
# Config Server application.yml
spring:
  security:
    user:
      name: configuser
      password: ${CONFIG_PASSWORD}  # from env variable
# Encrypt endpoint: POST /encrypt → returns cipher text
# Decrypt endpoint: POST /decrypt
encrypt:
  key: ${ENCRYPT_KEY}  # symmetric key from env
  • Vault backend (recommended for production): Use HashiCorp Vault or AWS Secrets Manager as the config backend instead of plain Git — secrets never in version control.
Interview tip: The gold standard for secrets management is HashiCorp Vault or AWS Secrets Manager as the Spring Cloud Config backend. Plain Git with encrypted properties is acceptable for less sensitive config but never for secrets like database passwords or API keys.

How do you implement request routing with Spring Cloud Gateway?
# application.yml
spring:
  cloud:
    gateway:
      routes:
        - id: order-service
          uri: lb://order-service  # lb:// = load-balanced Eureka lookup
          predicates:
            - Path=/api/orders/**
          filters:
            - StripPrefix=1  # removes /api prefix before forwarding
            - AddRequestHeader=X-Gateway-Source, spring-cloud-gateway
            - CircuitBreaker=name=orderCircuitBreaker,fallbackUri=/fallback/orders

        - id: user-service
          uri: lb://user-service
          predicates:
            - Path=/api/users/**
            - Method=GET,POST
          filters:
            - RewritePath=/api/users/(?<segment>.*), /users/${segment}

      default-filters:
        - DedupeResponseHeader=Access-Control-Allow-Credentials Access-Control-Allow-Origin

// Programmatic route definition
@Configuration
public class GatewayConfig {
    @Bean
    public RouteLocator customRoutes(RouteLocatorBuilder builder) {
        return builder.routes()
            .route("auth-route", r -> r
                .path("/api/auth/**")
                .filters(f -> f.addRequestHeader("X-Auth-Source", "gateway"))
                .uri("lb://auth-service"))
            .build();
    }
}
Interview tip: The API Gateway is the right place for cross-cutting concerns: JWT validation (via Spring Security), rate limiting (via RequestRateLimiter filter + Redis), and centralized logging of all external traffic. Services behind the gateway trust the gateway for auth.

What is Feign Client and why is it used in microservices?
  • Feign Client (OpenFeign in Spring Cloud) is a declarative HTTP client that generates implementations from annotated interfaces — eliminating boilerplate HTTP code for inter-service communication.
  • Instead of manually creating RestTemplate or WebClient calls, you define an interface that looks like a Spring MVC controller, and Feign generates the implementation at runtime.
// Enable Feign
@SpringBootApplication
@EnableFeignClients
public class OrderServiceApp { ... }

// Define Feign client interface
@FeignClient(name = "inventory-service",  // matches Eureka service name
             fallback = InventoryClientFallback.class)
public interface InventoryClient {

    @GetMapping("/api/inventory/{productId}")
    InventoryDto checkStock(@PathVariable Long productId);

    @PutMapping("/api/inventory/{productId}/reserve")
    void reserveStock(@PathVariable Long productId, @RequestParam int quantity);
}

// Fallback implementation (for circuit breaker)
@Component
public class InventoryClientFallback implements InventoryClient {
    @Override
    public InventoryDto checkStock(Long productId) {
        return InventoryDto.builder().available(false).message("Inventory service unavailable").build();
    }
}
Interview tip: Feign integrates seamlessly with Eureka (service discovery), Resilience4j (circuit breaker), Spring Security (OAuth2 token propagation), and Micrometer (metrics). It's the idiomatic inter-service communication tool in Spring Cloud ecosystems.

What is load balancing in microservices? How does Ribbon work?
  • Load balancing distributes incoming requests across multiple instances of a service to maximize availability and throughput while preventing any single instance from being overwhelmed.
  • Client-side vs server-side:
    • Server-side (traditional): Load balancer (Nginx, AWS ALB) distributes requests — the client sends to a single endpoint.
    • Client-side (Spring Cloud): The client fetches the list of instances from the service registry and chooses which instance to call — distributed load balancing, no single point of failure.
  • Netflix Ribbon: Was the client-side load balancer in Spring Cloud Netflix. Fetched instances from Eureka and used Round Robin (default) or custom algorithms. Now deprecated and in maintenance mode.
  • Spring Cloud LoadBalancer: The modern replacement for Ribbon. Built into Spring Cloud, uses @LoadBalanced RestTemplate/WebClient and Feign. Default algorithm: Round Robin.
Interview tip: Spring Cloud LoadBalancer replaced Ribbon as the default client-side load balancer in Spring Cloud 2020. It integrates with Micrometer for metrics and supports custom load balancing algorithms via ServiceInstanceListSupplier.

What is Resilience4j and how does it improve fault tolerance?
  • Resilience4j is a lightweight fault tolerance library for Java, designed specifically for functional programming with Java 8+. It is the recommended replacement for Netflix Hystrix in Spring Cloud.
  • Core modules:
    • Circuit Breaker: Stops cascading failures by opening the circuit when error rate exceeds a threshold
    • Retry: Automatically retries failed operations with configurable delay and backoff
    • Rate Limiter: Limits the number of calls per time period
    • Bulkhead: Limits concurrent calls to a resource (thread pool or semaphore isolation)
    • Time Limiter: Cancels calls that exceed a time limit
    • Cache: Caches successful responses
  • Integration: spring-cloud-starter-circuitbreaker-resilience4j + spring-boot-starter-aop
Interview tip: Resilience4j is built on vavr (functional Java) and has zero external dependencies beyond SLF4J. It integrates with Spring Boot Actuator to expose circuit breaker state via /actuator/circuitbreakers.
154
Difference between Circuit Breaker, Bulkhead, Retry, and Rate Limiter patterns.
PatternProblem solvedHow it works
Circuit BreakerCascading failures — prevent calling a failing serviceMonitors error rate; opens circuit (rejects calls) when threshold exceeded; half-opens to test recovery
BulkheadResource exhaustion — one slow service consuming all threadsLimits concurrent calls to a service; excess calls rejected or queued. Isolates thread pool per dependency.
RetryTransient failures — temporary network blips, brief service unavailabilityRetries failed calls with configurable count, delay, and exponential backoff; ignores non-retryable exceptions
Rate LimiterOverloading a dependency — calling too fastAllows at most N calls per time window; excess calls wait or are rejected
Time LimiterSlow calls blocking threads indefinitelyCancels calls that take longer than the configured timeout
Interview tip: In production, combine multiple patterns: Circuit Breaker + Retry + Time Limiter is a common combo. Apply them in the correct order: Retry (innermost) → Circuit Breaker → Rate Limiter → Time Limiter (outermost) to get the right behavior.

How do you implement a circuit breaker in Spring Boot with Resilience4j?
# application.yml
resilience4j:
  circuitbreaker:
    instances:
      inventoryService:
        failure-rate-threshold: 50        # Open after 50% failure rate
        wait-duration-in-open-state: 30s  # Stay open for 30s
        sliding-window-size: 10           # Monitor last 10 calls
        permitted-number-of-calls-in-half-open-state: 3
        slow-call-rate-threshold: 50      # Consider slow calls as failures too
        slow-call-duration-threshold: 2s
@Service
public class OrderService {

    private final InventoryClient inventoryClient;

    @CircuitBreaker(name = "inventoryService", fallbackMethod = "inventoryFallback")
    @Retry(name = "inventoryService")
    @TimeLimiter(name = "inventoryService")
    public CompletableFuture<InventoryDto> checkInventory(Long productId) {
        return CompletableFuture.supplyAsync(() -> inventoryClient.checkStock(productId));
    }

    // Fallback method — same return type, extra Throwable parameter
    public CompletableFuture<InventoryDto> inventoryFallback(Long productId, Throwable ex) {
        log.warn("Inventory service unavailable for product {}: {}", productId, ex.getMessage());
        return CompletableFuture.completedFuture(
            InventoryDto.builder().available(false).message("Service unavailable").build()
        );
    }
}
Interview tip: The circuit breaker fallback should return a degraded-but-useful response, not just an error. For example, show "availability unknown" instead of crashing — the user can still browse, even if real-time stock is unavailable.

What is distributed tracing? How do you implement it in Spring Boot?
  • Distributed tracing tracks a single request as it flows through multiple microservices, providing end-to-end visibility: which services handled the request, how long each step took, and where failures occurred.
  • Key concepts:
    • Trace: Represents an end-to-end request (root span + all child spans)
    • Span: Represents a single operation within a trace (one service call)
    • Trace ID: Unique ID propagated through all services in the request chain
    • Span ID: Unique ID for each individual operation
  • Spring Boot 3.x implementation (Micrometer Tracing):
<dependency>
    <groupId>io.micrometer</groupId>
    <artifactId>micrometer-tracing-bridge-brave</artifactId>
</dependency>
<dependency>
    <groupId>io.zipkin.reporter2</groupId>
    <artifactId>zipkin-reporter-brave</artifactId>
</dependency>
management:
  tracing:
    sampling:
      probability: 1.0  # 100% in dev; use 0.1 (10%) in production
  • Trace IDs are automatically propagated via HTTP headers (X-B3-TraceId, traceparent for W3C) by Spring Boot's instrumentation — including RestTemplate, WebClient, and Feign calls.
Interview tip: Spring Boot 3.x uses Micrometer Tracing (replacing Spring Cloud Sleuth). It supports both Brave (Zipkin) and OpenTelemetry (OTLP). The trace ID appears in logs automatically — combine with ELK stack to search all logs for a single trace across all services.

What are Sleuth and Zipkin? How do they work with Spring Boot?
  • Spring Cloud Sleuth (now deprecated, replaced by Micrometer Tracing in Spring Boot 3): Automatically instrumented Spring Boot components (RestTemplate, Feign, Kafka, etc.) to propagate trace and span IDs. Added trace context to logs automatically.
  • Zipkin: A distributed tracing system (open source, originally from Twitter/OpenZipkin). Services send trace data to Zipkin; Zipkin provides a UI to visualize request flows, service dependencies, and latency breakdowns.
  • Modern Spring Boot 3.x flow:
    1. Micrometer Tracing instruments outbound calls (auto-added trace/span headers)
    2. Services report spans to Zipkin via HTTP or Kafka
    3. Zipkin correlates spans by trace ID and shows the full request waterfall
# Spring Boot 3.x Zipkin configuration
management:
  zipkin:
    tracing:
      endpoint: http://zipkin:9411/api/v2/spans
  tracing:
    sampling:
      probability: 0.1  # Sample 10% of requests in production
Interview tip: In production, 100% sampling is too expensive. 1-10% is typical. Use AlwaysSampler for errors or specific user IDs to get full traces for debugging without the overhead of sampling everything.

How do you handle centralized logging in microservices?
  • Problem: Logs are distributed across dozens of service instances and containers — impossible to debug without centralization.
  • ELK Stack (most common):
    • Elasticsearch: Stores and indexes log data
    • Logstash / Fluentd / Filebeat: Collects and ships logs from services
    • Kibana: Visualizes and searches logs
  • Spring Boot configuration for ELK:
    • Use structured JSON logging (Logback with logstash-logback-encoder)
    • Include trace ID, span ID, service name, and environment in every log entry
    • Ship logs via Filebeat (sidecar) or direct Logstash TCP appender
  • Modern alternatives: Grafana Loki (lighter than ELK), AWS CloudWatch Logs, Datadog
<!-- logback-spring.xml -->
<appender name="JSON_CONSOLE" class="ch.qos.logback.core.ConsoleAppender">
    <encoder class="net.logstash.logback.encoder.LogstashEncoder">
        <customFields>{"service":"order-service","env":"prod"}</customFields>
    </encoder>
</appender>
Interview tip: Always include the trace ID in logs — it's the key that ties all logs for a single user request across all services. With distributed tracing + centralized logging, debugging a cross-service issue becomes: find trace ID → search ELK → see all spans.

What are the best practices for designing Spring Boot microservices?
  • Single Responsibility: Each service owns one bounded context (Domain-Driven Design). Decompose by business capability, not technical layers.
  • Database per service: No shared databases. Services communicate via APIs or events, never direct DB queries to another service's schema.
  • API-first design: Define contracts (OpenAPI/Swagger) before implementation. Use consumer-driven contract testing (Pact).
  • Twelve-Factor App principles: Config in environment, stateless processes, disposability, etc.
  • Asynchronous communication: Use events (Kafka, SQS) for non-critical cross-service operations — reduces coupling and improves resilience.
  • Resilience patterns: Circuit breakers, retries, bulkheads, and timeouts on all external calls.
  • Health checks: Implement /actuator/health with liveness and readiness probes for Kubernetes.
  • Observability: Centralized logging + distributed tracing + metrics (Prometheus/Grafana) from day one.
  • Versioned APIs: Version all APIs from the start — changing APIs in microservices is painful.
  • Security: Authenticate at the gateway, authorize at each service. Zero-trust network model.
Interview tip: When asked about microservice best practices, structure your answer around the three pillars: Design (bounded context, single DB), Reliability (resilience patterns, health checks), and Observability (logging, tracing, metrics). This shows architectural maturity.
