What does the @SpringBootApplication annotation do internally?
  • @SpringBootApplication is a meta-annotation that combines three core annotations into one:
    • @SpringBootConfiguration — marks the class as a Spring configuration class (specialized @Configuration), making it eligible for bean definitions.
    • @EnableAutoConfiguration — triggers Spring Boot's auto-configuration mechanism, scanning spring.factories / AutoConfiguration.imports and conditionally applying auto-configurations.
    • @ComponentScan — scans the package of the annotated class and all sub-packages for Spring-managed components (@Component, @Service, @Repository, @Controller, etc.).
  • The class annotated with @SpringBootApplication is the entry point, and its main() method calls SpringApplication.run() to bootstrap the entire application context.
  • Customization: You can override defaults — e.g., @SpringBootApplication(scanBasePackages = "com.myapp", exclude = {DataSourceAutoConfiguration.class}).
Interview tip: Place the main class in the root package of your application. This ensures @ComponentScan covers all sub-packages without explicit configuration.

Difference between @Component, @Service, @Repository, and @Controller.
  • All four are stereotype annotations — specializations of @Component. They all register the class as a Spring bean, but carry semantic meaning and enable layer-specific behavior.

Explain the use of @Configuration and @Bean annotations.
  • @Configuration marks a class as a source of bean definitions. Spring processes it with CGLIB proxying to ensure that @Bean methods always return the same singleton bean instance (even when called multiple times).
  • @Bean annotates a method inside a @Configuration class. The return value is registered as a Spring bean with the method name as the default bean name.
  • Use case: Defining third-party library classes as beans (since you can't annotate them with @Component), or complex bean construction requiring programmatic logic.
@Configuration
public class AppConfig {

    @Bean
    public ObjectMapper objectMapper() {
        return new ObjectMapper()
            .configure(DeserializationFeature.FAIL_ON_UNKNOWN_PROPERTIES, false);
    }

    @Bean
    public RestTemplate restTemplate() {
        return new RestTemplate();
    }
}
Interview tip: @Configuration classes are CGLIB-proxied — @Bean methods called programmatically return the cached singleton. @Component-based bean configs (using @Bean inside @Component) do NOT use CGLIB proxying — each call creates a new instance. This is a common interview trap.

What is the difference between @ComponentScan and @EnableAutoConfiguration?
  • @ComponentScan — scans your application's packages for classes annotated with @Component, @Service, @Repository, @Controller, @Configuration, etc. and registers them as beans. You control what is scanned.
  • @EnableAutoConfiguration — enables Spring Boot's third-party / framework-level auto-configuration. It reads predefined auto-configuration classes from spring.factories / AutoConfiguration.imports and conditionally applies them. This is how Tomcat, JPA, Security, etc. get configured without you writing any config.
  • Key difference: @ComponentScan is for your code; @EnableAutoConfiguration is for Spring Boot and library code.
Interview tip: Both are included in @SpringBootApplication. Understanding their distinct roles explains why you can override auto-configured beans — your beans (found by @ComponentScan) take priority over auto-configured ones (from @EnableAutoConfiguration).

How does @ConditionalOnProperty work in Spring Boot?
  • @ConditionalOnProperty conditionally registers a bean or applies a configuration class only when a specified property exists and matches a given value.
  • Parameters:
    • name / value — the property key(s) to check
    • havingValue — the value the property must have (default: any non-false value)
    • matchIfMissing — if true, the condition matches even if the property is absent (default: false)
@Configuration
@ConditionalOnProperty(name = "feature.payment.enabled", havingValue = "true", matchIfMissing = false)
public class PaymentConfig {
    @Bean
    public PaymentService paymentService() {
        return new StripePaymentService();
    }
}
  • Use cases: Feature flags, enabling/disabling integrations (email, SMS, external API), switching between implementations based on environment.
Interview tip: @ConditionalOnProperty is the backbone of feature toggles in Spring Boot. Combined with Spring profiles, it enables powerful environment-specific behavior without code changes.

Explain the purpose of @EnableConfigurationProperties and @ConfigurationProperties.
  • @ConfigurationProperties binds a group of related properties from application.properties/yml to a POJO, enabling type-safe, structured configuration.
  • @EnableConfigurationProperties(MyProperties.class) registers the @ConfigurationProperties-annotated class as a bean (alternatively, annotate it with @Component).
// application.yml
// app:
//   mail:
//     host: smtp.gmail.com
//     port: 587
//     username: user@example.com

@ConfigurationProperties(prefix = "app.mail")
public class MailProperties {
    private String host;
    private int port;
    private String username;
    // getters + setters
}

@Configuration
@EnableConfigurationProperties(MailProperties.class)
public class MailConfig {
    @Autowired
    private MailProperties mailProperties;
}
  • Benefits: IDE autocompletion, type validation, grouping related properties, support for relaxed binding (mail-host, MAIL_HOST, mailHost all map to mailHost field).
Interview tip: @ConfigurationProperties is strongly preferred over multiple @Value annotations for complex configuration — it's type-safe, testable, and self-documenting.

How does Spring Boot handle externalized configuration?
  • Spring Boot supports externalized configuration — values defined outside the packaged JAR, enabling the same code to run across environments without recompilation.
  • Priority order (highest to lowest):
    1. Command-line arguments
    2. JNDI attributes
    3. Java System properties (-D)
    4. OS environment variables
    5. application-{profile}.properties outside JAR
    6. application.properties outside JAR
    7. application-{profile}.properties inside JAR
    8. application.properties inside JAR
    9. @PropertySource declarations
    10. Default values in @SpringBootApplication
  • Spring Cloud Config Server: For microservices, a centralized Config Server externalizes configuration — all services fetch their config at startup from a Git-backed config repository.
  • Secrets: AWS Secrets Manager, HashiCorp Vault, or Kubernetes Secrets inject sensitive values as environment variables at runtime.
Interview tip: "12-Factor App methodology" is the guiding principle — config should be stored in the environment, not in the code. This is exactly what Spring Boot's externalized configuration implements.

Explain @Import and when to use it.
  • @Import allows you to explicitly import one or more @Configuration classes into another configuration, registering their beans into the current application context — without relying on component scanning.
  • Three forms:
    • Import a @Configuration class: @Import(SecurityConfig.class)
    • Import an ImportSelector: Programmatically decide which classes to import based on condition logic — used heavily in auto-configuration.
    • Import an ImportBeanDefinitionRegistrar: Dynamically register bean definitions at startup (e.g., JPA repositories, MyBatis mappers).
  • When to use: When you want to modularize configuration without enabling component scan for entire packages, or when building libraries/auto-configurations that conditionally import feature config classes.
@Configuration
@Import({DatabaseConfig.class, CacheConfig.class})
public class AppConfig { ... }
Interview tip: @Enable* annotations (like @EnableAsync, @EnableScheduling) internally use @Import to pull in the infrastructure configuration — understanding this demystifies how these annotations work.

What does @EnableScheduling do? Provide an example.
  • @EnableScheduling activates Spring's task scheduling infrastructure. Without it, @Scheduled annotations on methods are ignored.
  • It imports SchedulingConfiguration which creates a TaskScheduler bean (backed by a thread pool).
@SpringBootApplication
@EnableScheduling
public class MyApp { ... }

@Component
public class ReportJob {

    @Scheduled(cron = "0 0 8 * * MON-FRI")  // 8 AM weekdays
    public void generateDailyReport() {
        // business logic
    }

    @Scheduled(fixedDelay = 30000)  // 30s after last completion
    public void syncData() {
        // sync logic
    }

    @Scheduled(fixedRate = 60000)   // every 60s regardless of execution time
    public void healthCheck() {
        // check logic
    }
}
  • Configuring thread pool: By default, a single thread is used. For concurrent scheduling, define a TaskScheduler bean with multiple threads.
Interview tip: In a multi-instance deployment (horizontal scaling), scheduled tasks run on every instance — use a distributed lock (Shedlock, Redis lock) to ensure only one instance executes a job at a time.

Difference between @RestController and @Controller in Spring Boot.
  • @Controller is used for MVC web applications that return views (Thymeleaf, JSP). Handler methods return view names (strings), and ViewResolver resolves them to actual HTML templates.
  • @RestController = @Controller + @ResponseBody. Every method's return value is automatically serialized to JSON/XML and written directly to the HTTP response body. No view resolution occurs.
  • Use @Controller when: Building server-side-rendered web apps returning HTML views.
  • Use @RestController when: Building REST APIs that return JSON/XML payloads.
// REST API - returns JSON
@RestController
@RequestMapping("/api/users")
public class UserController {
    @GetMapping("/{id}")
    public UserDto getUser(@PathVariable Long id) {
        return userService.findById(id); // serialized to JSON
    }
}

// MVC - returns a view name
@Controller
public class HomeController {
    @GetMapping("/home")
    public String home(Model model) {
        model.addAttribute("message", "Hello!");
        return "home"; // resolves to home.html
    }
}
Interview tip: In modern microservice architectures, @RestController is the standard. @Controller is only needed in traditional web apps or when mixing API + view rendering in the same application.

What is the use of @EnableAsync in Spring Boot?
  • @EnableAsync activates Spring's asynchronous method execution capability. Without it, @Async annotations are ignored and methods execute synchronously.
  • It configures an Executor (thread pool) for running async methods. Default is a SimpleAsyncTaskExecutor (not recommended for production — creates a new thread per invocation).
  • Production configuration: Define a custom ThreadPoolTaskExecutor bean named taskExecutor.
@SpringBootApplication
@EnableAsync
public class MyApp { ... }

@Configuration
public class AsyncConfig {
    @Bean(name = "taskExecutor")
    public Executor taskExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(5);
        executor.setMaxPoolSize(10);
        executor.setQueueCapacity(500);
        executor.setThreadNamePrefix("Async-");
        executor.initialize();
        return executor;
    }
}

@Service
public class EmailService {
    @Async("taskExecutor")
    public CompletableFuture<Void> sendWelcomeEmail(String email) {
        // runs in the thread pool
        return CompletableFuture.completedFuture(null);
    }
}
Interview tip: @Async only works when called from outside the bean (Spring proxy must intercept the call). Self-invocation within the same class bypasses the proxy and executes synchronously — a very common interview question.

Explain the @Lazy annotation in Spring.
  • By default, Spring initializes all singleton beans eagerly at application startup. @Lazy changes this to lazy initialization — the bean is only created the first time it is requested.
  • Applied to:
    • @Bean method — that specific bean is lazy
    • @Component class — that component bean is lazy
    • @Autowired injection point — even if the target bean is eager, the proxy injected is lazy
    • spring.main.lazy-initialization=true — makes ALL beans lazy globally
  • Benefits: Faster startup time (useful for tests and serverless), reduced memory usage for rarely-used beans.
  • Drawbacks: Errors in lazy beans surface at first use, not at startup — harder to detect early. Not suitable for critical infrastructure beans.
Interview tip: spring.main.lazy-initialization=true can dramatically speed up test startup time. However, in production, pre-warming all beans at startup is preferred to detect misconfiguration early and avoid first-request latency spikes.

Difference between @Primary and @Qualifier annotations.
  • Both resolve ambiguity when multiple beans of the same type exist in the context.
  • @Primary — marks one bean as the default candidate. When Spring autowires a type with multiple candidates, the @Primary bean is chosen unless explicitly qualified.
  • @Qualifier("beanName") — at the injection point, explicitly specifies which bean to inject by name. Overrides @Primary.
@Bean @Primary
public DataSource primaryDataSource() { return new HikariDataSource(primaryConfig); }

@Bean
public DataSource replicaDataSource() { return new HikariDataSource(replicaConfig); }

@Service
public class ReportService {
    // Gets primaryDataSource (via @Primary)
    @Autowired
    private DataSource dataSource;

    // Gets replicaDataSource explicitly
    @Autowired @Qualifier("replicaDataSource")
    private DataSource replicaDataSource;
}
Interview tip: @Primary is for declaring which bean wins by default. @Qualifier is for consuming a specific bean at the injection point. Use @Primary sparingly — explicit @Qualifier is usually clearer in multi-datasource or multi-service scenarios.

What is the use of @Profile in Spring Boot?
  • @Profile restricts bean registration to specific Spring profiles. The annotated bean (or entire configuration class) is only instantiated when the specified profile is active.
  • Use cases:
    • Use H2 in-memory DB for dev/test, PostgreSQL for prod
    • Enable mock email service in dev, real SMTP in prod
    • Load different security configurations per environment
@Configuration
@Profile("dev")
public class DevDataSourceConfig {
    @Bean
    public DataSource dataSource() {
        return new EmbeddedDatabaseBuilder().setType(H2).build();
    }
}

@Configuration
@Profile("prod")
public class ProdDataSourceConfig {
    @Bean
    public DataSource dataSource() {
        return new HikariDataSource(prodProperties());
    }
}

// Negative profile — active when "cloud" is NOT active
@Component
@Profile("!cloud")
public class LocalFileStorage implements FileStorage { ... }
Interview tip: @Profile supports logical operators: @Profile({"dev", "test"}) (OR), @Profile("!prod") (NOT). Spring Boot 2.4+ also supports profile groups for grouping related profiles.

How does @Scope annotation work in Spring Boot?
  • @Scope defines the lifecycle and visibility of a Spring bean.
  • Available scopes:
    • singleton (default) — one instance per Spring container
    • prototype — new instance each time the bean is requested
    • request — one instance per HTTP request (web apps only)
    • session — one instance per HTTP session (web apps only)
    • application — one instance per ServletContext
    • Custom scopes (e.g., thread scope, tenant scope)
@Component
@Scope("prototype")
public class ReportGenerator { ... }

// For web scopes with proxy to inject into singleton beans:
@Component
@Scope(value = WebApplicationContext.SCOPE_REQUEST, proxyMode = ScopedProxyMode.TARGET_CLASS)
public class RequestContext { ... }
Interview tip: Injecting a prototype-scoped bean into a singleton bean is a classic gotcha — the prototype bean won't refresh because the singleton holds a reference after first injection. Solutions: @Lookup, ObjectFactory, or ApplicationContext.getBean().

What is @EnableTransactionManagement?
  • @EnableTransactionManagement activates Spring's annotation-driven transaction management infrastructure, enabling the use of @Transactional annotations on methods and classes.
  • It imports TransactionManagementConfigurationSelector which registers either a JDK proxy or AspectJ-based transaction advisor.
  • Spring Boot does this automatically — when spring-boot-starter-data-jpa or spring-boot-starter-jdbc is on the classpath, @EnableTransactionManagement is applied automatically via auto-configuration. You don't need to add it explicitly in Spring Boot apps.
  • Modes:
    • AdviceMode.PROXY (default) — JDK/CGLIB proxies intercept @Transactional methods
    • AdviceMode.ASPECTJ — compile/load-time weaving, bypasses proxy limitations (supports self-invocation)
Interview tip: The proxy mode means @Transactional on private methods or self-invocation doesn't work — the call bypasses the proxy. This is a very common Spring interview question.

What is the role of @PropertySource annotation?
  • @PropertySource adds a custom properties file to Spring's Environment, making its values available via @Value and the Environment API.
  • Useful for loading application-specific configuration from files that are not the standard application.properties.
@Configuration
@PropertySource("classpath:database.properties")
@PropertySource("classpath:mail.properties")  // multiple allowed
public class AppConfig {

    @Value("${db.url}")
    private String dbUrl;
}

// database.properties
// db.url=jdbc:postgresql://localhost:5432/mydb
// db.username=admin
  • Limitation: @PropertySource does not support YAML files out of the box — use a custom PropertySourceFactory or simply rely on Spring Boot's built-in application.yml support.
  • @PropertySources allows multiple @PropertySource declarations (or use Java 8+ repeatable annotation support).
Interview tip: For most Spring Boot apps, @PropertySource is rarely needed since application.properties and profile-specific files cover most needs. It's useful when integrating legacy property files or modular configuration.

Explain @EnableJpaRepositories and its usage.
  • @EnableJpaRepositories activates Spring Data JPA's repository scanning — it detects interfaces extending JpaRepository, CrudRepository, etc., and creates proxy implementations at runtime.
  • In Spring Boot, this is automatically applied when spring-boot-starter-data-jpa is on the classpath. Explicit declaration is only needed for advanced configurations.
  • Advanced usage scenarios:
    • Multiple datasources: Specify different basePackages, entityManagerFactoryRef, and transactionManagerRef for each datasource.
    • Custom base package: @EnableJpaRepositories(basePackages = "com.myapp.data")
// Multiple datasource configuration
@Configuration
@EnableJpaRepositories(
    basePackages = "com.myapp.order.repository",
    entityManagerFactoryRef = "orderEntityManagerFactory",
    transactionManagerRef = "orderTransactionManager"
)
public class OrderDataSourceConfig { ... }
Interview tip: Multiple datasource configuration is a classic senior-level Spring Boot interview topic. Know how to configure it: separate DataSource, EntityManagerFactory, TransactionManager, and @EnableJpaRepositories for each datasource.
