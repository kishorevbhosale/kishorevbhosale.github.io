What are the different types of testing in Spring Boot?
  • Unit Tests: Test a single class in isolation. All dependencies are mocked. Fast (milliseconds), no Spring context, no database. Use JUnit 5 + Mockito.
  • Integration Tests: Test multiple components working together with a real Spring context and (usually) a real or in-memory database. Slower, but verify component interactions.
  • Slice Tests: Test a specific layer of the application with a partial Spring context:
    • @WebMvcTest — controller + MVC infrastructure only
    • @DataJpaTest — JPA repositories + in-memory database
    • @JsonTest — JSON serialization only
    • @WebFluxTest — reactive web layer
  • System/E2E Tests: Test the entire application running with all dependencies. Often uses Testcontainers for real DB, Kafka, etc. Slowest but most realistic.
  • Contract Tests: Verify API contracts between services (Spring Cloud Contract, Pact). Enables independent deployment with confidence.
  • Performance Tests: Load/stress testing with JMeter, Gatling, k6.
Interview tip: Follow the Testing Pyramid — many fast unit tests at the base, fewer integration tests in the middle, minimal but critical E2E tests at the top. Don't invert the pyramid with too many slow, expensive integration tests.

How do you write a simple JUnit 5 test in Spring Boot?
// Pure unit test — no Spring context needed
class OrderServiceTest {

    private OrderService orderService;
    private OrderRepository orderRepository;
    private PaymentService paymentService;

    @BeforeEach
    void setUp() {
        orderRepository = Mockito.mock(OrderRepository.class);
        paymentService = Mockito.mock(PaymentService.class);
        orderService = new OrderService(orderRepository, paymentService);
    }

    @Test
    @DisplayName("createOrder should persist order and return DTO with generated ID")
    void createOrder_withValidRequest_shouldReturnOrderDto() {
        // Given
        CreateOrderRequest request = new CreateOrderRequest(1L, List.of(new OrderItem(101L, 2)));
        Order savedOrder = Order.builder().id(100L).status(OrderStatus.PENDING).build();
        Mockito.when(orderRepository.save(any(Order.class))).thenReturn(savedOrder);

        // When
        OrderDto result = orderService.createOrder(request);

        // Then
        assertThat(result.getId()).isEqualTo(100L);
        assertThat(result.getStatus()).isEqualTo("PENDING");
        Mockito.verify(orderRepository, times(1)).save(any(Order.class));
    }

    @ParameterizedTest
    @ValueSource(ints = {0, -1, -100})
    void createOrder_withInvalidQuantity_shouldThrowException(int quantity) {
        CreateOrderRequest request = new CreateOrderRequest(1L, List.of(new OrderItem(101L, quantity)));
        assertThatThrownBy(() -> orderService.createOrder(request))
            .isInstanceOf(IllegalArgumentException.class)
            .hasMessageContaining("quantity");
    }
}
Interview tip: Follow the Given/When/Then (GWT) structure — it makes tests self-documenting. Use @DisplayName to describe business behavior, not implementation details. Use @ParameterizedTest to avoid duplicating test methods for multiple inputs.

What is the role of @SpringBootTest annotation?
  • @SpringBootTest loads the full Spring application context for integration testing — equivalent to starting the actual application. All beans are configured, including auto-configuration.
  • Web environment modes:
    • MOCK (default) — loads a mock servlet environment; use with MockMvc
    • RANDOM_PORT — starts embedded server on a random port; use with TestRestTemplate or WebTestClient
    • DEFINED_PORT — starts on the configured port
    • NONE — loads context without any servlet environment
@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
@Transactional  // Rollback after each test
class OrderIntegrationTest {

    @Autowired
    private OrderRepository orderRepository;

    @LocalServerPort
    private int port;

    @Autowired
    private TestRestTemplate restTemplate;

    @Test
    void createOrder_shouldPersistInDatabase() {
        CreateOrderRequest request = new CreateOrderRequest(1L, List.of(new OrderItem(101L, 1)));

        ResponseEntity<OrderDto> response = restTemplate.postForEntity(
            "http://localhost:" + port + "/api/orders", request, OrderDto.class);

        assertThat(response.getStatusCode()).isEqualTo(HttpStatus.CREATED);
        assertThat(orderRepository.findById(response.getBody().getId())).isPresent();
    }
}
Interview tip: @SpringBootTest is the slowest test type — it starts the entire application. Use slice tests (@WebMvcTest, @DataJpaTest) whenever possible to test specific layers quickly. Use @SpringBootTest only for end-to-end scenarios that need the full context.

What is @DataJpaTest used for?
  • @DataJpaTest is a JPA slice test annotation that loads only the JPA-related components: entities, repositories, Hibernate, and an embedded in-memory database (H2 by default).
  • What it includes: JPA repositories, @Entity classes, Spring Data JPA configuration, embedded database (H2). By default, each test is wrapped in a transaction that rolls back after the test.
  • What it excludes: Services, controllers, security, full auto-configuration — making it significantly faster than @SpringBootTest.
@DataJpaTest
class ProductRepositoryTest {

    @Autowired
    private ProductRepository productRepository;

    @Autowired
    private TestEntityManager entityManager;

    @Test
    void findByCategory_shouldReturnMatchingProducts() {
        // Given — persist test data
        entityManager.persistAndFlush(new Product(null, "Laptop", "ELECTRONICS", 999.99, true));
        entityManager.persistAndFlush(new Product(null, "Phone", "ELECTRONICS", 699.99, true));
        entityManager.persistAndFlush(new Product(null, "Desk", "FURNITURE", 299.99, true));

        // When
        List<Product> electronics = productRepository.findByCategory("ELECTRONICS");

        // Then
        assertThat(electronics).hasSize(2);
        assertThat(electronics).extracting(Product::getName).containsExactlyInAnyOrder("Laptop", "Phone");
    }
}
Interview tip: Use TestEntityManager to set up test data in @DataJpaTest — it wraps JPA operations and works within the test transaction. For testing against a real database (closer to production), use Testcontainers with @DataJpaTest(replace = Replace.NONE).

What are Testcontainers? Why are they useful?
  • Testcontainers is a Java library that provides lightweight, throwaway Docker containers for use in tests. It starts real databases, message brokers, and other services as Docker containers during test execution, then stops and removes them after.
  • Why useful:
    • Real database behavior: Test against PostgreSQL, MySQL, MongoDB — not H2 in-memory DB with subtle compatibility differences
    • No mocking needed: Test Kafka producers/consumers, Redis caching, Elasticsearch queries with real implementations
    • Environment consistency: Same container image in local dev, CI, and staging — eliminates "works on my machine" issues
    • Cleanup: Containers are automatically removed after tests — no state leakage between test runs
  • Supported containers: PostgreSQL, MySQL, MongoDB, Redis, Kafka, RabbitMQ, Elasticsearch, LocalStack (AWS), and any Docker image.
Interview tip: Testcontainers is particularly valuable for testing database migration scripts (Flyway/Liquibase) and JPA queries against the real target database. H2 doesn't support PostgreSQL-specific features like jsonb, window functions, or custom types.

What are best practices for writing maintainable Spring Boot tests?
  • Test behavior, not implementation: Tests should verify what the code does (observable outcomes), not how it does it (internal method calls). Implementation-coupled tests break on every refactoring.
  • Follow Testing Pyramid: Many fast unit tests → fewer integration tests → minimal E2E tests. Avoid inverting the pyramid.
  • Use Given/When/Then structure: Clear setup, action, assertion. Self-documenting tests.
  • One assertion per test (logical): Test methods verify one business scenario. Multiple assertions asserting the same thing (e.g., all fields of a DTO) are fine; separate tests for separate scenarios.
  • Use descriptive test names: @DisplayName("createOrder_withOutOfStockProduct_shouldThrowInventoryException")
  • Minimize context reloads: Group tests by slice type. Each unique context configuration causes a reload. Shared test base classes with @SpringBootTest settings reuse contexts.
  • Use test data builders: Create builder utilities (TestOrderFactory.createValidOrder()) for reusable test data. Avoid copying the same object creation code across tests.
  • Avoid @SpringBootTest for unit tests: Use it only when you actually need the full context. Prefer @WebMvcTest, @DataJpaTest, or pure Mockito tests.
  • Test edge cases: Empty collections, null values, boundary conditions, concurrent access, timeout scenarios.
Interview tip: Good tests are the best documentation — they show exactly what the code does and why. If a test fails, it should immediately tell the developer what business behavior broke and why. Tests should be maintained with the same care as production code.
