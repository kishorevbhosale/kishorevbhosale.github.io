1. OOPs Concepts (Core)
1
What are the four pillars of OOP (Encapsulation, Inheritance, Polymorphism, Abstraction)? Explain each with real-world examples.
  • Encapsulation: Bundling data and the methods that operate on it within a single class, restricting direct access via access modifiers. Example: A BankAccount class with a private balance field — clients interact only through deposit() and withdraw().
  • Inheritance: A child class acquires properties and behaviors of a parent class, promoting code reuse. Example: SavingsAccount extends BankAccount inherits common account behavior and adds interest logic.
  • Polymorphism: One interface, multiple implementations — the same method call behaves differently depending on the object. Example: Shape.draw() renders a circle for Circle and a rectangle for Rectangle.
  • Abstraction: Hiding complex implementation details and exposing only the necessary interface. Example: JDBC API abstracts vendor-specific SQL drivers behind a uniform Connection/Statement API.
Interview tip: Use a single real-world domain (e.g., banking) to explain all four pillars — it demonstrates architectural thinking.
2
Differentiate between Abstract Class and Interface. When should you use which?
  • State: Abstract classes can hold instance variables and constructors; interfaces can only declare constants (public static final).
  • Methods: Abstract classes can mix abstract and concrete methods. Interfaces (pre-Java 8) had only abstract methods; since Java 8, they support default and static methods.
  • Inheritance: A class can extend only one abstract class but implement multiple interfaces.
  • When to use abstract class: When related classes share common state or behavior (e.g., Vehicle with a fuelLevel field).
  • When to use interface: To define a capability or contract that unrelated classes can adopt (e.g., Comparable, Serializable).
Rule of thumb: If you're modeling "what a thing is" — use abstract class. If you're modeling "what a thing can do" — use interface.
3
Can an interface have default and static methods (Java 8+ feature)? Provide use cases.
  • Yes. Java 8 introduced both to enable API evolution without breaking existing implementations.
  • Default methods provide a method body in the interface. Implementing classes inherit them but can override if needed. Use case: forEach() was added to Iterable as a default method — millions of existing implementations continued working unchanged.
  • Static methods belong to the interface itself and cannot be overridden. Called via InterfaceName.method(). Use case: Comparator.naturalOrder(), List.of() — factory/utility methods on the interface.
  • Java 9 added private methods in interfaces to share logic between default methods without exposing it publicly.
4
Explain multiple inheritance in Java and how Java resolves conflicts.
  • Java does not support multiple inheritance with classes — this avoids the diamond problem where ambiguity arises from two parent classes having the same method.
  • Java supports multiple inheritance of type via interfaces — a class can implement multiple interfaces.
  • If two interfaces provide conflicting default methods, the implementing class must explicitly override the method to resolve ambiguity.
  • Resolution rules: (1) Class methods always win over interface defaults. (2) The most specific sub-interface wins over a super-interface. (3) If still ambiguous, the compiler forces an explicit override.
5
Differentiate between method overloading and method overriding with examples.
  • Overloading (compile-time polymorphism): Same method name, different parameter list, within the same class. The compiler picks the right method based on argument types. Example: add(int, int) vs add(double, double).
  • Overriding (runtime polymorphism): A subclass provides its own implementation for a method already defined in the parent class. Same signature required. Example: Dog.speak() overrides Animal.speak().
  • Return type alone is not sufficient to distinguish overloaded methods.
  • Overloading is resolved at compile time by the compiler; overriding is resolved at runtime by the JVM using dynamic dispatch.
6
What are the rules of method overriding (return type, access modifiers, exceptions)?
  • Signature: Method name and parameter list must be exactly the same as the parent method.
  • Return type: Must be the same or a covariant (subtype) return type.
  • Access modifier: Cannot be more restrictive — e.g., if the parent method is protected, the child cannot make it private.
  • Exceptions: The overriding method cannot throw broader checked exceptions than the parent method. It may throw narrower or unchecked exceptions.
  • final, static, and private methods cannot be overridden.
  • Constructors cannot be overridden — they are not inherited.
7
What is dynamic method dispatch in Java? Provide an example.
  • Dynamic method dispatch is the mechanism by which a call to an overridden method is resolved at runtime, not compile time.
  • The JVM determines which version of the method to invoke based on the actual object type, not the reference type.
  • Example: Animal a = new Dog(); a.speak(); — even though the reference is Animal, the JVM calls Dog.speak() because the actual object is a Dog.
  • This is the foundation of runtime polymorphism in Java and enables the Open/Closed Principle.
8
Differentiate between compile-time polymorphism and runtime polymorphism.
  • Compile-time (static binding): Achieved via method overloading. The compiler selects the method at compile time based on the parameter signature. Slightly faster due to no runtime lookup.
  • Runtime (dynamic binding): Achieved via method overriding. The JVM resolves the method call at runtime based on the actual object type. More flexible and critical for design patterns like Strategy, Template Method, and Observer.
  • Compile-time polymorphism is resolved by the compiler; runtime polymorphism is resolved by the JVM.
9
Can a constructor be overridden? Why or why not?
  • No. Constructors are not inherited by subclasses, so there is nothing to override.
  • A constructor is tied to the specific class — its sole purpose is to initialize that class's state.
  • However, constructors can be overloaded within the same class (same name, different parameters).
  • A subclass invokes a parent constructor via super() — this is constructor chaining, not overriding.
10
Differentiate between HAS-A and IS-A relationships with examples.
  • IS-A represents inheritance: "Dog IS-A Animal" → class Dog extends Animal. Establishes a parent-child type hierarchy.
  • HAS-A represents composition/aggregation: "Car HAS-A Engine" → the Car class contains an Engine field.
  • IS-A creates tight coupling between parent and child. HAS-A promotes loose coupling and flexibility.
  • Best practice (Effective Java, Item 18): Favor composition (HAS-A) over inheritance (IS-A) — it provides better encapsulation, easier testing, and more flexible design.
11
Explain encapsulation with a practical code snippet.
  • Declare fields as private, expose controlled access through public getter/setter methods.
  • This protects internal state from invalid modifications and allows validation logic inside setters.
public class Employee {
    private String name;
    private double salary;

    public String getName() { return name; }

    public void setSalary(double salary) {
        if (salary < 0)
            throw new IllegalArgumentException("Salary cannot be negative");
        this.salary = salary;
    }
}
  • Benefits: Data hiding, controlled access, ability to change internal representation without affecting clients, and enforcement of class invariants.
12
Define coupling and cohesion in OOPs. Why is low coupling and high cohesion desirable?
  • Coupling: The degree of dependency between modules. Low coupling means modules are independent — changing one doesn't ripple through others.
  • Cohesion: The degree to which elements within a module belong together. High cohesion means a class does one thing well and follows the Single Responsibility Principle.
  • Low coupling + high cohesion = a clean, maintainable, testable codebase where components can be developed, tested, and deployed independently.
  • How to achieve: Program to interfaces (not implementations), use dependency injection, and follow SOLID principles.
13
What is a marker interface? Give examples (e.g., Serializable, Cloneable).
  • A marker interface is an empty interface (no methods or fields) used to signal metadata to the JVM or framework.
  • Examples: Serializable (tells the JVM the object can be serialized), Cloneable (enables clone()), Remote (marks an object for RMI).
  • In modern Java, annotations (@Entity, @Deprecated) often replace marker interfaces for attaching metadata.
  • Advantage of marker interfaces over annotations: They define a type, enabling compile-time type checking — e.g., ObjectOutputStream.writeObject() accepts any Serializable object.
14
Differentiate between composition and aggregation. Which is preferred and why?
  • Both represent HAS-A relationships, but differ in ownership semantics.
  • Composition (strong ownership): The child cannot exist independently. If the parent is destroyed, the child is destroyed too. Example: House and Room — rooms don't exist without the house.
  • Aggregation (weak ownership): The child can exist independently. Example: Department and Employee — employees can exist even if the department is dissolved.
  • Composition is generally preferred when you want strong lifecycle control and encapsulation — the owning class manages the full lifecycle of its dependencies.
15
What are the commonly overridden methods from Object class (equals, hashCode, toString, clone)? How do you override them correctly?
  • equals(): Define logical equality. Must satisfy the contract: reflexive, symmetric, transitive, consistent, and x.equals(null) returns false. Use instanceof check, then compare significant fields.
  • hashCode(): Must be overridden whenever equals() is overridden. Equal objects must produce the same hash code. Use Objects.hash(field1, field2, ...) for a clean implementation.
  • toString(): Return a concise, meaningful string representation for logging and debugging. Include key fields that identify the object.
  • clone(): Create a copy of the object. The class must implement Cloneable. Deep clone nested mutable objects. Prefer copy constructors or static factory methods over clone() — they are safer and more flexible.
Key rule: If you override equals() without overriding hashCode(), HashMap/HashSet will break — equal objects may land in different buckets.
2. Core Java – Fundamentals
16
Differentiate between JDK, JRE, JVM.
  • JVM (Java Virtual Machine): The runtime engine that executes bytecode. It is platform-specific — there is a different JVM for Windows, macOS, and Linux — but the bytecode it runs is platform-independent.
  • JRE (Java Runtime Environment): JVM + core libraries (java.lang, java.util, etc.) + supporting files. It is everything needed to run a compiled Java application.
  • JDK (Java Development Kit): JRE + development tools (javac compiler, jdb debugger, jar tool, javadoc). It is everything needed to develop and run Java applications.
  • Relationship: JDK ⊃ JRE ⊃ JVM. You need the JDK for development; end users only need the JRE.
Interview tip: Mention that since Java 11, Oracle no longer distributes a standalone JRE — the JDK is the single distribution unit.
17
What is a classloader in Java? Explain its hierarchy (Bootstrap, Extension, Application).
  • Classloader: A subsystem of the JVM responsible for loading .class files into memory at runtime. It follows the delegation model — a child classloader delegates to its parent before attempting to load a class itself.
  • Bootstrap ClassLoader: Written in native code. Loads core JDK classes from rt.jar (or the java.base module in Java 9+). It is the root of the hierarchy and has no parent.
  • Extension (Platform) ClassLoader: Loads classes from the jre/lib/ext directory or the platform modules. Parent is Bootstrap.
  • Application (System) ClassLoader: Loads classes from the application classpath (-cp or CLASSPATH). Parent is Extension.
  • Delegation model: When a class is requested, the Application loader delegates up to Extension, then to Bootstrap. Only if none of the parents can find it does the requesting loader attempt to load it. This prevents duplicate loading and ensures core classes cannot be overridden.
Interview tip: Mention that you can write custom classloaders (extending ClassLoader) for hot-reloading, plugin architectures, or application servers like Tomcat.
18
Difference between == and .equals(). When should each be used?
  • == (reference equality): Compares whether two references point to the exact same object in memory. For primitives, it compares values directly.
  • .equals() (logical equality): Compares the content/state of two objects. By default, Object.equals() behaves like ==, but classes like String, Integer, and LocalDate override it to compare values.
  • Use == for primitive comparisons and for checking if a reference is null.
  • Use .equals() for comparing object content — always use it for String, wrapper classes, and custom objects.
String a = new String("hello");
String b = new String("hello");
System.out.println(a == b);      // false — different objects
System.out.println(a.equals(b)); // true  — same content
Gotcha: Integer caches values -128 to 127, so == may return true for small values but false for larger ones. Always use .equals() for wrappers.
19
Difference between String, StringBuilder, StringBuffer. Which one to use in multi-threaded scenarios?
  • String: Immutable. Every modification creates a new object. Thread-safe by nature because it cannot be changed after creation.
  • StringBuilder: Mutable, not thread-safe. Fastest option for single-threaded string manipulation (concatenation in loops, building dynamic queries, etc.).
  • StringBuffer: Mutable, thread-safe — all methods are synchronized. Use in multi-threaded scenarios where multiple threads modify the same buffer.
  • Performance order: StringBuilder > StringBuffer > String (for repeated modifications).
  • Recommendation: Use StringBuilder in 99% of cases. Use StringBuffer only when the same mutable buffer is genuinely shared across threads — in practice, this is rare; prefer thread-local builders or immutable Strings.
20
Why are Strings immutable in Java? Explain design reasons.
  • String pool optimization: The JVM maintains a String pool (interning). Multiple references can safely share the same String object because it cannot be modified.
  • Thread safety: Immutable objects are inherently thread-safe — no synchronization needed when sharing Strings across threads.
  • Security: Strings are used for class loading, network connections, file paths, and database URLs. If Strings were mutable, a malicious caller could change a validated String after the security check.
  • Hash code caching: Since a String never changes, its hashCode() is computed once and cached. This makes Strings extremely efficient as HashMap keys.
  • Design principle: Immutability simplifies reasoning about code, eliminates defensive copying, and prevents a whole class of bugs.
21
What is String interning? How does it improve performance?
  • String interning stores at most one copy of each distinct String value in a pool maintained by the JVM (the String pool).
  • String literals are automatically interned at compile time. Strings created with new String() are not interned unless you explicitly call .intern().
  • Performance benefit: Reduces memory footprint when the same string value is used repeatedly. Also enables fast == comparisons on interned strings instead of character-by-character .equals().
  • Since Java 7, the String pool moved from PermGen to the main heap, making it eligible for garbage collection and reducing OOM risks.
String a = "hello";               // interned in pool
String b = new String("hello");    // separate heap object
String c = b.intern();             // returns pooled reference
System.out.println(a == c);        // true
Caution: Avoid excessive manual interning of dynamic strings — it can fill the pool and cause GC pressure.
22
Differentiate between final, finally, finalize.
  • final: A keyword that restricts modification. A final variable cannot be reassigned, a final method cannot be overridden, and a final class cannot be subclassed.
  • finally: A block in exception handling that always executes after try/catch, whether or not an exception occurs. Used for cleanup logic (closing resources, releasing locks).
  • finalize(): A method on Object called by the GC before reclaiming an object. It is deprecated since Java 9 — unreliable, non-deterministic, and causes GC delays. Use try-with-resources or Cleaner instead.
Interview tip: Mention that finally will not execute if System.exit() is called or the JVM crashes.
23
Can we override private, static, or final methods? Why or why not?
  • private methods — No. Private methods are not visible to subclasses, so there is nothing to override. A subclass can define a method with the same name, but it is a completely separate method — not an override.
  • static methods — No. Static methods belong to the class, not the instance. They are resolved at compile time (static binding). A subclass can define a static method with the same signature — this is called method hiding, not overriding. The method called depends on the reference type, not the object type.
  • final methods — No. The final keyword explicitly prevents overriding. This is used to enforce invariant behavior — e.g., Object.getClass() is final because its semantics must not change.
Key distinction: Overriding uses dynamic dispatch (runtime). Hiding uses static binding (compile time). Know the difference — interviewers test this with tricky code snippets.
24
Differentiate between checked and unchecked exceptions. Provide examples.
  • Checked exceptions: Subclasses of Exception (excluding RuntimeException). The compiler forces you to handle them with try-catch or declare them with throws. They represent recoverable conditions. Examples: IOException, SQLException, FileNotFoundException.
  • Unchecked exceptions: Subclasses of RuntimeException. Not checked at compile time. They represent programming errors that should be fixed in code. Examples: NullPointerException, ArrayIndexOutOfBoundsException, IllegalArgumentException.
  • Errors: Subclasses of Error (e.g., OutOfMemoryError, StackOverflowError). These are also unchecked and indicate serious JVM-level problems that applications should not catch.
Best practice: Use checked exceptions for conditions the caller can reasonably recover from. Use unchecked exceptions for bugs or contract violations. Avoid overusing checked exceptions — they pollute method signatures.
25
Difference between throw and throws keywords.
  • throw: Used inside a method body to explicitly throw an exception object. It transfers control to the nearest enclosing catch block or propagates up the call stack.
  • throws: Used in a method signature to declare that the method may throw certain checked exceptions. It informs the caller to handle or propagate them.
  • throw is followed by an exception instance; throws is followed by exception class names.
// throws — declaration in signature
public void readFile(String path) throws IOException {
    if (path == null)
        throw new IllegalArgumentException("Path cannot be null"); // throw — creating & throwing
    // ...
}
26
What is try-with-resources? Why is it preferred over finally for closing resources?
  • try-with-resources (Java 7+) automatically closes any resource that implements AutoCloseable when the try block exits — whether normally or via exception.
  • Why preferred over finally: Eliminates boilerplate null-checks and nested try-catch in the finally block. Handles suppressed exceptions correctly — if both the try block and close() throw, the close exception is attached as suppressed rather than swallowing the original.
  • Multiple resources can be declared in a single try statement — they are closed in reverse order of declaration.
// Clean: resources auto-closed
try (var conn = dataSource.getConnection();
     var stmt = conn.prepareStatement(sql);
     var rs = stmt.executeQuery()) {
    while (rs.next()) { /* process */ }
} // conn, stmt, rs all closed automatically
Interview tip: Java 9 allows effectively-final variables in try-with-resources: try (existingResource) { ... } — no need to redeclare.
27
What is autoboxing and unboxing in Java? Provide examples.
  • Autoboxing: Automatic conversion of a primitive type to its corresponding wrapper class by the compiler. Example: Integer x = 42; — the compiler converts int 42 to Integer.valueOf(42).
  • Unboxing: Automatic conversion of a wrapper class back to its primitive type. Example: int y = x; — the compiler converts to x.intValue().
  • Performance pitfall: Autoboxing in tight loops creates excessive wrapper objects and GC pressure. Prefer primitives for performance-critical code.
  • NullPointerException risk: Unboxing a null wrapper throws NPE — e.g., Integer val = null; int x = val; crashes at runtime.
List<Integer> list = new ArrayList<>();
list.add(10);          // autoboxing: int → Integer
int value = list.get(0); // unboxing: Integer → int
28
Differentiate between primitive types and wrapper classes.
  • Primitives (int, double, boolean, etc.): Stored on the stack (or inlined in objects). No object overhead. Cannot be null. Cannot be used in generics.
  • Wrappers (Integer, Double, Boolean, etc.): Full objects stored on the heap. Support null. Required for collections and generics (List<Integer>).
  • Memory: An int uses 4 bytes; an Integer uses ~16 bytes (object header + value + padding).
  • Caching: Integer.valueOf() caches values -128 to 127. Boolean.valueOf() always returns cached TRUE/FALSE.
  • When to use wrappers: Collections, nullable database columns, generics, and APIs that require objects. Use primitives everywhere else for performance.
3. Java 8+ Features (Core)
29
What are Lambda expressions? Provide syntax and real-world examples.
  • Lambda expressions are anonymous functions that provide a concise way to implement functional interfaces (interfaces with a single abstract method).
  • Syntax: (parameters) -> expression or (parameters) -> { statements; }
  • They enable functional programming in Java — passing behavior as an argument rather than creating anonymous inner classes.
  • Captures: Lambdas can capture effectively-final local variables from the enclosing scope.
// Before Java 8 — anonymous inner class
Comparator<String> comp = new Comparator<String>() {
    public int compare(String a, String b) { return a.length() - b.length(); }
};

// With lambda
Comparator<String> comp = (a, b) -> a.length() - b.length();

// Real-world: filtering, sorting, event handling
List<String> names = List.of("Alice", "Bob", "Charlie");
names.stream()
     .filter(n -> n.length() > 3)
     .sorted(Comparator.naturalOrder())
     .forEach(System.out::println);
Interview tip: The compiler infers parameter types from the target functional interface — you rarely need to declare them explicitly.
30
What are functional interfaces? Provide examples (Runnable, Supplier, Consumer, Function).
  • A functional interface has exactly one abstract method (SAM). It can have any number of default or static methods. The @FunctionalInterface annotation enforces this at compile time.
  • Runnable: void run() — takes nothing, returns nothing. Used for thread tasks.
  • Supplier<T>: T get() — takes nothing, returns a value. Used for lazy evaluation and factory methods.
  • Consumer<T>: void accept(T t) — takes a value, returns nothing. Used for side effects (logging, writing to DB).
  • Function<T, R>: R apply(T t) — takes a value, returns a transformed value. Used for mapping and transformations.
  • Predicate<T>: boolean test(T t) — takes a value, returns boolean. Used for filtering.
Supplier<UUID>     idGen   = UUID::randomUUID;
Consumer<String>   logger  = System.out::println;
Function<String, Integer> len = String::length;
Predicate<String>  notEmpty = s -> !s.isEmpty();
31
Differentiate between Predicate, Function, Consumer, Supplier.
  • Predicate<T>: T → boolean. Tests a condition. Use: filter(), validation, matching.
  • Function<T, R>: T → R. Transforms input to output. Use: map(), data transformation pipelines.
  • Consumer<T>: T → void. Performs a side effect on input. Use: forEach(), logging, writing.
  • Supplier<T>: () → T. Produces a value with no input. Use: Lazy initialization, factory methods, orElseGet().
  • Bi-variants: BiFunction<T,U,R>, BiConsumer<T,U>, BiPredicate<T,U> — accept two arguments.
  • Composition: Predicate supports .and(), .or(), .negate(). Function supports .andThen(), .compose(). This enables building complex behavior from simple building blocks.
32
What are default and static methods in interfaces? Why were they introduced?
  • Default methods: Methods with a body declared using the default keyword. Implementing classes inherit them but can override. Purpose: API evolution — add new methods to existing interfaces without breaking all implementations.
  • Static methods: Utility methods that belong to the interface itself. Called via InterfaceName.method(). Cannot be overridden. Purpose: Provide helper/factory methods related to the interface without a separate utility class.
  • Why introduced: Before Java 8, adding a method to an interface broke every class that implemented it. Default methods solved this backward-compatibility problem, which was essential for adding forEach() to Iterable, stream() to Collection, etc.
public interface PaymentGateway {
    void processPayment(BigDecimal amount);

    default void logTransaction(BigDecimal amount) {
        System.out.println("Processed: " + amount);
    }

    static PaymentGateway defaultGateway() {
        return amount -> System.out.println("Default: " + amount);
    }
}
33
Difference between stream() and parallelStream(). When to use which?
  • stream(): Processes elements sequentially on a single thread. Deterministic ordering. Suitable for most workloads.
  • parallelStream(): Splits the data into chunks and processes them concurrently using the ForkJoinPool.commonPool(). Can improve throughput for CPU-intensive operations on large datasets.
  • Use parallelStream when: (1) The dataset is large (10K+ elements), (2) the operation is CPU-bound and stateless, (3) the source supports efficient splitting (arrays, ArrayList), and (4) you have multiple cores available.
  • Avoid parallelStream when: (1) The dataset is small (overhead exceeds gains), (2) operations have side effects or are order-dependent, (3) the source splits poorly (LinkedList, I/O streams), or (4) the operation involves blocking I/O.
Interview tip: Always measure before parallelizing. The common ForkJoinPool is shared across the JVM — a slow parallel stream can starve other tasks.
34
Differentiate between intermediate and terminal stream operations with examples.
  • Intermediate operations: Return a new stream and are lazy — they are not executed until a terminal operation is invoked. They build up a pipeline. Examples: filter(), map(), flatMap(), sorted(), distinct(), peek(), limit(), skip().
  • Terminal operations: Trigger pipeline execution, consume the stream, and produce a result or side effect. A stream cannot be reused after a terminal operation. Examples: collect(), forEach(), reduce(), count(), findFirst(), anyMatch(), toArray().
  • Lazy evaluation: Intermediate operations are fused and optimized — e.g., filter().map().findFirst() stops as soon as the first matching element is found, without processing the entire collection.
long count = orders.stream()       // source
    .filter(o -> o.isActive())     // intermediate (lazy)
    .map(Order::getTotal)          // intermediate (lazy)
    .filter(t -> t > 100)         // intermediate (lazy)
    .count();                      // terminal — triggers execution
35
Explain the difference between map(), flatMap(), filter(), reduce() in Streams.
  • filter(Predicate): Selects elements matching a condition. One-to-zero-or-one mapping. Example: stream.filter(x -> x > 5) — keeps only elements greater than 5.
  • map(Function): Transforms each element. One-to-one mapping. Example: stream.map(String::toUpperCase) — converts each string.
  • flatMap(Function): Transforms each element into a stream, then flattens all resulting streams into one. One-to-many mapping. Example: orders.stream().flatMap(o -> o.getItems().stream()) — flattens nested lists.
  • reduce(identity, accumulator): Combines all elements into a single result through repeated application of a binary operator. Example: stream.reduce(0, Integer::sum) — sums all elements.
// flatMap vs map
List<List<String>> nested = List.of(List.of("a","b"), List.of("c","d"));
nested.stream().map(l -> l)         // Stream<List<String>>
nested.stream().flatMap(List::stream) // Stream<String>: a, b, c, d
36
How does collect() work? Explain common collectors (toList, toSet, joining, groupingBy, partitioningBy).
  • collect() is a terminal operation that performs a mutable reduction — it accumulates stream elements into a container (List, Set, Map, String, etc.) using a Collector.
  • Collectors.toList(): Collects into an ArrayList. Most common collector.
  • Collectors.toSet(): Collects into a HashSet, removing duplicates.
  • Collectors.joining(delimiter): Concatenates strings with an optional delimiter, prefix, and suffix.
  • Collectors.groupingBy(classifier): Groups elements into a Map<K, List<V>> based on a classification function. Equivalent to SQL GROUP BY.
  • Collectors.partitioningBy(predicate): Splits elements into a Map<Boolean, List<V>> — two groups: true and false.
Map<String, List<Employee>> byDept = employees.stream()
    .collect(Collectors.groupingBy(Employee::getDepartment));

Map<Boolean, List<Integer>> evenOdd = numbers.stream()
    .collect(Collectors.partitioningBy(n -> n % 2 == 0));

String csv = names.stream()
    .collect(Collectors.joining(", ", "[", "]")); // [Alice, Bob, Charlie]
37
What is Optional in Java? How does it help avoid NullPointerException?
  • Optional<T> is a container that may or may not hold a non-null value. It was introduced in Java 8 to explicitly model the absence of a value instead of returning null.
  • How it prevents NPE: Forces the caller to explicitly handle the empty case via ifPresent(), orElse(), orElseGet(), orElseThrow(), or map()/flatMap() chains. You can't accidentally dereference a null without making a conscious choice.
  • Best practices: Use Optional as a return type for methods that might have no result. Do not use it as a field, method parameter, or in collections — it adds unnecessary overhead.
// Without Optional — NPE-prone
String city = user.getAddress().getCity(); // NPE if address is null

// With Optional — safe chaining
String city = Optional.ofNullable(user)
    .map(User::getAddress)
    .map(Address::getCity)
    .orElse("Unknown");
38
Difference between findFirst() and findAny().
  • findFirst(): Returns the first element in encounter order. Deterministic — always returns the same result for the same input. Works identically in sequential and parallel streams, but in parallel streams it may incur a performance cost because threads must coordinate to preserve order.
  • findAny(): Returns any matching element — no ordering guarantee. In sequential streams, it typically returns the first element. In parallel streams, it returns whichever element the fastest thread finds, making it faster than findFirst().
  • When to use: Use findFirst() when order matters (e.g., "get the oldest record"). Use findAny() when any match is acceptable and you want maximum parallel performance.
Both are short-circuiting terminal operations — they stop processing as soon as a result is found.
39
Difference between forEach() and forEachOrdered().
  • forEach(): Performs an action for each element. In a sequential stream, it processes in encounter order. In a parallel stream, the order is non-deterministic — elements may be processed out of order for better performance.
  • forEachOrdered(): Guarantees processing in encounter order even in parallel streams. It forces synchronization, which reduces parallelism but ensures predictable output.
  • When to use: Use forEachOrdered() with parallel streams when output order matters (e.g., writing to a file or building an ordered result). Use forEach() when order doesn't matter (e.g., logging, sending notifications).
List.of("A","B","C","D").parallelStream()
    .forEach(System.out::print);        // may print: CBDA
List.of("A","B","C","D").parallelStream()

What are method references? What are their types?
  • Method references are a shorthand for lambdas that call an existing method. Syntax: ClassName::methodName. They improve readability when the lambda simply delegates to a method.
  • Four types:
  • Static method reference: ClassName::staticMethod → e.g., Integer::parseInt replaces s -> Integer.parseInt(s).
  • Instance method of a particular object: instance::method → e.g., System.out::println replaces x -> System.out.println(x).
  • Instance method of an arbitrary object of a type: ClassName::instanceMethod → e.g., String::toLowerCase replaces s -> s.toLowerCase().
  • Constructor reference: ClassName::new → e.g., ArrayList::new replaces () -> new ArrayList<>().
// Lambda vs method reference
names.stream().map(s -> s.toUpperCase())   // lambda
names.stream().map(String::toUpperCase)     // method reference

Differentiate between LocalDate, LocalTime, LocalDateTime, ZonedDateTime.
  • LocalDate: Date without time or timezone (e.g., 2025-03-15). Use for birthdays, holidays, calendar dates.
  • LocalTime: Time without date or timezone (e.g., 14:30:00). Use for daily schedules, alarm times.
  • LocalDateTime: Date + time without timezone (e.g., 2025-03-15T14:30:00). Use for timestamps in a single-timezone context.
  • ZonedDateTime: Date + time + timezone (e.g., 2025-03-15T14:30:00+05:30[Asia/Kolkata]). Use for scheduling across timezones, flight times, meeting coordination.
  • All are immutable and thread-safe. Part of the java.time package (JSR-310).
Rule of thumb: If users are in different timezones, use ZonedDateTime or store as Instant (UTC) and convert to local time for display.

Compare Date/Calendar API with java.time API. Why is the new API better?
  • Mutability: Date and Calendar are mutable — causes bugs in multi-threaded code. java.time classes are immutable and thread-safe.
  • API design: Calendar months are 0-indexed (January = 0), Date.getYear() returns year - 1900. java.time uses intuitive, human-readable APIs.
  • Timezone handling: Date stores a UTC timestamp but toString() uses the default timezone — confusing. java.time separates LocalDateTime, ZonedDateTime, and Instant explicitly.
  • Formatting: SimpleDateFormat is not thread-safe. DateTimeFormatter is immutable and thread-safe.
  • Rich operations: java.time supports Period, Duration, ChronoUnit, temporal adjusters, and fluent chaining for date arithmetic.
// Old way — error-prone

cal.set(2025, 2, 15); // month is 0-indexed, so 2 = March!

// New way — clear and correct

LocalDate nextWeek = date.plusWeeks(1);

4. Multithreading & Concurrency (Core)
43
Difference between a process and a thread.
  • Process: An independent execution unit with its own memory space (heap, stack, data segment). Processes are isolated — one process cannot directly access another's memory. Inter-process communication (IPC) is expensive (sockets, pipes, shared memory).
  • Thread: A lightweight unit of execution within a process. Threads share the process's heap and code segment but have their own stack and program counter. Context switching between threads is faster than between processes.
  • Creation cost: Creating a process is expensive (fork/exec). Threads are cheaper — the JVM creates them within the same process.
  • Communication: Threads communicate via shared memory (which requires synchronization). Processes require IPC mechanisms.
  • Failure isolation: A crashed process doesn't affect others. A crashed thread can bring down the entire process (e.g., an unhandled exception in one thread can corrupt shared state).
44
Explain the lifecycle of a thread.
  • NEW: Thread object created but start() not yet called.
  • RUNNABLE: After start() is called. The thread is ready to run and is waiting for CPU scheduling. Includes both "ready" and "running" sub-states at the OS level.
  • BLOCKED: Waiting to acquire an intrinsic lock (monitor) held by another thread — e.g., trying to enter a synchronized block.
  • WAITING: Waiting indefinitely for another thread's action — caused by wait(), join(), or LockSupport.park().
  • TIMED_WAITING: Waiting with a timeout — caused by sleep(ms), wait(ms), join(ms), or parkNanos().
  • TERMINATED: The run() method has completed (normally or via exception). The thread cannot be restarted.
Interview tip: You can inspect a thread's state using Thread.getState() or a thread dump (jstack). This is essential for diagnosing deadlocks and performance issues in production.

Difference between Runnable and Callable.
  • Runnable: void run() — takes no arguments, returns nothing, and cannot throw checked exceptions. Used with Thread and ExecutorService.execute().
  • Callable<V>: V call() throws Exception — returns a result and can throw checked exceptions. Used with ExecutorService.submit(), which returns a Future<V>.
  • When to use Callable: When you need the result of an asynchronous computation or need to propagate exceptions to the caller.
  • Both are functional interfaces and can be implemented with lambdas.
// Runnable — fire-and-forget
executor.execute(() -> sendEmail(user));

// Callable — get a result
Future<Integer> future = executor.submit(() -> computeScore(user));
int score = future.get(); // blocks until result is ready

Differentiate between a synchronized method and a synchronized block.
  • Synchronized method: Locks the entire object (this) for instance methods or the Class object for static methods. Simple but coarse-grained — the entire method is a critical section.
  • Synchronized block: Locks on a specific object you choose, and only the code within the block is synchronized. More fine-grained — minimizes the critical section and reduces contention.
  • Performance: Synchronized blocks are preferred because you can lock on a private lock object (avoiding external lock interference) and keep the synchronized section as small as possible.
// Synchronized method — locks 'this' for the whole method
public synchronized void transfer(Account to, int amount) { ... }

// Synchronized block — locks only what's needed
private final Object lock = new Object();
public void transfer(Account to, int amount) {
    // unsynchronized pre-processing
    synchronized (lock) {
        this.balance -= amount;
        to.balance += amount;
    }
    // unsynchronized post-processing
}
Best practice: Always synchronize on a private final lock object — never on this or a publicly accessible object to prevent external code from inadvertently or maliciously holding your lock.

What is the volatile keyword? When should it be used?
  • volatile ensures that reads and writes to a variable go directly to main memory, bypassing CPU caches. It guarantees visibility — a write by one thread is immediately visible to all other threads.
  • It also prevents instruction reordering (provides a happens-before guarantee) around the volatile variable.
  • Limitations: Volatile does not provide atomicity. count++ on a volatile int is still not thread-safe (it is a read-modify-write operation).
  • Use cases: Boolean flags (volatile boolean running = true;), double-checked locking in singleton, publishing immutable objects across threads.
  • When NOT to use: When you need compound atomic operations — use AtomicInteger, synchronized, or Lock instead.
// Classic use: shutdown flag
private volatile boolean running = true;

public void run() {
    while (running) { /* process work */ }
}
public void stop() { running = false; }

Difference between wait(), notify(), and notifyAll().
  • wait(): Releases the monitor lock and puts the current thread into a WAITING state until another thread calls notify() or notifyAll() on the same object. Must be called inside a synchronized block.
  • notify(): Wakes up one arbitrary thread that is waiting on this object's monitor. The awakened thread competes for the lock and does not proceed until it re-acquires it.
  • notifyAll(): Wakes up all threads waiting on this object's monitor. They all compete for the lock; only one proceeds at a time.
  • When to use notifyAll(): Prefer notifyAll() in most cases — notify() risks waking a thread that cannot make progress, leading to missed signals.
  • Spurious wakeups: Always call wait() inside a while loop that re-checks the condition, not an if statement.
synchronized (queue) {
    while (queue.isEmpty()) {
        queue.wait();   // release lock, wait for signal
    }
    item = queue.poll();
}

// Producer side
synchronized (queue) {
    queue.add(item);
    queue.notifyAll(); // wake all waiting consumers
}
49
Difference between sleep() and wait().
  • sleep(ms): A static method of Thread. Pauses the current thread for the specified duration. Does not release any locks. The thread enters TIMED_WAITING and resumes automatically after the timeout.
  • wait(): An instance method of Object. Must be called inside a synchronized block. Releases the monitor lock and enters WAITING state. The thread resumes only when notify()/notifyAll() is called.
  • Purpose: sleep() is for time-based pausing. wait() is for inter-thread communication (coordinating producer-consumer, etc.).
  • Lock behavior: This is the critical difference — sleep() holds locks (potential deadlock risk), wait() releases them (allows other threads to proceed).
50
Difference between ExecutorService and ForkJoinPool.
  • ExecutorService: A general-purpose thread pool for submitting independent, heterogeneous tasks. Tasks are placed in a shared queue and picked up by worker threads. Best for I/O-bound, request-processing, and fire-and-forget workloads.
  • ForkJoinPool: Designed for divide-and-conquer parallelism. Tasks recursively fork into subtasks and join results. Uses work-stealing — idle threads steal tasks from busy threads' queues, maximizing CPU utilization.
  • Task model: ExecutorService uses Runnable/Callable. ForkJoinPool uses RecursiveTask (returns result) and RecursiveAction (no result).
  • parallelStream() and CompletableFuture use the common ForkJoinPool by default.
  • When to use ForkJoinPool: CPU-intensive, recursively decomposable problems (merge sort, matrix multiplication, large array processing).
51
Explain ConcurrentHashMap and CopyOnWriteArrayList.
  • ConcurrentHashMap: A thread-safe Map that allows concurrent reads without locking and uses fine-grained locking (bucket-level or CAS operations) for writes. No segment-level locking since Java 8 — instead uses CAS + synchronized on individual nodes. Reads never block. Does not allow null keys or values.
  • CopyOnWriteArrayList: A thread-safe List that creates a new copy of the underlying array on every write (add, set, remove). Reads are lock-free and fast since they operate on a snapshot. Ideal for read-heavy, write-rare scenarios (e.g., listener lists, configuration caches).
  • ConcurrentHashMap trade-off: High-performance concurrent access; size() and isEmpty() are approximations.
  • CopyOnWriteArrayList trade-off: Writes are expensive (O(n) copy). Only viable when reads vastly outnumber writes.
52
Difference between synchronized collections and concurrent collections.
  • Synchronized collections (Collections.synchronizedMap(), Hashtable, Vector): Wrap every method with a single lock on the entire collection. Only one thread can access the collection at a time — severely limits throughput under contention.
  • Concurrent collections (ConcurrentHashMap, CopyOnWriteArrayList, ConcurrentLinkedQueue): Use fine-grained locking, lock-free algorithms (CAS), or copy-on-write strategies. Multiple threads can read and write concurrently with minimal contention.
  • Iterator behavior: Synchronized collections throw ConcurrentModificationException (fail-fast). Concurrent collections use weakly consistent iterators (fail-safe) — they reflect some or all changes made after iterator creation but never throw CME.
  • Recommendation: Always prefer concurrent collections over synchronized wrappers in new code — they scale far better under load.

What is a deadlock? How can you detect and prevent it?
  • Deadlock: Two or more threads are blocked forever, each waiting for a lock held by the other. Four necessary conditions (Coffman): mutual exclusion, hold and wait, no preemption, and circular wait.
  • Detection: (1) Take a thread dump using jstack <pid> or jcmd <pid> Thread.print. (2) Use ThreadMXBean.findDeadlockedThreads() programmatically. (3) Use monitoring tools like VisualVM or JConsole.
  • Prevention strategies:
  • Lock ordering: Always acquire locks in a consistent, global order — breaks circular wait.
  • Timeout: Use tryLock(timeout) with ReentrantLock instead of unbounded synchronized.
  • Minimize lock scope: Hold locks for the shortest time possible.
  • Avoid nested locks: Reduce the number of locks held simultaneously.
  • Use higher-level concurrency utilities: ConcurrentHashMap, BlockingQueue, CompletableFuture — they manage synchronization internally.
Interview tip: Be ready to write a simple deadlock example and explain how you'd fix it by imposing a lock ordering.
54
Difference between Atomic classes and synchronized blocks.
  • Atomic classes (AtomicInteger, AtomicLong, AtomicReference): Use CAS (Compare-And-Swap) — a hardware-level instruction that atomically compares and updates a value. Lock-free, non-blocking, and highly performant under low-to-moderate contention.
  • synchronized blocks: Use intrinsic locks (monitors). Blocking — a thread waiting for the lock is suspended and context-switched. Better for protecting complex, multi-step critical sections.
  • Performance: Atomics outperform synchronized for simple operations (counters, flags). Under extreme contention, CAS retries can cause spinning — LongAdder (Java 8) addresses this with striped counters.
  • Scope: Atomics protect a single variable. Synchronized can protect multiple variables and compound operations atomically.
// Atomic — lock-free counter
AtomicInteger counter = new AtomicInteger(0);
counter.incrementAndGet(); // CAS under the hood

// Synchronized — multi-variable protection
synchronized (lock) {
    balance -= amount;
    transactionLog.add(entry);
}

Explain CompletableFuture and its advantages over traditional threads.
  • CompletableFuture (Java 8) is an asynchronous computation framework that supports non-blocking, composable, and declarative async programming.
  • Advantages over Thread/Future:
  • Non-blocking composition: Chain stages with thenApply(), thenCompose(), thenCombine() — no future.get() blocking required.
  • Exception handling: exceptionally(), handle() — built-in, fluent error handling.
  • Combining futures: allOf(), anyOf() — wait for all/any to complete.
  • Custom executors: Control which thread pool runs each stage via *Async(executor) variants.
  • No callback hell: Readable, pipeline-style code compared to nested callbacks.
CompletableFuture.supplyAsync(() -> fetchUser(id))        // async step 1
    .thenApply(user -> enrichProfile(user))                // transform
    .thenCompose(profile -> fetchRecommendations(profile)) // async chain
    .thenAccept(recs -> sendResponse(recs))                // consume
    .exceptionally(ex -> { log.error("Failed", ex); return null; });
Interview tip: Know the difference between thenApply() (synchronous transform) and thenCompose() (returns another CompletableFuture — like flatMap).

Differentiate between Semaphore, CountDownLatch, CyclicBarrier.
  • Semaphore: Controls access to a shared resource by maintaining a set of permits. acquire() takes a permit (blocks if none available); release() returns one. Use case: Connection pool limiting, rate limiting (max N concurrent database connections).
  • CountDownLatch: A one-shot synchronization barrier. Initialized with a count; countDown() decrements it. Threads calling await() block until the count reaches zero. Cannot be reused. Use case: Main thread waiting for N worker threads to finish initialization.
  • CyclicBarrier: A reusable synchronization point where N threads wait for each other. When all threads call await(), the barrier trips and optionally runs a barrier action. Can be reused (resets automatically). Use case: Parallel algorithms where threads must synchronize at each iteration (e.g., parallel matrix computation).
  • Key differences: CountDownLatch is one-time and decremented by any thread. CyclicBarrier is reusable and requires a fixed number of waiting threads. Semaphore is not a barrier — it controls concurrent access count.

What is ThreadLocal? Provide use cases.
  • ThreadLocal provides per-thread isolated copies of a variable. Each thread accessing the variable via get()/set() works with its own independent copy — no synchronization needed.
  • How it works: Each Thread object holds a ThreadLocalMap. The ThreadLocal key maps to the thread-specific value. Access is O(1).
  • Use cases:
  • SimpleDateFormat: Not thread-safe — store a per-thread instance to avoid synchronization overhead.
  • User context / request context: Store user ID, tenant ID, or transaction ID per request thread in web servers.
  • Database connections: Store per-thread connection in legacy systems.
  • Performance counters: Accumulate per-thread stats without contention.
private static final ThreadLocal<SimpleDateFormat> dateFormat =
    ThreadLocal.withInitial(() -> new SimpleDateFormat("yyyy-MM-dd"));
Warning: Always call ThreadLocal.remove() in a finally block when using thread pools — otherwise, values from a previous task leak to the next task on the same pooled thread, causing memory leaks and data corruption.

What is false sharing in multithreading? How do you prevent it?
  • False sharing occurs when two threads modify independent variables that happen to reside on the same CPU cache line (typically 64 bytes). Even though there is no logical data sharing, the hardware cache coherence protocol (MESI) forces the cache line to be invalidated and reloaded on every write by either thread.
  • Impact: Destroys cache performance — effectively serializes access and can be 10-100x slower than expected.
  • Prevention strategies:
  • Padding: Add unused fields to push variables to separate cache lines. Java 8 introduced @Contended annotation (sun.misc.Contended / jdk.internal.vm.annotation.Contended) which adds automatic padding.
  • Data structure design: Align frequently written fields so each thread's hot data occupies its own cache line.
  • Use LongAdder: Instead of AtomicLong, use LongAdder which uses striped cells (each on a separate cache line) to eliminate false sharing for counters.
Interview tip: False sharing is a low-level performance concern — mentioning it shows deep systems knowledge. The LMAX Disruptor is a famous example that carefully avoids false sharing for ultra-low latency.
59
What is busy waiting? Why is it a performance issue?
  • Busy waiting (spin-waiting): A thread continuously checks a condition in a tight loop without yielding the CPU — e.g., while (!flag) { /* do nothing */ }.
  • Why it's a problem: (1) Wastes CPU cycles — the spinning thread consumes 100% of a core doing no useful work. (2) Prevents other threads/processes from using that core. (3) Increases power consumption and heat.
  • Alternatives:
  • wait()/notify(): The thread sleeps until signaled — zero CPU usage while waiting.
  • BlockingQueue: take() blocks efficiently using Condition.
  • Lock + Condition: condition.await() parks the thread.
  • LockSupport.park(): Low-level thread parking.
  • When busy waiting IS acceptable: Ultra-low-latency systems (high-frequency trading, LMAX Disruptor) where the wait is expected to be extremely short (nanoseconds) and the cost of context-switching exceeds the cost of spinning.
60
How do you prevent thread starvation?
  • Thread starvation: A thread is perpetually unable to gain access to a shared resource or CPU time because other threads monopolize it. The starving thread is runnable but never gets scheduled.
  • Common causes: (1) Unfair lock acquisition — high-priority or aggressive threads always grab the lock first. (2) Thread priority abuse. (3) Long-running synchronized blocks. (4) Thread pool exhaustion.
  • Prevention strategies:
  • Fair locks: Use new ReentrantLock(true) — FIFO ordering ensures longest-waiting thread gets the lock next.
  • Avoid priority manipulation: Don't rely on Thread.setPriority() — behavior is OS-dependent and unreliable.
  • Minimize lock holding time: Keep critical sections short to give all threads a fair chance.
  • Use concurrent data structures: ConcurrentHashMap, ConcurrentLinkedQueue — minimize explicit locking.
  • Proper thread pool sizing: Ensure the pool has enough threads for the workload. Use separate pools for CPU-bound and I/O-bound tasks to prevent one type from starving the other.
Interview tip: Distinguish between starvation (thread runs but can't get resources), deadlock (threads block each other permanently), and livelock (threads actively respond to each other but make no progress).
5. Collections Framework (Core)
61
Difference between List, Set, Map.
  • List: Ordered collection that allows duplicates. Elements are accessed by index. Implementations: ArrayList, LinkedList.
  • Set: Unordered collection that does not allow duplicates. Uses equals()/hashCode() for uniqueness. Implementations: HashSet, LinkedHashSet, TreeSet.
  • Map: Key-value pair collection. Keys are unique; values can be duplicated. Not part of Collection interface — it's a separate hierarchy. Implementations: HashMap, TreeMap, LinkedHashMap.
  • Key difference: List is about positional access, Set is about uniqueness, Map is about key-based lookup.
Interview tip: Mention that Map does not extend Collection — this is a common gotcha interviewers look for.
62
Difference between HashMap, LinkedHashMap, TreeMap.
  • HashMap: O(1) average for get()/put(). No ordering guarantees. Allows one null key and multiple null values.
  • LinkedHashMap: Same O(1) performance as HashMap but maintains insertion order (or optionally access order) via a doubly-linked list. Slightly higher memory overhead.
  • TreeMap: O(log n) for get()/put(). Keys are sorted by natural ordering or a custom Comparator. Backed by a Red-Black tree. Does not allow null keys.
  • When to choose: HashMap for pure performance, LinkedHashMap when iteration order matters, TreeMap when you need sorted/range queries.
63
Difference between HashSet, LinkedHashSet, TreeSet.
  • HashSet: Backed by a HashMap internally. O(1) add/remove/contains. No ordering. Best general-purpose Set.
  • LinkedHashSet: Backed by a LinkedHashMap. O(1) operations while maintaining insertion order. Useful when you need deduplication but want predictable iteration order.
  • TreeSet: Backed by a TreeMap (Red-Black tree). O(log n) operations. Elements are kept in sorted order. Supports NavigableSet methods like floor(), ceiling(), subSet().
  • None of them allow duplicate elements — that's the fundamental Set contract.
64
Difference between ArrayList and LinkedList.
  • ArrayList: Backed by a dynamic array. O(1) random access by index. O(n) for insertion/deletion in the middle (requires shifting). Cache-friendly due to contiguous memory layout.
  • LinkedList: Doubly-linked list. O(1) insertion/deletion at head/tail. O(n) random access by index (requires traversal). Higher memory overhead per element (each node stores two pointers).
  • In practice: ArrayList is the default choice in 95% of cases — CPU cache locality makes it faster even for many insertion scenarios. LinkedList wins only for heavy head/tail insertion/removal with no random access.
  • LinkedList also implements Deque, so it can be used as a stack or queue.
Interview tip: Say "I almost always default to ArrayList — LinkedList's theoretical O(1) insert advantage is negated by pointer chasing and cache misses in practice."
65
Difference between HashMap and Hashtable.
  • Thread safety: Hashtable is synchronized (every method is synchronized). HashMap is not synchronized.
  • Null keys/values: HashMap allows one null key and multiple null values. Hashtable does not allow any null keys or values — throws NullPointerException.
  • Performance: HashMap is faster because it has no synchronization overhead.
  • Legacy: Hashtable is a legacy class (since JDK 1.0). HashMap was introduced in JDK 1.2 with the Collections Framework.
  • Modern alternative: For thread-safe maps, use ConcurrentHashMap instead of Hashtable — it provides much better concurrency with bucket-level locking.

How does HashMap work internally? (hashCode, equals, buckets, rehashing, load factor).
  • Structure: HashMap is an array of buckets (initially 16). Each bucket holds a linked list (or a balanced tree if chain length ≥ 8, since Java 8).
  • put() flow: (1) Compute hashCode() of the key. (2) Apply internal hash spreading: hash = hashCode ^ (hashCode >>> 16). (3) Determine bucket index: index = hash & (capacity - 1). (4) If the bucket is empty, insert a new node. If not, traverse the chain, using equals() to check for an existing key — replace value if found, else append to the chain.
  • get() flow: Same hashing to find the bucket, then equals() to locate the exact entry in the chain/tree.
  • Load factor (default 0.75): When size > capacity × loadFactor, the map is rehashed — capacity doubles and all entries are redistributed.
  • Treeification (Java 8+): When a bucket chain exceeds 8 nodes and capacity ≥ 64, the chain converts to a Red-Black tree (O(log n) lookup). It un-treeifies when the count drops below 6.
Interview tip: Walk through the full put/get lifecycle step-by-step — this is the single most asked collections question.
67
Differentiate between fail-fast and fail-safe iterators.
  • Fail-fast: Throws ConcurrentModificationException if the collection is structurally modified during iteration (other than through the iterator's own remove()). Uses an internal modCount field. Examples: iterators of ArrayList, HashMap, HashSet.
  • Fail-safe: Works on a snapshot or copy of the collection, so structural modifications don't affect the iteration. No exception is thrown. Examples: iterators of ConcurrentHashMap, CopyOnWriteArrayList.
  • Trade-off: Fail-fast iterators are lightweight (no copy) but fragile. Fail-safe iterators are robust but may not reflect real-time changes and use more memory.
  • Note: The JDK documentation uses the term "weakly consistent" rather than "fail-safe" for concurrent collection iterators.
68
Difference between Comparable and Comparator.
  • Comparable (java.lang): Defines natural ordering via compareTo(T o). The class itself implements it. Only one sort sequence per class. Example: String, Integer implement Comparable.
  • Comparator (java.util): Defines custom ordering via compare(T o1, T o2). External to the class — you can define multiple comparators for different sort strategies without modifying the class.
  • When to use Comparable: When there is a single, obvious natural ordering (e.g., employees by ID).
  • When to use Comparator: When you need multiple orderings (by name, salary, hire date) or when you can't modify the class.
// Comparable — inside the class
public class Employee implements Comparable<Employee> {
    public int compareTo(Employee o) {
        return Integer.compare(this.id, o.id);
    }
}

// Comparator — external, reusable
Comparator<Employee> bySalary = Comparator.comparingDouble(Employee::getSalary);

What is a PriorityQueue and how does it work internally?
  • PriorityQueue is an unbounded queue that orders elements based on natural ordering or a provided Comparator. The head is always the smallest element (min-heap by default).
  • Internal structure: Backed by a binary min-heap implemented as a resizable array. The heap property ensures the smallest element is always at index 0.
  • Time complexity: offer()/poll() are O(log n) due to sift-up/sift-down. peek() is O(1). remove(Object) is O(n) because it requires a linear scan.
  • Not thread-safe. Use PriorityBlockingQueue for concurrent scenarios.
  • Use cases: Task scheduling, Dijkstra's algorithm, top-K problems, event-driven simulations.

Differentiate between Enumeration, Iterator, and ListIterator.
  • Enumeration: Legacy interface (JDK 1.0). Methods: hasMoreElements(), nextElement(). Read-only — no remove(). Used with legacy classes like Vector, Hashtable.
  • Iterator: Introduced in JDK 1.2 as a replacement. Methods: hasNext(), next(), remove(). Supports safe element removal during iteration. Works with all Collection types. Fail-fast.
  • ListIterator: Extends Iterator. Works only with List. Supports bidirectional traversal (hasPrevious(), previous()), add(), set(), and index retrieval.
  • Rule: Always prefer Iterator over Enumeration. Use ListIterator when you need to traverse a list in both directions or modify elements during iteration.

What is an IdentityHashMap? Where is it used?
  • IdentityHashMap uses reference equality (==) instead of object equality (equals()) for key comparison.
  • Two keys are considered equal only if they are the exact same object instance in memory.
  • Internally: Uses a linear-probing hash table (not separate chaining like HashMap). Hash is computed from System.identityHashCode().
  • Use cases: Serialization/deserialization frameworks (tracking already-visited object graphs), deep copy utilities, topology-preserving transformations, and proxy-based operations where logical equality would cause false matches.

How are WeakHashMap and SoftReference used in caching?
  • WeakHashMap: Keys are held via WeakReference. When a key has no strong references elsewhere, the GC reclaims it and the entry is automatically removed. Ideal for canonical maps and metadata caches where entries should vanish when the key object is no longer in use.
  • SoftReference: Objects are collected only when the JVM is low on memory. They survive GC as long as heap space is available. Ideal for memory-sensitive caches (e.g., image caches, database query result caches).
  • Key difference: Weak references are eagerly collected at the next GC cycle. Soft references are collected lazily, only under memory pressure.
  • In practice: Libraries like Guava's CacheBuilder use soft/weak references internally: CacheBuilder.newBuilder().softValues().build().

What is a ConcurrentSkipListMap and when would you use it?
  • ConcurrentSkipListMap is a thread-safe, sorted map based on a skip list data structure. It implements ConcurrentNavigableMap.
  • Performance: O(log n) for get(), put(), remove(). Uses lock-free CAS operations — no global locking.
  • Ordering: Keys are sorted by natural order or a custom Comparator, supporting range queries like subMap(), headMap(), tailMap().
  • When to use: When you need a concurrent + sorted map. It's the concurrent equivalent of TreeMap. Typical use cases: real-time leaderboards, time-series data stores, and ordered event logs in concurrent applications.
  • Trade-off: Higher memory overhead than ConcurrentHashMap due to multi-level index pointers.
74
How does LinkedHashMap maintain insertion order?
  • LinkedHashMap extends HashMap and adds a doubly-linked list threading through all entries.
  • Each entry node has before and after pointers in addition to the standard next pointer used for bucket chaining.
  • On put(), a new entry is appended to the tail of the linked list, preserving insertion order.
  • Access-order mode: Constructed with new LinkedHashMap<>(capacity, loadFactor, true). In this mode, every get() or put() moves the accessed entry to the tail — this is the foundation for building an LRU cache.
  • removeEldestEntry(): Override this method to auto-evict the oldest entry when the map exceeds a size limit — a clean, built-in LRU pattern.

What happens if you mutate a key after inserting it into a HashMap?
  • If you mutate fields that affect hashCode(), the entry becomes unreachable. The key's new hash code points to a different bucket, but the entry is still stored in the old bucket.
  • get() with the mutated key will return null — the map cannot find it because it looks in the wrong bucket.
  • containsKey() returns false. The entry still exists in memory — it becomes a phantom entry, causing a memory leak.
  • remove() also fails to locate the entry for the same reason.
List<String> key = new ArrayList<>(List.of("a"));
Map<List<String>, String> map = new HashMap<>();
map.put(key, "value");

key.add("b"); // mutate key — hashCode changes

map.get(key);          // null — lost!
map.containsKey(key);  // false
Interview tip: This is why mutable objects should never be used as HashMap keys.
76
Why must HashMap keys be immutable?
  • Stable hashCode: Immutable keys guarantee that hashCode() never changes after insertion, so the entry always remains in the correct bucket.
  • Consistent equals: Immutability ensures equals() comparisons remain stable — lookups reliably find the correct entry.
  • If a key mutates: The entry becomes orphaned in the old bucket, causing lookup failures and silent memory leaks.
  • Best practice: Use String, Integer, or other immutable types as keys. If you must use a custom class, make it immutable: final class, private final fields, no setters, defensive copies of mutable fields.
77
Why is load factor important in HashMap?
  • Load factor determines when the HashMap resizes. Default is 0.75 — rehashing occurs when size > capacity × 0.75.
  • Lower load factor (e.g., 0.5): Fewer collisions, faster lookups, but more memory wasted on empty buckets and more frequent resizing.
  • Higher load factor (e.g., 1.0): More memory-efficient, but longer chains, more collisions, and slower lookups.
  • 0.75 is the sweet spot: It provides approximately 1 collision per 3 populated buckets — a good balance between time and space complexity.
  • Rehashing is expensive: All entries are redistributed to new buckets. If you know the expected size upfront, set initial capacity = expectedSize / loadFactor + 1 to avoid rehashing.
78
What is the time complexity of TreeMap for put()/get()?
  • O(log n) for put(), get(), remove(), and containsKey().
  • TreeMap is backed by a Red-Black tree (self-balancing BST), guaranteeing O(log n) height regardless of insertion order.
  • Range operations like subMap(), headMap(), tailMap(), firstKey(), lastKey() are also O(log n).
  • Iteration over all entries is O(n) — in-order traversal of the tree yields sorted results.
  • Comparison with HashMap: HashMap is O(1) average case but offers no ordering. Choose TreeMap when you need sorted keys or range queries; choose HashMap when you need pure speed.
79
How does ConcurrentHashMap handle collisions?
  • Same collision strategy as HashMap: Separate chaining with linked lists, converting to Red-Black trees when a bucket exceeds 8 nodes (treeification threshold).
  • Thread-safe collision handling (Java 8+): Uses CAS (Compare-And-Swap) for inserting into an empty bucket. Uses synchronized on the head node of the bucket when appending to an existing chain — only the affected bucket is locked.
  • No global lock: Unlike Hashtable, multiple threads can safely write to different buckets concurrently.
  • Read operations are lock-free: Nodes use volatile fields, so get() requires no locking and benefits from memory visibility guarantees.
  • Treeification under contention: When chains become trees, concurrent reads still traverse efficiently in O(log n) without locking.
6. Memory & JVM

Explain JVM Architecture (ClassLoader, Method Area, Heap, Stack, Native Method Area, PC Registers).
  • ClassLoader Subsystem: Loads .class files into JVM memory. Three-tier hierarchy: Bootstrap (loads core Java API from rt.jar), Extension/Platform (loads from ext directory), Application (loads from classpath). Follows the delegation model — child delegates to parent first.
  • Method Area (Metaspace in Java 8+): Stores class metadata, static variables, constant pool, and method bytecode. Shared across all threads.
  • Heap: Runtime data area for all object instances and arrays. Divided into Young Generation (Eden + Survivor spaces) and Old Generation. Shared across all threads. GC operates here.
  • Stack (per thread): Each thread gets its own stack. Stores frames — each frame holds local variables, operand stack, and a reference to the runtime constant pool. LIFO structure; created on method entry, destroyed on exit.
  • PC (Program Counter) Register: One per thread. Holds the address of the currently executing JVM instruction.
  • Native Method Stack: Holds information for native (JNI) method calls. Separate from the Java stack.
Interview tip: Draw a quick diagram showing ClassLoader → Method Area/Heap (shared) and Stack/PC/Native Stack (per-thread). Visual explanations impress interviewers.
81
What is garbage collection? How does it work?
  • Garbage collection (GC) is the automatic process of reclaiming heap memory occupied by objects that are no longer reachable from any live thread or static reference.
  • Reachability analysis: GC starts from GC roots (thread stacks, static fields, JNI references) and traverses the object graph. Any object not reachable from a root is eligible for collection.
  • Generational hypothesis: Most objects die young. The heap is divided into Young Gen (short-lived objects) and Old Gen (long-lived objects). Minor GC collects Young Gen; Major/Full GC collects Old Gen.
  • Young Gen flow: New objects are allocated in Eden. After a minor GC, survivors are copied to Survivor spaces (S0/S1). After surviving several GC cycles (configurable tenuring threshold), objects are promoted to Old Gen.
  • You cannot force GC: System.gc() is merely a request — the JVM may ignore it.
82
What are GC algorithms (Serial, Parallel, CMS, G1)?
  • Serial GC (-XX:+UseSerialGC): Single-threaded, stop-the-world. Suitable for small heaps and client-side applications. Simplest and lowest overhead.
  • Parallel GC (-XX:+UseParallelGC): Multi-threaded young and old gen collection. Maximizes throughput. Default in Java 8. Good for batch processing.
  • CMS (-XX:+UseConcMarkSweepGC): Concurrent Mark-Sweep — performs most marking concurrently with application threads. Low pause times but suffers from heap fragmentation (no compaction). Deprecated since Java 9, removed in Java 14.
  • G1 GC (-XX:+UseG1GC): Default since Java 9. Divides heap into equal-sized regions instead of fixed generations. Prioritizes regions with the most garbage (Garbage First). Balances throughput and latency with configurable pause-time targets (-XX:MaxGCPauseMillis).
  • ZGC / Shenandoah (Java 11+): Ultra-low latency collectors (< 10ms pauses) for very large heaps. Concurrent compaction with colored pointers (ZGC) or Brooks pointers (Shenandoah).
83
Explain strong, weak, soft, and phantom references with use cases.
  • Strong reference: Default. Object obj = new Object(); The object is never collected as long as a strong reference exists. Most common type.
  • Soft reference (SoftReference<T>): Collected only when JVM is low on memory. Ideal for memory-sensitive caches — data stays cached as long as memory permits.
  • Weak reference (WeakReference<T>): Collected at the next GC cycle regardless of memory pressure. Used by WeakHashMap for canonical maps, listener registries, and avoiding memory leaks in observer patterns.
  • Phantom reference (PhantomReference<T>): Enqueued in a ReferenceQueue after the object is finalized but before memory is reclaimed. Cannot retrieve the referent (get() always returns null). Used for custom cleanup logic as a safer alternative to finalize() — e.g., releasing native resources.
Interview tip: Remember the collection eagerness order: Strong > Soft > Weak > Phantom.
84
Differentiate between stack memory and heap memory.
  • Stack: Per-thread, stores method frames (local variables, method parameters, return addresses). LIFO. Automatically freed when a method returns. Very fast allocation/deallocation.
  • Heap: Shared across all threads. Stores all object instances and arrays. Managed by the Garbage Collector. Slower allocation, but objects can live beyond the creating method's scope.
  • Size: Stack is small (typically 512KB–1MB per thread, configurable via -Xss). Heap is much larger (configurable via -Xms/-Xmx).
  • Errors: Stack overflow → StackOverflowError (deep recursion). Heap exhaustion → OutOfMemoryError.
  • Thread safety: Stack is inherently thread-safe (private to each thread). Heap data requires synchronization when shared.
85
Difference between OutOfMemoryError and StackOverflowError.
  • OutOfMemoryError: Thrown when the JVM cannot allocate memory on the heap (or Metaspace). Common causes: memory leaks, loading too much data, insufficient -Xmx. Variants include Java heap space, Metaspace, GC overhead limit exceeded.
  • StackOverflowError: Thrown when a thread's call stack exceeds its maximum depth. Almost always caused by infinite or excessively deep recursion. Fix by converting recursion to iteration, increasing -Xss, or fixing the base case.
  • Both are subclasses of VirtualMachineError (extends Error) — they indicate serious JVM-level failures and should generally not be caught.
  • Key distinction: OOM is a heap problem (shared memory), SOE is a stack problem (per-thread memory).
86
How do you create immutable classes? Provide best practices.
  • Declare the class as final — prevents subclassing that could override and break immutability.
  • Make all fields private final — ensures they are set once and never exposed directly.
  • No setter methods — no way to modify state after construction.
  • Defensive copy in constructor: If accepting mutable objects (e.g., Date, List), create a deep copy to prevent external mutation.
  • Defensive copy in getters: Return copies of mutable fields, not references to the originals.
  • Initialize all fields via constructor.
public final class Money {
    private final BigDecimal amount;
    private final Currency currency;

    public Money(BigDecimal amount, Currency currency) {
        this.amount = amount;    // BigDecimal is immutable
        this.currency = currency; // Currency is immutable
    }

    public BigDecimal getAmount() { return amount; }
    public Currency getCurrency() { return currency; }
}
Interview tip: Mention Java 16+ record types — they generate immutable classes by default: record Money(BigDecimal amount, Currency currency) {}

What is escape analysis in JVM optimizations?
  • Escape analysis is a JIT compiler optimization that determines whether an object's reference escapes the scope of the method or thread that created it.
  • If the object does not escape the method: The JVM can perform scalar replacement — decompose the object into its primitive fields and allocate them on the stack instead of the heap, eliminating GC overhead entirely.
  • If the object does not escape the thread: The JVM can eliminate synchronization locks on that object (lock elision), since no other thread can access it.
  • Enabled by default in HotSpot JVM (-XX:+DoEscapeAnalysis). Works with the C2 compiler.
  • Practical impact: Short-lived objects created inside tight loops (e.g., iterators, boxed primitives, small DTOs) are often stack-allocated, dramatically reducing GC pressure.
88
Differentiate between JIT compiler and interpreter.
  • Interpreter: Reads and executes bytecode one instruction at a time. Fast startup but slower steady-state performance. Every invocation of the same method re-interprets it.
  • JIT (Just-In-Time) Compiler: Identifies hot methods (frequently executed code paths) and compiles them into optimized native machine code at runtime. Slower to kick in (compilation overhead) but much faster execution once compiled.
  • HotSpot JVM uses both: Starts with interpretation for quick startup. Once a method's invocation count crosses a threshold, the JIT compiler takes over. Two tiers: C1 (client compiler — fast compilation, basic optimizations) and C2 (server compiler — aggressive optimizations like inlining, escape analysis, loop unrolling).
  • Tiered compilation (default since Java 8): Combines C1 and C2 — C1 first for quick warm-up, C2 later for peak performance.
89
What is Metaspace (Java 8+) vs PermGen (Java 7-)?
  • PermGen (Permanent Generation): Fixed-size area within the JVM heap (configurable via -XX:MaxPermSize). Stored class metadata, interned strings, and static variables. Prone to OutOfMemoryError: PermGen space in applications with heavy dynamic class loading (e.g., app servers with frequent redeployments).
  • Metaspace (Java 8+): Replaced PermGen. Allocates class metadata in native memory (outside the JVM heap). Grows dynamically by default — no fixed upper limit unless set via -XX:MaxMetaspaceSize.
  • Key improvements: Eliminates PermGen OOM errors in most cases. Native memory allocation is more flexible and efficient. Metaspace is garbage-collected when classloaders are collected.
  • Interned strings were moved to the heap (starting Java 7), not Metaspace — so they benefit from regular GC.
90
What are stop-the-world events in GC?
  • A stop-the-world (STW) event is a GC phase where all application threads are paused so the collector can safely examine and manipulate the heap.
  • Why necessary: The GC needs a consistent snapshot of the heap to identify live objects. If application threads keep creating/modifying references during this phase, the collector's view becomes inconsistent.
  • Impact: During STW, the application is completely frozen — no user requests are served, no computations run. Long STW pauses cause latency spikes, timeouts, and degraded user experience.
  • Minimizing STW: Modern collectors (G1, ZGC, Shenandoah) minimize STW by performing most work concurrently with application threads. Only brief phases (initial mark, final remark) require STW.
  • ZGC and Shenandoah achieve sub-10ms pauses even on terabyte-sized heaps by using concurrent compaction and load barriers.

Tools to monitor JVM memory (jconsole, jvisualvm).
  • jconsole: Built-in JMX-based GUI tool. Monitors heap/non-heap memory, thread count, loaded classes, and CPU usage in real-time. Connect locally or remotely via JMX port.
  • jvisualvm (VisualVM): More powerful than jconsole. Supports heap dumps, thread dumps, CPU/memory profiling, sampler, and plugin ecosystem. Best all-in-one diagnostic tool.

  • jmap: Generates heap dumps (jmap -dump:format=b,file=heap.hprof <pid>) and histogram of object distribution. Analyze dumps with Eclipse MAT or VisualVM.

How do you detect and fix memory leaks in Java?
  • Detection — symptoms: Gradual increase in heap usage over time, frequent Full GCs, eventual OutOfMemoryError. Monitor with jstat, Grafana/Prometheus JMX metrics, or APM tools.
  • Heap dump analysis: Take heap dumps at different intervals (jmap or -XX:+HeapDumpOnOutOfMemoryError). Analyze with Eclipse MAT — look for the dominator tree and leak suspect report to find objects retaining unexpectedly large amounts of memory.
  • Common causes: (1) Collections growing unboundedly (e.g., static maps without eviction). (2) Unclosed resources (connections, streams). (3) Listeners/callbacks not deregistered. (4) ThreadLocal values not removed in thread pools. (5) ClassLoader leaks in container environments.
  • Fixes: Use try-with-resources for I/O. Bound caches with eviction policies (Guava, Caffeine). Remove listeners in lifecycle callbacks. Call ThreadLocal.remove() in a finally block. Use WeakReference for observer patterns.
  • Profiling tools: VisualVM sampler, YourKit, JProfiler, async-profiler for allocation profiling.
7. Advanced / Tricky Java Questions
93
Why doesn’t Java support multiple inheritance with classes?
  • Diamond problem: If class C extends both A and B, and both A and B have a method doWork(), the compiler cannot determine which version C should inherit — this creates ambiguity.
  • State conflict: Unlike interfaces, classes carry state (instance variables). Multiple inheritance of state leads to complex memory layout issues and constructor chaining ambiguities.
  • Java’s solution: Allow multiple inheritance of type (interfaces) but not state (classes). A class can implement multiple interfaces but extend only one class.
  • Java 8 default methods reintroduced a controlled form of multiple inheritance for behavior — the compiler enforces explicit conflict resolution when two interfaces provide the same default method.
94
Difference between deep copy and shallow copy.
  • Shallow copy: Creates a new object but copies only the references to nested objects — the original and copy share the same internal objects. Modifying a nested object in one affects the other.
  • Deep copy: Creates a new object and recursively copies all nested objects. The original and copy are completely independent — no shared mutable state.
  • Object.clone() performs a shallow copy by default. You must override clone() to manually deep-copy mutable fields.
  • Deep copy techniques: (1) Manual field-by-field copy. (2) Serialization/deserialization. (3) Copy constructors. (4) Libraries like Apache Commons SerializationUtils.clone().
// Shallow copy — addresses list is shared
Person copy = original.clone();
copy.getAddresses().add(new Address()); // affects original too!

// Deep copy — independent
Person deepCopy = new Person(original); // copy constructor clones addresses list

Differentiate between clone() and copy constructor.
  • clone(): Defined in Object. Requires implementing Cloneable marker interface. Returns Object (requires casting). Performs shallow copy by default. Fragile — bypasses constructor, breaks with inheritance, and has complex contract rules.
  • Copy constructor: A constructor that takes an instance of the same class and copies its fields. No need for Cloneable. Type-safe (no casting). Full control over deep vs. shallow copy. Works cleanly with inheritance.
  • Effective Java (Item 13): Joshua Bloch recommends avoiding clone() and preferring copy constructors or static factory methods (of(), copyOf()).
public class Employee {
    private final String name;
    private final List<String> skills;

    public Employee(Employee other) {
        this.name = other.name;
        this.skills = new ArrayList<>(other.skills); // deep copy
    }
}

What is the transient keyword? Why use it?
  • transient marks a field to be excluded from serialization. When an object is serialized, transient fields are skipped and receive their default values upon deserialization (null, 0, false).
  • Use cases: (1) Sensitive data (passwords, tokens) that should never be persisted or transmitted. (2) Derived/cached fields that can be recomputed. (3) Non-serializable fields (e.g., Thread, Socket, database connections).
  • With custom serialization: You can still serialize transient fields manually by implementing writeObject()/readObject() — e.g., encrypting a password before writing it.
  • transient vs static: Static fields are also not serialized, but for a different reason — they belong to the class, not the instance.
97
Difference between Serialization and Externalization.
  • Serialization (Serializable): Marker interface. JVM handles the entire serialization process automatically using reflection. All non-transient, non-static fields are serialized. Simple but no control over the format — can be slow and verbose.
  • Externalization (Externalizable): Extends Serializable. Requires implementing writeExternal() and readExternal(). You have full control over what and how fields are serialized. Can be significantly faster and produce smaller output.
  • Constructor: Serializable deserialization bypasses the constructor. Externalizable requires a public no-arg constructor — it’s called first, then readExternal() populates the fields.
  • When to use Externalizable: Performance-critical scenarios where you want to serialize only specific fields in a custom binary format. Otherwise, Serializable with transient is simpler.
  • Modern alternative: Most production systems use JSON/Protobuf/Avro instead of Java serialization.
98
Why is hashCode() important in collections?
  • Bucket placement: Hash-based collections (HashMap, HashSet, Hashtable) use hashCode() to compute the bucket index where an entry is stored. Without it, the collection cannot efficiently locate elements.
  • O(1) performance: A good hashCode() distributes entries uniformly across buckets, ensuring O(1) average time for get(), put(), contains().
  • Poor hashCode(): If all objects return the same hash code, all entries land in one bucket — the map degrades to a linked list with O(n) lookup.
  • Contract: If a.equals(b) is true, then a.hashCode() == b.hashCode() must also be true. The reverse is not required (collisions are allowed).
99
Why must equals() and hashCode() be consistent?
  • HashMap/HashSet contract: These collections first use hashCode() to find the bucket, then equals() to find the exact match within the bucket.
  • If equals() says “same” but hashCode() differs: Two logically equal objects land in different buckets. map.get(equalKey) returns null because it searches the wrong bucket. HashSet stores “duplicates” because it can’t detect the existing entry.
  • If hashCode() is the same but equals() says “different”: This is fine — it’s just a collision. Both entries live in the same bucket and are distinguished by equals().
  • Rule: Always override both together. Use the same fields in both methods. IDE generators (IntelliJ, Eclipse) and Objects.hash()/Objects.equals() help enforce consistency.
Interview tip: State the contract precisely: “Equal objects must have equal hash codes, but equal hash codes do not imply equal objects.”

Difference between synchronized and ReentrantLock.
  • synchronized: Built-in JVM keyword. Automatically acquires and releases the intrinsic lock. Simple syntax. Lock scope is tied to the block/method. Non-interruptible — a thread waiting for a lock cannot be interrupted.
  • ReentrantLock: Explicit lock from java.util.concurrent.locks. Must manually call lock() and unlock() (always in a finally block).
  • Advantages of ReentrantLock: (1) tryLock() — attempt to acquire without blocking. (2) tryLock(timeout, unit) — timed waiting. (3) lockInterruptibly() — can be interrupted while waiting. (4) Fairness policy — option to grant lock to the longest-waiting thread. (5) Multiple Condition objects for fine-grained wait/notify.
  • When to use: synchronized for simple cases. ReentrantLock when you need tryLock, timed lock, interruptible lock, fairness, or multiple conditions.
ReentrantLock lock = new ReentrantLock(true); // fair lock
try {
    if (lock.tryLock(2, TimeUnit.SECONDS)) {
        try {
            // critical section
        } finally {
            lock.unlock();
        }
    }
} catch (InterruptedException e) {
    Thread.currentThread().interrupt();
}

Why is Java called platform-independent?
  • “Write Once, Run Anywhere” (WORA): Java source code is compiled into bytecode (.class files) by javac. Bytecode is a platform-neutral intermediate representation — not tied to any specific OS or hardware architecture.
  • JVM abstraction: The JVM interprets (or JIT-compiles) bytecode into native machine code specific to the host platform. Each OS has its own JVM implementation (Windows, macOS, Linux), but all execute the same bytecode.
  • Platform-independent code, platform-dependent JVM: The bytecode is portable; the JVM is not. Oracle/OpenJDK provide JVM builds for all major platforms.
  • Caveat: Java is platform-independent at the bytecode level, but native libraries (JNI), file path separators, and OS-specific features can introduce platform dependencies in practice.

Difference between compile-time constants (final static) and runtime constants.
  • Compile-time constant: A static final variable initialized with a constant expression (literal or expression of literals). The compiler inlines the value at every usage site. Changing the value requires recompilation of all dependent classes.
  • Runtime constant: A static final variable initialized with a value computed at runtime (e.g., method call, constructor, config lookup). The value is resolved when the class is loaded — not inlined by the compiler.
// Compile-time constant — inlined by javac
static final int MAX_SIZE = 100;
static final String LABEL = "fixed";

// Runtime constant — resolved at class load time
static final int RANDOM = new Random().nextInt();
static final String ENV = System.getenv("APP_ENV");
  • Practical impact: If you change a compile-time constant in a library, all consuming classes must be recompiled — otherwise they still use the old inlined value. This is a subtle bug source in large codebases.
103
Why is String commonly used as a HashMap key?
  • Immutable: String is inherently immutable — its hashCode() and equals() never change after creation. This guarantees stable bucket placement in HashMap.
  • Cached hashCode: String caches its hash code after the first computation (private int hash). Subsequent calls to hashCode() return the cached value in O(1) — excellent for repeated lookups.
  • Well-implemented equals/hashCode: String’s equals() and hashCode() are rigorously implemented and consistent, fully honoring the contract.
  • String pool: Interned strings (especially literals) are shared, reducing memory and enabling == comparison in some optimized paths.
  • Ubiquitous: Most natural identifiers (names, IDs, URLs, config keys) are already strings, making them the most ergonomic key type.

What happens if you override equals() without overriding hashCode()?
  • Hash-based collections break silently. Two logically equal objects (per equals()) may return different hashCode() values (inherited from Object, which uses the memory address).
  • HashMap: put(key1, value) stores in bucket A. get(key2) (where key1.equals(key2)) looks in bucket B — returns null.
  • HashSet: set.add(obj1) and set.add(obj2) both succeed even though obj1.equals(obj2) — the set stores “duplicates.”
  • This violates the general contract of Object.hashCode(): equal objects must produce equal hash codes.
class Employee {
    String id;

    @Override
    public boolean equals(Object o) {
        return o instanceof Employee && id.equals(((Employee) o).id);
    }
    // hashCode NOT overridden — uses Object.hashCode() (memory address)
}

Set<Employee> set = new HashSet<>();
set.add(new Employee("E1"));
set.add(new Employee("E1"));
System.out.println(set.size()); // 2 — BUG! Should be 1

Differentiate between ConcurrentModificationException and IllegalStateException.
  • ConcurrentModificationException: Thrown by fail-fast iterators when the underlying collection is structurally modified during iteration (e.g., calling list.remove() inside a for-each loop instead of iterator.remove()). Detects via internal modCount mismatch.
  • IllegalStateException: Thrown when a method is called at an illegal or inappropriate time. For iterators, it occurs when remove() is called without a preceding next(), or remove() is called twice consecutively.
  • ConcurrentModificationException is about what modified the collection (external modification detected). IllegalStateException is about incorrect API usage (calling methods in the wrong order).
// ConcurrentModificationException
List<String> list = new ArrayList<>(List.of("a", "b", "c"));
for (String s : list) {
    list.remove(s); // CME! Use iterator.remove() instead
}

// IllegalStateException
Iterator<String> it = list.iterator();
it.remove(); // ISE! Must call next() before remove()
8. OOPs Concepts (Extended)
106
Difference between abstraction and encapsulation.
  • Abstraction hides complexity — it exposes only the relevant behavior and hides implementation details. It answers "what does the object do?"
  • Encapsulation hides data — it bundles fields and methods together and restricts direct access via access modifiers. It answers "how is internal state protected?"
  • Abstraction is achieved through abstract classes and interfaces. Encapsulation is achieved through private fields + public getters/setters.
  • Example: A PaymentGateway interface abstracts the payment process (pay()), while the concrete StripePayment class encapsulates the API key, retry logic, and internal state.
  • Relationship: Encapsulation is a mechanism that enables abstraction — you cannot properly abstract a module unless its internals are encapsulated.
Interview tip: Think of abstraction as "design-level hiding" and encapsulation as "implementation-level hiding." They complement each other.
107
Can an abstract class have a constructor? Why?
  • Yes. An abstract class can and often should have a constructor.
  • The constructor is not called directly (you cannot do new AbstractClass()), but it is invoked via super() from the subclass constructor during object creation.
  • Purpose: Initialize shared state that all subclasses need — e.g., setting an id, createdAt timestamp, or validating common invariants.
  • Example: abstract class Shape { protected final String color; Shape(String color) { this.color = color; } } — every concrete shape inherits and initializes color.
  • This follows the Template Method principle — common initialization lives in the parent, specific behavior in the child.
108
Can an interface extend multiple interfaces?
  • Yes. An interface can extend multiple interfaces using the extends keyword with a comma-separated list.
  • Example: interface Closeable extends AutoCloseable, Serializable { }
  • This is safe because interfaces define only contracts (method signatures), not state — there is no ambiguity about which field to inherit.
  • This is how Java achieves multiple inheritance of type without the pitfalls of multiple inheritance of state.
  • Real-world usage: List extends Collection, which extends Iterable — building rich type hierarchies via interfaces.
109
Why does Java support single inheritance for classes but multiple inheritance for interfaces?
  • Classes carry state (fields) — multiple inheritance of state creates ambiguity about which parent's field to use, leading to the diamond problem.
  • Interfaces carry only contracts (abstract method signatures) — no state conflict is possible, so multiple inheritance of type is safe.
  • Since Java 8, interfaces can have default methods with body — but if a conflict arises, the compiler forces an explicit override, eliminating ambiguity.
  • Design philosophy: Java chose simplicity and safety over flexibility. Single class inheritance avoids fragile base class issues while multiple interface inheritance provides the needed polymorphism.
110
Explain the diamond problem. How Java 8 resolved it with default methods?
  • Diamond problem: Class D extends both B and C, which both extend A. If B and C override a method from A differently, D doesn't know which version to use — this is the ambiguity.
  • Java avoids this for classes by disallowing multiple class inheritance.
  • Java 8 default methods re-introduce this risk with interfaces. Java resolves it with three rules:
  • Rule 1: A class method always wins over an interface default method.
  • Rule 2: A more specific (sub) interface wins over a less specific (parent) interface.
  • Rule 3: If ambiguity remains, the implementing class must explicitly override and choose using InterfaceName.super.method().
interface A { default void greet() { System.out.println("A"); } }
interface B extends A { default void greet() { System.out.println("B"); } }
interface C extends A { default void greet() { System.out.println("C"); } }

class D implements B, C {
    public void greet() {
        B.super.greet(); // explicit resolution
    }
}
111
What happens if two interfaces define the same default method?
  • If a class implements two interfaces with the same default method signature, the code will not compile until the class explicitly overrides the conflicting method.
  • Inside the override, you can delegate to either interface using InterfaceName.super.method().
  • If only one interface provides a default and the other declares it as abstract, the class must still provide an implementation.
  • If the two default methods have different return types that are not covariant, the conflict cannot be resolved — it's a compile-time error regardless.
interface Logger { default void log(String msg) { System.out.println("Logger: " + msg); } }

What is object slicing in OOPs?
  • Object slicing occurs when a derived class object is assigned to a base class variable, and the derived-class-specific fields are "sliced off," losing subclass data.
  • This is primarily a C++ problem because C++ allows pass-by-value for objects — copying a Dog into an Animal variable strips the Dog-specific data.
  • Java does not have object slicing because all objects are accessed via references (not copied by value). Assigning a Dog to an Animal reference simply creates an upcasted reference — the actual Dog object on the heap is unchanged.
  • In Java, polymorphism still works correctly through references — Animal a = new Dog(); a.speak(); calls Dog.speak().
Interview tip: If asked this in a Java interview, acknowledge the concept from C++ and explain why Java's reference semantics make it a non-issue.

Difference between deep inheritance vs shallow inheritance.
  • Deep inheritance: Many levels in the hierarchy (e.g., A → B → C → D → E). Each subclass adds incremental behavior.
  • Shallow inheritance: Few levels — typically one or two (e.g., A → B, A → C). Subclasses directly specialize the base.
  • Deep hierarchies are fragile: Changes to a parent class ripple through all descendants. Hard to understand, debug, and test — known as the fragile base class problem.
  • Shallow hierarchies are preferred: Easier to reason about, more maintainable, and align with the principle of composition over inheritance.
  • Best practice: Keep inheritance to a maximum of 2-3 levels. Beyond that, refactor to use composition or interfaces.
Interview tip: Cite Effective Java Item 18: "Favor composition over inheritance" — deep hierarchies are a code smell in modern Java design.

Explain SOLID, DRY, KISS, YAGNI principles in Java OOP design.
  • S — Single Responsibility: A class should have only one reason to change. E.g., separate OrderService (business logic) from OrderRepository (persistence).
  • O — Open/Closed: Open for extension, closed for modification. Use interfaces and polymorphism to add new behavior without modifying existing code.
  • L — Liskov Substitution: Subclasses must be substitutable for their parent without altering correctness. A Square extends Rectangle that breaks setWidth()/setHeight() independence violates this.
  • I — Interface Segregation: Prefer many small, focused interfaces over a single fat one. Clients should not be forced to depend on methods they don't use.
  • D — Dependency Inversion: Depend on abstractions, not concrete implementations. High-level modules should not depend on low-level modules — both should depend on interfaces.
  • DRY (Don't Repeat Yourself): Every piece of knowledge should have a single, authoritative representation. Extract shared logic into reusable methods or classes.
  • KISS (Keep It Simple, Stupid): Choose the simplest solution that works. Avoid over-engineering — e.g., don't use a Strategy pattern when a simple if-else suffices for two cases.
  • YAGNI (You Aren't Gonna Need It): Don't build features or abstractions until they are actually required. Premature generalization wastes effort and adds complexity.
Interview tip: When explaining SOLID, always pair each principle with a concise real-world Java example — it shows you've applied them in practice, not just memorized definitions.

Differentiate between instance variable, class variable, and local variable.
  • Instance variable: Declared inside a class but outside any method. Each object gets its own copy stored on the heap. Initialized to default values (0, null, false).
  • Class variable (static): Declared with the static keyword. One shared copy per class, stored in the Metaspace. Accessible via ClassName.variable.
  • Local variable: Declared inside a method, constructor, or block. Stored on the stack. Must be explicitly initialized before use — no default value. Scope is limited to the enclosing block.
  • Thread safety: Instance variables need synchronization in multi-threaded contexts. Static variables are shared across all threads — extra caution required. Local variables are inherently thread-safe (each thread has its own stack).

How do you design an immutable class? Best practices?
  • Declare the class as final — prevents subclasses from overriding behavior and breaking immutability.
  • Make all fields private and final — ensures fields are assigned once during construction and never exposed.
  • No setter methods — do not provide any method that modifies state after construction.
  • Deep copy mutable fields in the constructor — if a field is a mutable object (e.g., Date, List), create a defensive copy to prevent external mutation.
  • Return defensive copies from getters — never return direct references to mutable internal fields.

    private final Currency currency;
    private final Date createdAt;

        this.amount = amount;
        this.currency = currency;
        this.createdAt = new Date(createdAt.getTime()); // defensive copy
    }

    }
}

What is the copy-on-write mechanism?
  • Copy-on-write (COW) is an optimization strategy where a shared data structure is not copied until a write operation occurs — reads share the same underlying data.
  • When a mutation is requested, the entire internal array/structure is cloned, the modification is applied to the clone, and the reference is swapped atomically.
  • Java examples: CopyOnWriteArrayList and CopyOnWriteArraySet from java.util.concurrent.
  • Best for: Read-heavy, write-rare scenarios — e.g., listener/observer lists, configuration caches, event subscription lists.
  • Trade-off: Writes are expensive (full array copy), but reads are lock-free and extremely fast. Not suitable for frequently modified collections.

Which functional programming principles did Java 8 adopt?
  • First-class functions: Lambda expressions and method references allow passing behavior as arguments — functions are treated as values.
  • Pure functions / side-effect-free processing: Stream operations encourage stateless transformations — map(), filter(), reduce() don't modify the source.
  • Immutability: Emphasis on creating new values rather than mutating existing ones — Stream pipelines produce new collections.
  • Lazy evaluation: Intermediate stream operations are lazy — they execute only when a terminal operation triggers the pipeline.

Differentiate between reflection and introspection.

How does dynamic class loading work in Java?

Can we make main() method final, static, synchronized, or overloaded?
  • static: It must be static — the JVM invokes main() without creating an instance of the class. The signature public static void main(String[] args) is mandatory for the entry point.
  • final: Yes, you can make it final. It prevents subclasses from overriding it, but the JVM can still invoke it as the entry point.
  • synchronized: Yes, valid. The JVM acquires the class-level monitor before entering main(). Rarely useful in practice.
  • Overloaded: Yes, you can overload main() with different parameter lists (e.g., main(int x)), but only main(String[] args) serves as the JVM entry point. The overloaded versions are just regular static methods.
  • strictfp: Also valid — enforces IEEE 754 floating-point precision within main().

Differentiate between weak references and phantom references.
  • Weak Reference (WeakReference<T>): The object is eligible for GC as soon as no strong references exist. get() returns the object until it's collected, then returns null.
  • Phantom Reference (PhantomReference<T>): get() always returns null. You can never access the referent. It is enqueued into a ReferenceQueue after the object has been finalized but before its memory is reclaimed.
  • Use case — Weak: Caches (WeakHashMap) — entries are auto-evicted when keys are no longer strongly reachable. Canonicalizing mappings.
  • Use case — Phantom: Post-mortem cleanup — releasing native resources (file handles, sockets) more reliably than finalize(). Used by Cleaner (Java 9+).
  • Key difference: Weak refs let you access the object before GC; phantom refs only notify you after the object is already gone — they are purely for cleanup scheduling.

Can a stream be reused? Why not?
  • No. A stream can be consumed only once. After a terminal operation (collect(), forEach(), reduce()) is invoked, the stream is considered closed.
  • Attempting to reuse a consumed stream throws IllegalStateException: stream has already been operated upon or closed.
  • Why: Streams are designed as single-use data pipelines, not data structures. They may be backed by I/O sources or one-time iterators that cannot be rewound.
  • Workaround: Store the source collection and create a new stream each time — Supplier<Stream<T>> streamSupplier = () -> list.stream();

Difference between peek() and map().
  • map() is a transformation operation — it takes each element, applies a function, and returns a new stream of transformed elements. Signature: Stream<R> map(Function<T, R>).
  • peek() is a side-effect operation — it takes each element, performs a Consumer action (typically for debugging), and returns the same stream unchanged. Signature: Stream<T> peek(Consumer<T>).
  • Return value: map() can change the type of the stream (Stream<String> → Stream<Integer>); peek() always returns the same type.
  • Intended use: peek() is designed primarily for debugging — e.g., .peek(e -> System.out.println("Processing: " + e)). Using peek() for mutation is an anti-pattern.
  • Warning: peek() may not execute if the stream short-circuits or if no terminal operation is called — it depends on lazy evaluation.

What is a short-circuiting operation in streams? Examples.

  • Short-circuiting intermediate ops: limit(n) — stops after n elements; takeWhile(predicate) (Java 9) — stops when predicate becomes false.
  • Short-circuiting terminal ops: findFirst(), findAny() — returns as soon as one match is found; anyMatch(), allMatch(), noneMatch() — short-circuit on first decisive element.

Difference between reduce() and collect().
  • reduce() is an immutable folding operation — it combines elements into a single result by repeatedly applying a BinaryOperator. Each step produces a new intermediate value. Best for associative operations like sum, min, max, concatenation.
  • collect() is a mutable reduction — it accumulates results into a mutable container (e.g., List, Map, StringBuilder). Uses a Collector with supplier, accumulator, and combiner functions.

When should we use parallelStream()? What are the risks?
  • Use when: (1) Large dataset (10,000+ elements), (2) CPU-intensive operations per element (not I/O bound), (3) Stateless, non-interfering, associative operations, (4) The source supports efficient splitting (e.g., ArrayList, arrays — not LinkedList).
  • Risk — shared ForkJoinPool: parallelStream() uses the common ForkJoinPool. A slow task can starve other parallel streams in the same JVM. Mitigate by submitting to a custom pool.
  • Risk — ordering: Parallel processing may not maintain encounter order. If order matters, use forEachOrdered() or ensure the stream is ORDERED.
  • Risk — thread safety: If the operation modifies shared mutable state (e.g., adding to a non-concurrent list), you get race conditions. Always use thread-safe collectors or stateless lambdas.
  • Risk — overhead: For small datasets or simple operations, the thread coordination overhead exceeds the parallelism benefit — sequential is faster.
Interview tip: Always benchmark before using parallelStream(). In microservices handling many concurrent requests, the common ForkJoinPool contention is a real production issue.

How does stream pipeline execution work internally?
  • Lazy evaluation: Intermediate operations (filter, map, sorted) are not executed immediately — they build a pipeline of stages. Execution starts only when a terminal operation is invoked.
  • Vertical processing (loop fusion): Elements are processed one at a time through the entire pipeline, not in separate passes. For .filter().map().collect(), each element is filtered, then mapped, then collected before the next element is processed.
  • Short-circuiting optimization: If a terminal op like findFirst() is used, the pipeline stops as soon as the result is found — remaining elements are never processed.
  • Internal representation: Each stage wraps a Sink chain. The terminal operation calls Spliterator.forEachRemaining() which pushes elements through the sink chain.
  • Stateless vs stateful: Stateless ops (filter, map) process elements independently and are cheap. Stateful ops (sorted, distinct) need to see all elements first, creating a barrier in the pipeline.

Difference between Optional.of(), Optional.ofNullable(), Optional.empty().
  • Optional.of(value): Creates an Optional containing the given non-null value. Throws NullPointerException immediately if value is null. Use when you are certain the value is present.
  • Optional.ofNullable(value): Creates an Optional that may or may not contain a value. Returns Optional.empty() if value is null. Use when the value might be null — this is the safest and most common factory method.
  • Optional.empty(): Returns a singleton empty Optional instance. Use to explicitly represent "no value" — e.g., as a return type from a lookup method when no result is found.
// Guaranteed non-null — use of()

Best practices when using Optional in large projects.
  • Use as return type, not as field or parameter. Optional was designed for method return types to signal "this may not have a value." Avoid using it as a class field (it's not Serializable) or method parameter (adds unnecessary complexity).
  • Never return null from a method that returns Optional. Return Optional.empty() instead — returning null from an Optional-returning method is a contract violation.
  • Prefer ifPresent() / map() / flatMap() over get(). Calling get() without isPresent() throws NoSuchElementException. Use functional-style chaining instead.
  • Use orElseGet() over orElse() for expensive defaults. orElse() always evaluates the default value; orElseGet() evaluates lazily only when needed.
  • Don't use Optional for collections. Return an empty collection (Collections.emptyList()) instead of Optional<List<T>> — it adds an unnecessary wrapping layer.
  • Leverage or() (Java 9) and stream() (Java 9) for cleaner pipeline integration.

Difference between mapMulti() (Java 16) and flatMap().

  • mapMulti() uses an imperative BiConsumer<T, Consumer<R>> approach — you push zero or more results into a downstream Consumer without creating intermediate Stream objects.
  • Performance: mapMulti() avoids the overhead of creating a Stream per element — beneficial when each element maps to a small number of results (0, 1, or 2).
  • Readability: flatMap() is more declarative and idiomatic for most use cases. mapMulti() is more imperative but can be clearer for conditional or complex expansion logic.
// flatMap — creates a Stream per element

    for (Item item : order.getItems()) {
        downstream.accept(item);
    }
})
  • When to prefer mapMulti(): When each element produces 0-2 outputs (avoids unnecessary Stream creation), when you need type witness for type inference, or when the mapping logic is inherently imperative (e.g., conditional filtering + transformation in one step).

What is the difference between ConcurrentHashMap and SynchronizedMap?
  • Locking granularity: ConcurrentHashMap uses fine-grained locking — segment-level in Java 7 and CAS + node-level synchronized blocks in Java 8+. SynchronizedMap wraps the entire map with a single mutex lock.
  • Read concurrency: ConcurrentHashMap allows concurrent reads without any locking. SynchronizedMap blocks all operations (including reads) while any thread holds the lock.
  • Null handling: ConcurrentHashMap does not allow null keys or null values. SynchronizedMap (wrapping a HashMap) allows one null key and multiple null values.
  • Performance: Under high concurrency, ConcurrentHashMap significantly outperforms SynchronizedMap because contention is limited to individual buckets rather than the whole map.
Interview tip: Always prefer ConcurrentHashMap for high-concurrency scenarios. Collections.synchronizedMap() is a legacy approach and rarely the right choice in modern Java.

How does ConcurrentHashMap achieve thread safety internally?
  • Java 7 (Segment locking): The map is divided into segments (default 16), each with its own ReentrantLock. Only the segment being modified is locked, allowing 16 concurrent writes.
  • Java 8+ (CAS + synchronized): Segments are removed. Inserts into empty buckets use CAS (Compare-And-Swap). Updates to existing bucket chains use synchronized on the head node of that bucket.
  • Lock-free reads: Read operations are lock-free because node values and next pointers are declared volatile, ensuring visibility without locks.
  • Concurrent resizing: During resize, multiple threads cooperate to transfer entries from old to new table concurrently, each claiming a range of buckets to migrate.

  • Use case — BlockingQueue: Ideal for producer-consumer patterns where you want threads to wait rather than spin or poll.

Explain the producer-consumer problem using BlockingQueue.

  • Consumer: Calls queue.take() — blocks automatically if the queue is empty until an item is available.
  • No manual sync: BlockingQueue handles all thread coordination internally. No need for explicit wait/notify or locks.

    }

// Consumer thread

    while (true) {
        Integer item = queue.take();  // blocks if queue is empty
        process(item);
    }
}).start();

How do you detect and resolve deadlocks in Java?
  • Detection: Use jstack <pid> to dump thread states. Use jconsole or VisualVM to inspect deadlocked threads. Programmatically use ThreadMXBean.findDeadlockedThreads().

What is the difference between livelock and starvation?
  • Livelock: Threads are active and running but keep responding to each other without making actual progress. Classic analogy: two people in a hallway repeatedly stepping aside for each other in the same direction.
  • Starvation: A thread is unable to gain regular access to shared resources because other higher-priority threads continuously monopolize the scheduler or locks.
  • Fix livelock: Introduce randomized back-off or jitter so threads break the symmetric response pattern.
  • Fix starvation: Use fair locks (new ReentrantLock(true)) that grant access in FIFO order, or adjust thread priorities.

Compare the thread-safety of StringBuffer vs StringBuilder.

What is the ForkJoin framework? Provide a use case.
  • Purpose: Designed for divide-and-conquer parallelism. A large task is recursively split into smaller subtasks, processed in parallel, and results are combined.
  • Core classes: ForkJoinPool manages worker threads. RecursiveTask<V> returns a result. RecursiveAction returns void.
  • Work-stealing algorithm: Idle worker threads steal tasks from the tail of busy threads' deques, maximizing CPU utilization.
  • Use cases: Parallel merge sort, recursive directory traversal, parallel array sum, and the internal implementation of parallelStream().
class SumTask extends RecursiveTask<Long> {
    private final int[] array;
    private final int start, end;
    private static final int THRESHOLD = 1000;

    protected Long compute() {
        if (end - start <= THRESHOLD) {
            long sum = 0;
            for (int i = start; i < end; i++) sum += array[i];
            return sum;
        }
        int mid = (start + end) / 2;
        SumTask left = new SumTask(array, start, mid);
        SumTask right = new SumTask(array, mid, end);
        left.fork();       // submit left to pool
        return right.compute() + left.join();  // compute right, then join left
    }
}

Can two threads access different synchronized methods on the same object simultaneously?

What is the difference between StampedLock and ReentrantReadWriteLock?
  • Optimistic reads: StampedLock (Java 8) supports tryOptimisticRead() which returns a stamp without acquiring a lock. You validate the stamp later — if no write occurred, the read succeeds with zero lock overhead.
  • Read lock overhead: ReentrantReadWriteLock always acquires a read lock (even reads have lock/unlock cost). StampedLock's optimistic mode avoids this entirely.
  • Reentrancy: StampedLock is NOT reentrant — acquiring the same lock twice in the same thread causes a deadlock. ReentrantReadWriteLock is reentrant.
  • Throughput: StampedLock offers better throughput for read-dominated workloads due to optimistic locking.

int currentValue = this.value;
if (!lock.validate(stamp)) {
    // A write occurred — fall back to a full read lock
    stamp = lock.readLock();
    try { currentValue = this.value; }
    finally { lock.unlockRead(stamp); }
}

Difference between newFixedThreadPool(), newCachedThreadPool(), and newSingleThreadExecutor().
  • FixedThreadPool: Fixed number of threads with an unbounded LinkedBlockingQueue. Good for predictable, steady workloads. Risk: unbounded queue can cause OOM under load.
  • CachedThreadPool: Creates threads on demand, reuses idle threads (60-second timeout), uses SynchronousQueue (zero capacity). Good for many short-lived tasks. Risk: can create unbounded threads under burst load.

What is backpressure in concurrent systems?
  • Problem: When a producer generates data faster than the consumer can process it, causing unbounded queue growth and potential OutOfMemoryError.
  • Bounded queues: Use a bounded BlockingQueue — the producer blocks on put() when the queue is full, naturally slowing down production.
  • Rate limiting: Throttle the producer using a rate limiter (e.g., Guava RateLimiter).
  • Drop strategies: Drop the newest, oldest, or random items when the queue is full.
  • Reactive streams: Frameworks like Reactor and RxJava have built-in backpressure support where the subscriber signals demand to the publisher.

What happens if you submit too many tasks to a thread pool?
  • Unbounded queue (LinkedBlockingQueue): Tasks queue indefinitely. Memory usage grows without bound, eventually causing OutOfMemoryError.
  • Bounded queue (ArrayBlockingQueue): Once the queue is full and maximum thread count is reached, the RejectedExecutionHandler is invoked.

Design a bounded blocking queue from scratch.
  • Use a ReentrantLock with two Condition objects: notFull and notEmpty.
  • put() acquires the lock, waits on notFull if the queue is at capacity, adds the element, and signals notEmpty.
  • take() is the symmetric opposite — waits on notEmpty, removes an element, signals notFull.
  • This is exactly how ArrayBlockingQueue is implemented internally in the JDK.
import java.util.LinkedList;
import java.util.Queue;
import java.util.concurrent.locks.Condition;
import java.util.concurrent.locks.ReentrantLock;

public class BoundedBlockingQueue<T> {
    private final Queue<T> queue = new LinkedList<>();
    private final int capacity;
    private final ReentrantLock lock = new ReentrantLock();
    private final Condition notFull = lock.newCondition();
    private final Condition notEmpty = lock.newCondition();

    public BoundedBlockingQueue(int capacity) {
        this.capacity = capacity;
    }

    public void put(T item) throws InterruptedException {
        lock.lock();
        try {
            while (queue.size() == capacity) {
                notFull.await();
            }
            queue.add(item);
            notEmpty.signal();
        } finally {
            lock.unlock();
        }
    }

    public T take() throws InterruptedException {
        lock.lock();
        try {
            while (queue.isEmpty()) {
                notEmpty.await();
            }
            T item = queue.poll();
            notFull.signal();
            return item;
        } finally {
            lock.unlock();
        }
    }

    public int size() {
        lock.lock();
        try {
            return queue.size();
        } finally {
            lock.unlock();
        }
    }
}
Interview tip: Using two separate conditions (notFull / notEmpty) instead of a single lock with wait()/notify() avoids waking up the wrong type of waiting thread — more efficient and less error-prone.

Implement a multithreaded word counter using ConcurrentHashMap.
  • ConcurrentHashMap.merge() is an atomic operation — it inserts the value if absent or applies the remapping function if the key exists.
  • This eliminates the need for external synchronization when multiple threads update the same map.
  • Split the input data into chunks and assign each chunk to a separate thread.
import java.util.List;
import java.util.concurrent.*;

public class ConcurrentWordCounter {
    public static void main(String[] args) throws Exception {
        List<String> lines = List.of(
            "hello world hello",
            "world foo bar",
            "hello bar baz",
            "foo hello world"
        );

        ConcurrentHashMap<String, Long> wordCounts = new ConcurrentHashMap<>();
        ExecutorService executor = Executors.newFixedThreadPool(4);
        List<Future<?>> futures = new java.util.ArrayList<>();

        for (String line : lines) {
            futures.add(executor.submit(() -> {
                for (String word : line.split("\\s+")) {
                    wordCounts.merge(word, 1L, Long::sum);
                }
            }));
        }

        for (Future<?> f : futures) f.get(); // wait for all tasks
        executor.shutdown();

        wordCounts.forEach((word, count) ->
            System.out.println(word + " -> " + count));
        // hello -> 4, world -> 3, foo -> 2, bar -> 2, baz -> 1
    }
}
Interview tip: merge() is preferred over compute() for simple accumulation because it's cleaner. For more complex transformations, use compute() or computeIfAbsent().

Why must HashMap keys be immutable?
  • If a key's hashCode() changes after insertion, the entry becomes unreachable — it's physically stored in the old bucket but lookups compute the new hash, pointing to a different bucket.
  • HashMap uses hashCode() at insert time to determine the bucket index. Mutating the key moves its logical position but not its physical one.
  • Mutable keys lead to data loss (entries that exist but can't be found), memory leaks (unreachable entries never get removed), and unpredictable behavior in containsKey/get/remove.
  • This is why String, Integer, and other immutable types are ideal keys.
// Demonstration of the danger
Map<List<String>, String> map = new HashMap<>();
List<String> key = new ArrayList<>(List.of("a", "b"));
map.put(key, "value");

key.add("c"); // mutates the key — hashCode changes!

System.out.println(map.get(key));           // null — can't find it
System.out.println(map.containsKey(key));   // false
System.out.println(map.size());             // 1 — entry exists but is orphaned
Interview tip: This is a common trick question. Mention that the equals()/hashCode() contract must be consistent — if two objects are equal, they must have the same hash code, and this must remain stable over time.

Why is String commonly used as a key in HashMap?
  • Immutable: String is final and its internal char[]/byte[] is never exposed — guarantees a consistent hashCode() across the lifetime of the object.
  • Cached hash code: String computes its hash once and stores it in a private hash field. Subsequent calls to hashCode() return the cached value in O(1).
  • String pool: Interned strings allow == comparison in many cases, and the well-distributed hash function minimizes collisions.
  • Human-readable: Makes debugging and logging straightforward — keys are meaningful.

What happens if hashCode() is not overridden properly?
  • The default Object.hashCode() returns a value based on the memory address — logically equal objects get different hash codes.
  • This breaks the HashMap/HashSet contract: equal objects end up in different buckets.
  • put() and get() won't find existing entries — causing duplicates and failed lookups.
  • The general contract: if a.equals(b) then a.hashCode() == b.hashCode(). The reverse is not required (collisions are allowed) but should be minimized.
class Employee {
    String name;
    int id;

    // equals() overridden but hashCode() is NOT
    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof Employee)) return false;
        Employee e = (Employee) o;
        return id == e.id && name.equals(e.name);
    }
    // Missing: hashCode() override!
}

Map<Employee, String> map = new HashMap<>();
Employee e1 = new Employee("Alice", 1);
Employee e2 = new Employee("Alice", 1); // logically equal

map.put(e1, "Engineer");
System.out.println(map.get(e2)); // null! Different hashCode => different bucket
Interview tip: Always override hashCode() when you override equals(). Modern IDEs and libraries like Lombok (@EqualsAndHashCode) generate both consistently.

Difference between ConcurrentModificationException and IllegalStateException.
  • ConcurrentModificationException: Thrown by fail-fast iterators when the collection is structurally modified during iteration (other than through the iterator's own remove() method). This includes modifications by the same thread or another thread.
  • IllegalStateException: Thrown when a method is called at an illegal or inappropriate time — for example, calling Iterator.remove() before calling next(), or calling next() without checking hasNext() when no elements remain.
  • Key difference: ConcurrentModificationException relates to structural modification of the collection; IllegalStateException relates to method call order/state of the iterator or object.
// ConcurrentModificationException
List<String> list = new ArrayList<>(List.of("a", "b", "c"));
for (String s : list) {
    list.remove(s);  // structural modification during iteration — throws CME
}

// IllegalStateException
Iterator<String> it = list.iterator();
it.remove(); // called before next() — throws IllegalStateException

How to handle large datasets efficiently with Streams?
  • Lazy evaluation: Intermediate operations (filter, map) don't process data until a terminal operation (collect, forEach, reduce) triggers execution. This avoids unnecessary computation.
  • Parallel processing: Use parallelStream() for CPU-bound operations on large collections — it splits the data across the common ForkJoinPool.
  • Avoid large intermediate collections: Use forEach() or reduce() for aggregation instead of collecting millions of elements into a List.
  • Streaming I/O: Use Files.lines() or BufferedReader.lines() to process files line-by-line without loading the entire file into memory.
  • Short-circuit operations: findFirst(), anyMatch(), limit() stop processing as soon as the result is determined.
Interview tip: Mention that streams are single-use — once a terminal operation is invoked, the stream is consumed. Attempting to reuse throws IllegalStateException.

When should we not use parallel streams?
  • Small data sources: Thread scheduling and fork/join overhead exceeds the benefit for small collections.
  • Order-dependent operations: If the result must preserve encounter order, parallelism provides no speedup (or even degrades it).
  • Side effects or shared mutable state: Operations that mutate external state (e.g., writing to a shared variable) introduce race conditions.
  • Non-thread-safe collectors: Collectors that accumulate into non-concurrent data structures can produce corrupt results.
  • Saturated ForkJoinPool: parallelStream() uses ForkJoinPool.commonPool() — if other tasks are already using it, adding more workload can cause thread starvation.
  • I/O-bound operations: The common pool has only CPU-core-count threads — I/O-bound work (HTTP calls, DB queries) will block these threads and starve CPU-bound tasks.
Interview tip: A great answer mentions that you should benchmark before using parallel streams — intuition about when parallelism helps is often wrong. Use JMH for micro-benchmarks.

Can a Stream run infinitely? How do you handle it?
  • Yes. Stream.generate(Math::random) and Stream.iterate(0, n -> n + 1) produce infinite streams — they generate elements on demand without bound.
  • You must use short-circuiting terminal operations to terminate: limit(n), findFirst(), findAny(), anyMatch().
  • Java 9+ added takeWhile(predicate) for conditional termination: Stream.iterate(1, n -> n + 1).takeWhile(n -> n <= 100).
  • Without a short-circuiting operation, the program runs forever or throws OutOfMemoryError if elements are collected.
// Infinite stream — must be limited
Stream.generate(Math::random)
    .limit(5)
    .forEach(System.out::println);

// Java 9+ iterate with predicate (bounded)
Stream.iterate(1, n -> n <= 100, n -> n + 1)
    .forEach(System.out::println); // prints 1 to 100

// takeWhile (Java 9+)
Stream.iterate(1, n -> n * 2)
    .takeWhile(n -> n < 1000)
    .forEach(System.out::println); // 1, 2, 4, 8, ..., 512

Why prefer CompletableFuture over Thread or Executor?
  • Non-blocking chaining: thenApply(), thenCompose(), thenCombine() allow building async pipelines without blocking any thread.
  • Built-in exception handling: exceptionally() and handle() provide clean fallback/recovery — no need for try-catch blocks around Future.get().
  • Combining futures: allOf() waits for all to complete; anyOf() returns the first to complete — essential for parallel service calls.
  • Default thread pool: Runs on ForkJoinPool.commonPool() — no manual thread management needed for most use cases.
  • Cleaner than Thread/Executor: Eliminates manual thread coordination with wait()/notify() or blocking Future.get() calls.
// Combining two independent async calls
CompletableFuture<String> userFuture  = CompletableFuture.supplyAsync(() -> fetchUser());
CompletableFuture<String> orderFuture = CompletableFuture.supplyAsync(() -> fetchOrders());

CompletableFuture<String> combined = userFuture.thenCombine(orderFuture,
    (user, orders) -> user + " has " + orders);

combined.thenAccept(System.out::println)
        .exceptionally(ex -> { System.err.println(ex); return null; });
Interview tip: Mention that in Spring Boot, @Async methods return CompletableFuture — this is the standard pattern for non-blocking service calls in enterprise applications.

What are the pitfalls of using Optional as a field in entities?
  • Not Serializable: Optional does not implement Serializable — breaks JPA/Hibernate entities, RMI, and message serialization frameworks.
  • Memory overhead: Each Optional field wraps the value in an extra object — wasteful for entities with many fields or large collections of entities.
  • Design intent: Optional was designed as a return type to express "this method might return nothing" — not as a general-purpose nullable field wrapper.
  • Frameworks don't support it: Jackson, JPA, and many ORM frameworks expect concrete types for fields. Using Optional as a field type requires custom serializers and converters.
  • Best practice: Prefer nullable fields with null checks in entity classes. Use Optional only in return types of public APIs.
// Bad — Optional as a field
public class User {
    private Optional<String> middleName; // not Serializable, bloats memory
}

// Good — nullable field, Optional return type
public class User {
    private String middleName; // nullable

    public Optional<String> getMiddleName() {
        return Optional.ofNullable(middleName);
    }
}

How to debug a thread deadlock in a production JVM?
  • jstack <pid>: Dumps all thread states — look for threads in BLOCKED state waiting on each other's locks. JVM automatically detects and reports deadlocks in the output.
  • JConsole / VisualVM: Visual tools with a "Detect Deadlock" button that highlights the deadlocked threads and the locks they hold.
  • Programmatic detection: ThreadMXBean.findDeadlockedThreads() returns an array of thread IDs involved in the deadlock — useful for building monitoring/alerting into your application.
  • Prevention strategies:
    • Always acquire locks in a consistent global order across all threads.
    • Use tryLock(timeout) from ReentrantLock instead of synchronized — it allows timeout-based acquisition and avoids infinite waits.
    • Avoid nested locking where possible.
    • Log lock acquisition and release events in critical sections for post-mortem analysis.
// Programmatic deadlock detection
import java.lang.management.*;

ThreadMXBean bean = ManagementFactory.getThreadMXBean();
long[] deadlockedIds = bean.findDeadlockedThreads();

if (deadlockedIds != null) {
    ThreadInfo[] infos = bean.getThreadInfo(deadlockedIds, true, true);
    for (ThreadInfo info : infos) {
        System.err.println("Deadlocked thread: " + info.getThreadName());
        System.err.println("  Waiting for lock: " + info.getLockName());
        System.err.println("  Held by thread:   " + info.getLockOwnerName());
    }
}
Interview tip: In production, set up periodic deadlock checks via ThreadMXBean in a monitoring thread — this provides proactive detection before users notice symptoms.

Real-world cases where CopyOnWriteArrayList is preferred?
  • Listener/observer lists: GUI event listeners or pub-sub subscriber lists where listeners are iterated (notified) frequently but rarely added or removed.
  • Configuration lists: Application settings loaded at startup and read on every request but updated only during deployment or config reload.
  • Event bus subscriber lists: Frameworks like Guava EventBus use copy-on-write semantics to iterate handlers safely while allowing runtime registration.
  • Snapshot iteration: Iterators operate on a snapshot of the array taken at creation time — guaranteed no ConcurrentModificationException, even if another thread modifies the list.
  • Trade-off: Every write (add, set, remove) copies the entire internal array — O(n) per write. Only use when reads vastly outnumber writes.

When to prefer LinkedList over ArrayList?
  • Frequent insertion/deletion at the beginning or middle: With an iterator positioned at the target, LinkedList inserts/removes in O(1) vs. O(n) element shifting for ArrayList.
  • Deque operations: LinkedList implements Deque — addFirst(), addLast(), pollFirst(), pollLast() are all O(1).
  • No resize overhead: LinkedList never needs to resize an internal array — useful if memory allocation bursts are problematic.
  • Reality check: In practice, ArrayList is preferred 95%+ of the time because CPU cache locality makes its sequential access dramatically faster than LinkedList's pointer-chasing across scattered heap memory.
Interview tip: The nuanced answer is: "Almost always use ArrayList. The theoretical O(1) insert advantage of LinkedList rarely matters in practice because finding the insertion point is O(n) and cache misses dominate."

What is the impact of load factor tuning in HashMap?
  • Default load factor = 0.75: A well-tested balance between space efficiency and lookup speed. The map resizes when size > capacity × loadFactor.
  • Lower load factor (e.g., 0.5): More frequent resizing and higher memory usage, but fewer collisions per bucket — faster get()/put().
  • Higher load factor (e.g., 0.9): Less memory and fewer resizes, but more collisions — slower lookups as bucket chains grow longer.
  • Optimization for known size: Set initial capacity = expectedEntries / loadFactor + 1 to avoid any resizing. Example: for 100 entries at 0.75 load factor, set capacity to 134.
  • Resizing cost: Each resize doubles the internal array and rehashes every entry — an O(n) operation that causes latency spikes if triggered during request processing.
// Pre-sized map to avoid resizing
int expectedEntries = 1000;
float loadFactor = 0.75f;
int initialCapacity = (int) (expectedEntries / loadFactor) + 1;

Map<String, String> map = new HashMap<>(initialCapacity, loadFactor);

// Guava shorthand
// Map<String, String> map = Maps.newHashMapWithExpectedSize(1000);
Interview tip: Mentioning that you pre-size HashMap in performance-critical code shows production-level awareness. The default constructor creates a map with capacity 16 — fine for small maps, wasteful rehashing for large known-size maps.
